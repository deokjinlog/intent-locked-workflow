# v2.0.0 implementer byte-copy + reorder 3-stage 패턴 (메이저 릴리즈)

> **Version note (v2.0.0)**: subagent dispatch 패턴이 LLM transcription → byte-copy + 분담으로 근본 변경. plan 작성 룰의 정밀도 요구치도 같이 올라감 (`**원본**` 블록 byte-equal 강제). breaking change 가 plan-side 에 있으므로 minor bump (v1.1.18) 가 아닌 **major bump (v2.0.0)** 가 적절.

## 한 줄 요약

기존 implementer LLM transcription 방식 → **byte-copy 강제 + 사용자 수정 충돌 시 reorder subagent 분담** 으로 전환. drift 제거 + 비용 절감 + 사용자 수정 silent overwrite 차단.

## 배경 / 문제

### 현재 (v1.1.14~v1.1.17)

`js-super-subagent-driven-development` 의 implementer subagent 는 plan 의 `**수정 후**` 블록을 읽고 LLM 이 직접 파일에 transcription. 즉:

1. implementer 가 plan task 의 코드 블록을 읽음
2. 자체 판단으로 Write/Edit tool 호출
3. **LLM 이 코드를 다시 옮겨 적는 모델** — drift 가능

### 발견된 문제점

- **transcription drift**: implementer 가 "improve" 의도로 코드 살짝 수정 (코멘트 추가 / import 순서 변경 / 변수명 미세 변경 / docstring 누락 등). spec-reviewer 가 catch 하지만 catch 후 fix loop 비용 발생.
- **silent overwrite 위험**: 사용자가 plan 실행 전 파일 직접 수정한 경우, implementer 가 "어 코드가 다르네" 무시하고 그냥 plan 대로 덮어씀. 사용자 수정 손실.
- **implementer 비용**: 매 task 마다 LLM 추론 (sonnet 기본) 사용. transcription 자체는 LLM 추론이 필요 없는 작업.

### upstream 비교

upstream `subagent-driven-development/implementer-prompt.md` 도 동일한 LLM transcription 방식. 별도 byte-copy enforcement 없음. 즉 위 문제 모두 upstream 에도 존재. js-super 가 byte-copy 강제하면 **upstream 과 차별화되는 명백한 개선**.

## 제안 — 3-stage 분담안

### Stage 1: Implementer (haiku 고정, byte-copy only)

| 동작 | 상세 |
|---|---|
| 모델 | **haiku 고정** (현재 plan 의 `**Model**:` 필드 무시 — auto-flow + byte-copy 모드에서는 의미 없음) |
| 입력 | task 본문 + `**원본**` + `**수정 후**` 블록 |
| 동작 | `**원본**` 의 byte-copy = Edit `old_string`, `**수정 후**` 의 byte-copy = Edit `new_string`. Create task 면 Write 의 `content` = `**수정 후**` byte-copy. |
| 금지 | 코드 paraphrase / 들여쓰기 normalize / 변수명 변경 / 코멘트 추가 / docstring 자동 생성 — 일체 X |
| 실패 처리 | Edit "old_string not found" 발생 시 즉시 BLOCKED 보고, 재시도 X (LLM 판단 X) |

implementer-prompt 본문에 추가될 룰:

> **STRICT BYTE-COPY (v2.0.0+)**: When the task has `**원본**` / `**수정 후**` blocks, your Edit tool's `old_string` argument MUST be byte-identical to the `**원본**` block content. Your `new_string` argument MUST be byte-identical to the `**수정 후**` block content. DO NOT paraphrase, reformat, or "improve". DO NOT normalize whitespace. DO NOT add comments. If Edit fails with "old_string not found", report `Status: BLOCKED — 원본 mismatch at <file>` and stop. The main agent will dispatch a reorder subagent.

### Stage 2: Reorder/Resolve (sonnet, 조건부)

`Stage 1` 실패 시에만 발화. 별도 prompt 파일 (`./reorder-prompt.md`).

| 동작 | 상세 |
|---|---|
| 모델 | **sonnet** 고정 (자유 판단 필요 — Korean prose 도 가능) |
| 입력 | task + `**원본**` + `**수정 후**` + 실제 파일 현재 상태 + Stage 1 의 BLOCKED reason |
| 동작 | 사용자 수정 의도 보존하면서 plan 변경 reconcile. 다음 중 하나 선택: (a) 사용자 수정과 plan 변경이 호환 가능 → 둘 다 적용, (b) 충돌 → 메인에게 사용자 게이트 요청 (`Status: NEEDS_USER — <설명>`) |
| 금지 | 사용자 수정 silent overwrite. 의도 모호 시 자유 판단 X — NEEDS_USER 로 escalate. |

### Stage 3: Spec Reviewer (sonnet, 기존 그대로)

기존 패턴 변경 X. line-by-line 정합성 검증.

### 흐름 요약

```
Wave 안 task 1개:

  ┌──────────────┐
  │ Implementer  │  haiku, byte-copy only
  │              │
  └──────────────┘
      │
      ├── Edit 성공 ─────────────┐
      │                          │
      ├── BLOCKED (원본 미스매치) │
      │                          │
      ▼                          │
  ┌──────────────┐               │
  │ Reorder      │  sonnet, conditional
  │              │  사용자 수정 reconcile
  └──────────────┘               │
      │                          │
      ├── 성공 ──────────────────┤
      │                          │
      ├── NEEDS_USER (충돌 모호) │
      │                          │
      ▼                          │
  사용자 게이트                    │
  (메인이 AskUserQuestion 발화)    │
                                  │
                                  ▼
                          ┌──────────────┐
                          │ Spec Reviewer │  sonnet, 기존
                          └──────────────┘
                                  │
                                  ▼
                          Wave finalization
```

## 좋은 점 (정량/정성)

### 정량

- **implementer 비용 ↓** — 매 task sonnet → haiku. 14 task feature 기준 sonnet 14회 vs haiku 14회 + sonnet conditional (충돌 1~2건 시 sonnet 1~2회). **누적 토큰 ~50% 절감 추정** (haiku 가 sonnet 대비 10x 저렴).
- **transcription drift 0** — plan = 파일 byte-equal 보장. spec-reviewer fix loop 빈도 감소.
- **wall-clock 시간 ↓** — haiku 가 sonnet 보다 빠른 응답. wave 안 task pair-parallel 의 worst-case latency 감소.

### 정성

- **plan = single source of truth** — plan 본문이 곧 파일 상태. 사후 review 시 plan 만 봐도 충분.
- **silent overwrite 차단** — 사용자가 plan 실행 전 파일 수정한 케이스가 명시적으로 catch.
- **upstream 차별화** — js-super 만의 검증된 안전성 패턴 (upstream 은 LLM transcription 그대로).

## 우려 사항 / 단점

### 1. plan-side 정밀도 부담 ↑ (가장 큰 trade-off)

`**원본**` 블록이 실제 파일과 **byte-equal** 보장 필요. 현재 룰은 "byte-equal 권장" 정도 — 실제로는 약간 어겨도 LLM 이 fuzzy match. byte-copy 모드에서는 미스매치 시 false-fail.

**Mitigation**: writing-plans skill 에 helper 추가 — plan 작성 시점에 `**원본**` 블록이 현재 파일과 byte-equal 인지 자동 검증. 미스매치 시 plan 작성자에게 즉시 catch.

```python
# scripts/plan_byte_check.py (신규)
def verify_plan_block_byte_equal(plan_path: Path, repo_root: Path) -> List[Mismatch]:
    """plan 의 모든 **원본** 블록이 실제 파일과 byte-equal 인지 검증."""
    ...
```

### 2. 들여쓰기 / 트레일링 공백 false-fail

markdown 렌더링용으로 plan code block 의 들여쓰기가 살짝 다른 경우, byte-equal 깨짐.

**Mitigation 옵션**:
- (A) plan 작성자가 손수 정리 (writing-plans skill 의 verify 단계가 catch)
- (B) implementer-prompt 에 "fenced code block 의 leading 4-space 제거 후 비교" 같은 normalize 룰 — 단 이 normalize 자체가 drift 위험. **거부 권장**.

### 3. 다중 Edit task 의 부분 실패 시 rollback

task 가 5건의 sequential Edit 호출하는데 3번째에서 fail 하면 1~2번은 이미 적용됨. byte-copy 모드는 LLM 판단 없으므로 자동 rollback 안 함.

**Mitigation**: implementer 가 BLOCKED 보고 시 메인이 wave finalization 단계에서 `git checkout -- <task.files>` 로 working tree rollback. 기존 failure isolation 패턴과 동일.

### 4. plan-runner Python tool 도입 가능성 (선택사항)

byte-copy 가 deterministic 하다면 LLM 거치지 않아도 됨. plan 파싱 → Edit/Write 자동 실행하는 Python helper 가능. 단:
- 메인이 Skill tool 호출 비용 < Python tool 호출 비용 (현재 harness 모델)
- LLM 의 self-review 기능 (도중에 이상 감지 시 escalate) 손실
→ **v1.1.19+ 후속 검토 항목**, 본 release 에는 미포함.

### 5. 사용자가 "AI 가 알아서 처리해주길" 원하는 케이스

drift 제거가 곧 "AI 의 자율 보정 기능 제거". 일부 사용자는 LLM 의 미세 보정 (PEP 8 자동 적용 등) 을 선호할 수 있음.

**Mitigation**: byte-copy 는 auto-flow 진입 + subagent 모드에 한정. 일반 `/execute-plan` (executing-plans inline) 은 영향 0 — 기존 LLM 패턴 유지. 사용자 선택 보존.

## 다른 안 (대안 비교)

### 대안 A: Escalation 패턴 (별도 reorder stage 없이 implementer 재dispatch)

implementer BLOCKED → 메인이 같은 implementer-prompt 를 sonnet 으로 재dispatch.

**거부 이유**:
- 같은 prompt 로 두 가지 일 (구현 + reconcile) 시키면 silent 위험 ↑
- byte-copy 룰 vs reconcile 자유 판단 룰이 같은 prompt 안에서 모순. 모델 승급으로 어느 쪽 우선인지 불명확.
- 관측성 ↓ — 모델 승급 = "충돌 발생" 신호이지만 prompt 가 같아서 디버깅 시 분리 어려움.

분담안의 **명시적 prompt 분리** 가 더 안전.

### 대안 B: Spec reviewer 가 reorder 도 통합

implementer fail → spec reviewer (sonnet) 가 정합성 검증 + 필요시 reconcile.

**거부 이유**:
- spec reviewer 의 역할 혼합 — "is implementation correct?" vs "if user-edited, fix order". 두 가지 일이 한 prompt 에 묶이면 silent 수정 위험.
- spec reviewer 가 reconcile 까지 하면 "검증" 과 "수정" 경계 흐려짐.

### 대안 C: 현상 유지 (LLM transcription)

- 사용자 수정 silent overwrite 위험 그대로.
- drift 그대로 — spec-reviewer fix loop 비용 그대로.
- upstream 과 차별화 X.

→ **거부**: 이미 알려진 문제 해결 가치 큼.

### 대안 D: byte-copy 만 (reorder stage 없음, fail 시 종료)

implementer BLOCKED → wave finalization 에서 task 격리 (failure isolation 그대로).

**부분 채택 가능**: 단순한 모델. 단 사용자 수정이 흔한 시나리오 (e.g., dogfood 중 사용자가 직접 수정 + auto-flow 재실행) 에서 task 빈번하게 BLOCKED. UX 나쁨.

→ 분담안의 reorder stage 가 사용자 마찰 줄임. **분담안 우월**.

### 비용 영향

- implementer 모델: sonnet → haiku. **누적 토큰 ~50% 추정 절감**.
- reorder dispatch (조건부, sonnet): 충돌 발생률 1~5% 추정 시 미미.
- spec reviewer: 변경 X.
- writing-plans 의 plan_byte_check: 추가 dispatch 없음 (Python helper).

### 위험

- plan 작성자의 정밀도 부담 ↑ — 단 plan_byte_check helper 가 catch. 부담은 plan 단계에서 해소되고 실행 단계는 안전.
- 들여쓰기 false-fail 가능 — plan_byte_check 가 작성 단계에서 catch.

### 우선순위

- **high (메이저 v2.0.0)** — js-super 의 핵심 dispatch 모델 근본 개편. plan-side 룰도 같이 강화 (`**원본**` byte-equal 강제) 라 breaking change. v1.1.16 backlog 2건 (DAG ordering bug + preflight regex) micro-patch 먼저 후 진행. v1.1.16 stable → v1.1.X 추가 패치 누적 가능 → v2.0.0 메이저 점프.
