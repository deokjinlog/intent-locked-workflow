# v1.1.14 후보 — finishing-a-development-branch 슬림화 (되묻기 게이트 제거)

> 작성: 2026-05-10
> 우선순위: 중간 — 1인 개발 철학과의 일관성 + 마찰 제거 효과 명확.

## 문제

`skills/finishing-a-development-branch/SKILL.md` 가 워크플로 종료 시점에 **AskUserQuestion 게이트로 되묻기** 함:

`SKILL.md:53~83` Step 3:
```
질문: "구현 완료. 다음 단계는?"
4 options: merge / pr / keep / discard
```

→ 1인 개발자는 마무리 시점에 이미 무엇을 할지 알고 있음. **추가 입력 요구는 마찰**일 뿐.

또 Option 4 (discard) 선택 시 **"discard" 타이핑 추가 강제** (`SKILL.md:137~147`) — 두 번째 되묻기.

### js-super 철학과 충돌

js-super 가 표방하는 1인 개발 철학:
- 사전 `verifying-spec` 게이트 + TDD + RISK + 변경이력으로 안전성 확보
- final reviewer 같은 추가 게이트 제거 (`js-super-subagent-driven-development` 에서 quality reviewer 제거 결정)

→ 그런데 **마무리에 또 4-option 게이트** 가 있는 건 이 철학과 어긋남.

## 해결 — Option B (안전망만 남기고 게이트 제거)

### 유지

- **Step 1 테스트 자동 실행** — merge 전 tests pass 강제. 깨진 코드 푸시 방지 안전망. 가치 명확.

### 제거

- **Step 2 base branch 자동 추정** — 본인이 이미 알고 있음. 추정 시간 낭비.
- **Step 3 AskUserQuestion 4-option 게이트** — 되묻기. 1인 개발자는 이미 결정한 상태로 진입.
- **Step 4 git command 템플릿 (merge / push / PR / discard 각 boilerplate)** — `git checkout && git merge` 같은 기본 명령은 외워서 박을 수 있음. PR template 도 자동 생성보다 사용자가 직접 쓰는 게 자연스러움.
- **Step 4 Option 4 "discard" 타이핑 게이트** — discard 자체가 거의 안 일어나는 시나리오. 안 한다고 안전 문제 발생 안 함.

### 분리

- **Step 5 worktree cleanup** → 별도 skill `cleanup-worktree` 로 분리.
  - `setting-up-worktrees` 와 페어. 사용자가 명시 호출 시만 작동.
  - 자동 호출 안 함 (마무리 시점이라고 자동 정리하면 Option keep 선택 시 자료 손실 가능).
  - 또는 단순히 사용자가 `git worktree remove <path>` 직접 실행 (skill 자체 불필요할 수도).

### TO-BE 워크플로 종료 메시지

```
✅ 모든 task 완료 + 테스트 통과.
   - 변경 commit: <SHA list>
   - RISK 트리거: <카테고리 카운트>
   - 누락/초과: <list 또는 "없음">

마무리하세요. (예: git checkout main && git merge <branch>, 또는 PR 생성, 또는 그대로 두기)
```

→ 게이트 없음. 사용자가 직접 처리.

## 위험 / 엣지케이스

- **discard 시 실수 방지 안전망 사라짐** — 사용자가 직접 `git branch -D` 시 confirm 없음. 트레이드오프 의도적으로 받아들임. 1인 개발자는 보통 git 동작 알고 있고, 정말 큰 작업이면 PR 생성 후 삭제 함.
- **base branch 자동 추정 사라짐** — 본인이 명시 안 하면 어디로 머지할지 모름. 그런데 어차피 본인이 머지하므로 본인이 결정하면 됨.
- **테스트 실패 시 동작** — 현재처럼 STOP 메시지 유지. 안전망 핵심.
- **executing-plans / subagent-driven-development 와의 통합** — 현재 두 skill 모두 finishing-a-development-branch 를 명시적 sub-skill 로 호출. 슬림화 후에도 호출 자체는 유지하되 내용이 가벼워짐. 또는 호출 자체 제거하고 종료 메시지로 대체 — 어느 쪽이 더 깔끔한지 PRD 단계에서 결정.
