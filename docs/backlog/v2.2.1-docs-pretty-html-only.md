# v2.2.1 후보 — docs-pretty 를 .html 전용 fire-and-forget 으로 전환

> 작성: 2026-05-16
> 우선순위: 중간 — v2.2.0 (.html 동봉본) 의 후속 단순화. 단독 채택 시 v2.2.0 단계는 건너뛰는 안도 가능.

## 배경 (v2.2.0 과의 관계)

v2.2.0 백로그 (`v2.2.0-docs-pretty-html-companion.md`) 는 docs-pretty 발동 시 Sonnet subagent **2개 병렬** dispatch (A: `.md` format-only / B: `.html` 신규 생성) 안. dogfood 가설:

- AI 는 어차피 `.md` 만 읽음 → `.md` format-only pass 의 가치는 **사람용**
- 그런데 `.html` 가 사람용 가독성 사본이 되면 `.md` format-only 가공은 **중복**
- 그러면 docs-pretty 가 메인 흐름 동기 블록할 이유가 사라짐 (사람용 자료는 백그라운드로 처리해도 됨)

→ v2.2.1 = **`.md` 가공 제거 + `.html` 생성 fire-and-forget**.

## 해결 — docs-pretty 책임 단순화

### Before (현재 / v2.2.0)

```
RAW .md 작성
  → docs-pretty 발동 (sync, blocking)
    → Sonnet 가 .md format-only pass
    → (v2.2.0 안에서는 추가로 .html 도 병렬 생성)
  → user review 정형 .md
```

메인 흐름이 docs-pretty 완료까지 **대기**.

### After (v2.2.1)

```
RAW .md 작성
  → docs-pretty 발동 (fire-and-forget — .html 생성 subagent 만 dispatch)
  → 메인은 즉시 다음 task (user review RAW .md)
  → (배경에서 .html subagent 가 완성되면 사이드카로 떨어짐)
```

메인 흐름이 docs-pretty 결과 **대기 안 함**. `.md` 가공 자체가 사라짐.

## 흐름별 동기화 정책

### 초기 작성 iteration (첫 change-history entry 전)

- 매 user fix turn 마다 메인이 `.md` 갱신 + `.html` regen subagent dispatch + 즉시 다음 turn
- **디바운스**: 사용자가 연속 fix 던질 경우 마지막 fix 후만 regen — 비용 ↓
- staleness 윈도우 짧음 (다음 fix 들어오기 전 .html 완성되면 OK, 늦으면 그냥 다음 버전이 덮어씀)

### Live doc 진입 후 (첫 change-history entry 이후)

기존 docs-pretty 의 boundary (첫 change-history entry → STOP) 유지. 단 `.html` 동기화는 두 경로로:

| 경로 | 트리거 | 처리 |
|---|---|---|
| 자동 | `change-propagation` 스킬 마지막 단계 | `/regen-html <slug>` 자동 호출 |
| 수동 | 사용자가 직접 `.md` 편집 (오타 한두 글자 등) | 사용자가 `/regen-html <slug>` 수동 호출 |

→ "의미 있는 변경" (change-propagation) 만 자동, 그 외는 stale 허용.

### Stale 표시 (.html 푸터)

`.html` 푸터에 generation 기준 시점 박음:

```html
<footer>
  이 파일은 <slug>-requirements.md 의 [CH-YYYYMMDD-NNN @ YYYY-MM-DD HH:MM] 기준입니다.
  이후 .md 수정 사항은 반영되지 않을 수 있습니다 — 필요 시 /regen-html <slug>.
</footer>
```

사용자가 시각적으로 stale 인지 가능.

## 신규 / 변경 / 삭제 항목

### 신규: `commands/regen-html.md` 커맨드

- `/regen-html <slug>` slash 트리거
- 인자: feature slug (e.g., `2026-05-16-foo`)
- 동작: `<slug>-{requirements,tech-design,implementation-plan}.md` 셋 중 존재하는 모든 .md 에 대해 `.html` 재생성 subagent dispatch
- 인자 누락 시 latest slug 추론 (가장 최근 수정된 features 폴더)
- skill 이 아니라 **command** — 사용자 명시 호출용 1회성 액션

### 변경: `docs-pretty` SKILL.md 본문

- `.md` format-only pass 책임 **제거**
- Procedure 섹션 → `.html` subagent fire-and-forget dispatch 로 교체
- 동작 boundary (초기 작성 iteration 만, 첫 change-history entry 후 STOP) 보존
- pre-flight 룰 (target 존재 + footer 비어있음 + filename 패턴) 보존

### 변경: `change-propagation` SKILL.md 본문

- 마지막 단계에 `/regen-html <slug>` 자동 호출 명시
- live doc 의 `.html` 동기화 보장

### 영향 X (그대로 유지)

- `brainstorming` / `designing-direction` / `writing-plans` / `executing-plans` 본문 변경 0 — 모두 `.md` 만 읽음
- `auto-*` 본문 변경 0
- `og-*` 본문 변경 0 — upstream mirror
- `code-pretty` 별도 skill, 본 변경과 독립
- `change-history` 본문 변경 0 — `.html` 은 change-history footer 추적 대상 X

## fire-and-forget dispatch 명세

메인이 docs-pretty 발동 시:

```
Task tool 호출 1회 (subagent_type=general-purpose 또는 신규 html-renderer):
  - 모델: Sonnet
  - 입력: 현재 RAW .md 경로
  - 출력: 같은 폴더에 .html 사이드카 작성
  - 실행 모드: run_in_background=true (메인은 결과 대기 X)
```

- 메인은 subagent 완료 notify 받으면 **silent log** (사용자에게 push X)
- 실패 시: silent log + 사용자가 다음에 `.html` 열려고 할 때 부재 인지 → `/regen-html` 수동
- 디바운스: 메인이 마지막 fix turn 마다 이전 subagent 가 살아있으면 cancel + 새 dispatch

## `.html` 자체 룰 (v2.2.0 룰 보존)

| 항목 | 룰 |
|---|---|
| 의미 보존 | `.md` 의 문장 / 항목 / 결정 사항 / 숫자 / 표 내용 1:1 매핑. 의역 / 요약 / 재구조화 X |
| 시각화 | 헤더 hierarchy, 표 정렬·줄무늬·sticky, DAG/관계는 Mermaid 또는 inline SVG, syntax highlight |
| Self-contained | 외부 CDN / JS / CSS 의존 X. inline 빌드. |
| 다크/라이트 | prefers-color-scheme 자동 |
| AI 노출 | **X** — 모든 AI 흐름은 `.md` 만 읽음 |

## 검증

- 메인 흐름 정시간 측정 — docs-pretty 발동 전후 메인 turn elapsed 차이 < 100ms (fire-and-forget 검증)
- `.html` subagent 완료 timestamp 가 메인 다음 turn 시작 후에 도달하는지 fixture
- change-propagation 호출 후 `.html` 자동 재생성 fixture
- 직접 `.md` 편집 + `/regen-html` 수동 호출 후 `.html` 갱신 fixture
- `.html` 푸터에 stale 표시 (CH-id + 시각) 존재 검증

## Anti-patterns

- 메인이 .html subagent 결과 대기 → fire-and-forget 의미 무화
- `.html` 을 source of truth 로 사용 → 절대 X
- `.md` format-only pass 부활 시도 → v2.2.1 의 단순화 무화
- 모든 Edit 도구 호출 감지해서 자동 `.html` regen → 불완전 + 노이즈
- change-propagation 외 자동 hook 추가 (직접 편집은 수동 `/regen-html` 만)

## 결정 사항 / 미정

### 결정

- `.html` 만 생성, `.md` 가공 X
- fire-and-forget (메인 대기 X)
- `/regen-html <slug>` 는 **command** (skill X)
- change-propagation 마지막 단계 자동 호출
- 푸터 stale 표시 박음

### 미정

- v2.2.0 (.md 가공 + .html 병렬) 을 거치고 v2.2.1 로 진행 vs **v2.2.0 단계 건너뛰고 v2.2.1 안을 v2.2.0 으로 채택** (저자 의견: 후자가 깔끔)
- `.html` gitignore 여부 (v2.2.0 미정 항목 상속)
- 디바운스 윈도우 정확한 값 (1초? 3초?)
- subagent_type — 기존 general-purpose vs 신규 html-renderer 전용
- 실패 silent log 의 위치 (`/tmp/cc-html-regen.log` ?)

- 메인 turn 자체는 **거의 0 비용** (Task dispatch 만)
- subagent Sonnet 호출은 기존 docs-pretty 와 동일 (1회) — 비용 변동 X
- 디바운스 적용 시 연속 fix 케이스에서 비용 ↓ (마지막 fix 만 regen)
- v2.2.0 의 "2개 병렬" 비용보다 절반 (.md format-only pass 제거)
