# v1.1.14 후보 — designing-direction 7-topic adaptive 슬림화

> 작성: 2026-05-10
> 우선순위: 높음 — 사용자 catch ("개발디자인 계획서가 훨씬 더 느린 거같이 느껴져") 직접 대응. PRD adaptive rubric (v1.1.8 안 B) 와 동일 패턴 적용.

## 문제

`skills/designing-direction/SKILL.md:20` Step 3 가 무조건 7개 토픽 dialogue 진행:

> 아키텍처 (2-3 candidates) → 영향 컴포넌트 → 데이터 모델 → 외부 인터페이스 → 핵심 결정+대안 → 위험 → 테스트 전략

작은 피처 (1-3 task, UI 변경 / 리팩터링 / 내부 로직 정리) 도 7개 다 거치므로 **사용자 입력 1.5~2배** 누적. brainstorming PRD 6-section + designing-direction 7-topic 으로 단계 합치면 upstream brainstorming 대비 압도적으로 무거움.

## 해결 — Adaptive 7-topic (PRD adaptive rubric 패턴 차용)

### 항상 필요 (4개)

| # | 토픽 | 이유 |
|---|---|---|
| 1 | 아키텍처 | 모든 피처의 출발점 |
| 2 | 영향 컴포넌트 | cascading 의 기반 |
| 5 | 결정+대안 비교 | trade-off 추적 |
| 6 | 위험 (preliminary) | RISK 주석 단서 |

→ 어떤 피처든 4개는 필수.

### 조건부 (3개) — 자동 분류로 활성/스킵

| # | 토픽 | 활성 조건 |
|---|---|---|
| 3 | 데이터 모델 | requirements.md 에 DB/스키마 키워드 (테이블, 마이그레이션, 컬럼, supabase 등) 있을 때 |
| 4 | 외부 인터페이스 | requirements.md 에 API/event 키워드 (엔드포인트, REST, GraphQL, webhook, event, 외부 노출 등) 있을 때 |
| 7 | 테스트 전략 | task 추정 ≥ 4 (PRD 추정) 이거나 사용자 명시 요청 시 |

→ 비활성 토픽은 tech-design.md schema 에 "N/A — <skip 이유>" 한 줄로 박고 끝.

### 자동 분류 룰 (코드 헬퍼)

`scripts/design_topic_classifier.py` 신규:

```python
def classify_topics(requirements_md_path: Path) -> dict[str, bool]:
    body = requirements_md_path.read_text()
    return {
        "architecture": True,        # 항상
        "components": True,          # 항상
        "data_model": _has_schema_keywords(body),
        "external_interface": _has_api_keywords(body),
        "decisions": True,           # 항상
        "risks": True,               # 항상
        "test_strategy": _is_complex_feature(body),  # task 추정 >= 4
    }

def _has_schema_keywords(text: str) -> bool:
    keywords = ["테이블", "마이그레이션", "컬럼", "supabase", "schema", "DB", "데이터베이스"]
    return any(k in text for k in keywords)

def _has_api_keywords(text: str) -> bool:
    keywords = ["API", "엔드포인트", "REST", "GraphQL", "webhook", "event", "외부 노출"]
    return any(k in text for k in keywords)

def _is_complex_feature(text: str) -> bool:
    # PRD 의 task 추정 섹션에서 숫자 추출, 또는 본문 길이 임계값
    import re
    m = re.search(r'(\d+)\s*tasks?', text, re.IGNORECASE)
    if m:
        return int(m.group(1)) >= 4
    return len(text) > 3000  # fallback: PRD 본문 길이
```

→ 사용자 묻지 않고 자동 결정. designing-direction Step 3 진입 직전 호출.

## 추가 슬림화 — Step 2 Survey 중복 제거

`skills/designing-direction/SKILL.md:19` Step 2 가 brainstorming 에서 이미 식별한 영향 컴포넌트를 **또 grep/Read 하는 흐름**:

**AS-IS**:
> **Survey existing code** — Grep/Read impacted areas for each FR

→ 메인이 처음부터 grep 다시 호출. brainstorming PRD §2 (영향 받는 컴포넌트/파일) 결과를 활용하지 않음.

**TO-BE**:
> **Survey existing code** — Read `<slug>-requirements.md` §2 (영향 컴포넌트 목록) first. Only Grep/Read additionally if tech-design 결정 (아키텍처 / data flow / pattern) 에 필요한 깊이가 부족할 때.

→ brainstorming 결과 재활용 + 추가 필요시만 grep. 메인 토큰 + tool call 절감.

**완전 중복 아닌 부분 명시** (남겨둘 가치):
- brainstorming Survey: PRD 레벨 ("어떤 파일?")
- designing-direction Survey: tech-design 레벨 ("그 파일들의 구조 / pattern / data flow") — 결정 트리거 시점에만 추가 grep

- 소형: 7~14 → **4~8** (-40~50%)
- 중형: 7~14 → **5~10**
- 대형: 7~14 → **7~14** (동일)

## 위험 / 엣지케이스

- **classifier 잘못 분류** (소형으로 판단했는데 실제로는 데이터 모델 필요) → spec drift. 완화: verifying-spec gate 가 누락 검출. 또한 designing-direction 시작 시 "감지된 토픽: 1,2,5,6 — 데이터 모델 추가하시겠습니까?" 한 줄 안내로 사용자 override 가능.
- **classifier 키워드 미매칭** (예: "Postgres" 만 있고 "DB" 없는 경우) → keyword 리스트 보강 + fallback 룰. 또는 사용자 override 안내로 안전망.
- **PRD 본문 길이 임계값 (3000자) 가 임의** — task 추정 명시가 우선이고 길이는 fallback. 임계값은 PRD 통계로 보정 가능.
