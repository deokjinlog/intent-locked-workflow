# v2.0.2 setting-up-worktrees — 로컬 빌드 환경 파일 일반화

> **Version note (v2.0.2)**: `setting-up-worktrees` 가 현재 `.env*` 만 자동 복사 → web/Node 한정. Android (`local.properties`, keystore), iOS (`xcconfig`, `GoogleService-Info.plist`), desktop 등 빌드 환경에서는 워크트리 생성 후 수동 복사 필요. minor patch (skill 본문 패턴 list 확장 + 개념 rename) 이라 v2.0.2 적합.

## 한 줄 요약

`setting-up-worktrees` 의 auto-glob 룰을 `.env*` 한정 → **"로컬 빌드 환경 파일" 일반화**. 플랫폼별 known 패턴 list (Android / iOS / desktop / Node) 명시적 추가. 개념 rename ("env 파일" → "로컬 빌드 환경 파일"). HARD-GATE / Anti-Pattern 표현 동기.

## 배경 / 문제 (dogfood finding)

v2.0.0~v2.0.1 사용자가 Android Compose chat 화면 피처 작업 중 `/worktree` 발화. 결과:

- `.env*` glob → Android 프로젝트는 매치 0건 → "env 복사 skip" notice
- 실제 필요했던 파일들 (`local.properties` — SDK 경로, `keystore.properties` — 서명 키 경로, `*.keystore` — 서명 키) **모두 누락**
- 사용자가 워크트리 진입 후 수동 복사 — 워크트리의 zero-friction 가치 손실

같은 패턴 다른 플랫폼:

| 플랫폼 | 누락되는 파일 (현재) |
|---|---|
| Android | `local.properties`, `keystore.properties`, `*.keystore`, `*.jks`, (`google-services.json` 일부) |
| iOS | `*.xcconfig` (local), `GoogleService-Info.plist` (일부), `Secrets.swift` (gitignored 시) |
| Desktop / 일반 | `secrets.properties`, `secrets.json`, `config.local.*` |
| Node / Web | `.env`, `.env.local`, `.env.production`, `.envrc` |

### Root cause

`skills/setting-up-worktrees/SKILL.md` 본문 Step 2:

```bash
for f in "$ROOT"/.env*; do
    ...
done
```

glob 패턴이 `.env*` hard-coded. 다른 플랫폼 패턴 처리 부재. HARD-GATE / Anti-Pattern 표 / 한국어 출력 메시지 모두 "env 파일" 용어 한정.

## 제안 — 명시 패턴 list 확장 (Approach A 단독)

### 채택안 — known 패턴 list + 개념 rename

skill 본문 + commands/worktree.md 본문 변경:

1. **개념 rename**: "env 파일" → "로컬 빌드 환경 파일" 또는 "local config files"
2. **patterns list 명시**: Step 2 의 auto-glob 룰 확장:
   - Node/Web: `.env`, `.env.*`, `.envrc`
   - Android: `local.properties`, `keystore.properties`, `*.keystore`, `*.jks`
   - iOS: `*.xcconfig` (local-only, `GoogleService-Info.plist`, `Secrets.swift`
   - Desktop / 일반: `secrets.properties`, `secrets.json`, `config.local.*`
3. **exclude 패턴 유지**: `*.example`, `*.sample`, `*.template` (committed templates)
4. **추가 exclude (build artifact)**: 만약 (B) gitignore-based 자동 감지 도입하면 `*.log`, `dist/`, `build/`, `.DS_Score` 등 명시 제외. 단 본 release 는 (A) 단독이라 불필요.

### 보조 — `.gitignore` heuristic (v2.1+ 후속 검토, 본 release 미포함)

프로젝트 루트 gitignored 파일 자동 감지 + build artifact exclude list 결합. 결정론적 + 새 플랫폼 자동 cover.

**거부 이유 (본 release)**:
- false positive 비용 큼 (build output / IDE 디렉토리 복사 시 워크트리 오염)
- 명시 list 만으로 80% 이상 cover 충분

→ **defer to v2.1+** — 본 release (A) 운영 결과 보고 검토.

## 좋은 점 (정량/정성)

### 정량
- **다중 플랫폼 즉시 cover** — Android / iOS / desktop 사용자 워크트리 생성 즉시 빌드 가능
- **수동 복사 작업 0** — 워크트리 진입 → 바로 빌드. zero-friction 회복
- **dogfood 마찰 감소** — 다음 외부 피처 (Android / iOS / desktop) 마다 동일 패턴 반복 제거

### 정성
- **플랫폼 중립적 컨셉** — js-super 가 web 전용 도구가 아니라는 메시지
- **명시 list 의 가독성** — 사용자가 "내 플랫폼 cover 되나?" skill 본문에서 즉시 확인 가능

## 우려 사항 / 단점

### 1. 새 플랫폼 등장 시 list 수동 추가 필요

명시 list 라 Flutter / React Native / Rust 등 신규 플랫폼 등장 시 patch 필요.

**Mitigation**: README / SKILL.md 본문에 "신규 플랫폼 추가는 PR 환영" 안내. 또는 v2.1+ 의 gitignore-based 자동 감지 도입.

### 2. 민감 파일 (`*.keystore`, `*.jks`) 복사 시 보안 우려

서명 키 파일이 워크트리에 복사됨. 단 `.worktrees/` 가 gitignored 라 git 노출 0. 같은 머신 안 복사라 외부 노출 확대 없음.

**Mitigation**: 본 v0.x 의 `.env` 복사 정책 그대로 — "이미 디스크에 있는 파일, 같은 머신 안 복사" 원칙 유지. 사용자 명시 exclude (`X 파일은 복사하지 마`) 옵션은 기존 HARD-GATE 그대로.

### 3. iOS 의 `GoogleService-Info.plist` / Android 의 `google-services.json` — 일부 프로젝트는 committed

Firebase config 파일이 일부 프로젝트는 gitignored, 일부는 committed. 무조건 복사하면 committed 케이스에서 워크트리에 중복 파일 (git checkout 과 별도).

**Mitigation**: 룰 본문에 "committed 인 경우 git checkout 으로 자동 복원, gitignored 인 경우만 복사" 명시. 또는 단순화 — 무조건 복사 (덮어쓰기 0 + 같은 머신 안 동일 파일이라 정확성 유지).

### 4. 일부 플랫폼의 fragile 파일 (`local.properties` 의 절대 경로)

`local.properties` 는 머신 절대 경로 포함 (`sdk.dir=/Users/.../Android/sdk`). 다른 머신 워크트리 검토 시 깨질 수 있음 — 단 워크트리는 항상 같은 머신.

**Mitigation**: 워크트리는 same-machine 운영이라 깨짐 없음. 다른 머신 동기 의도면 별도 issue.

### 5. 본 release scope 비대 위험

만약 v2.1+ 의 gitignore-based 자동 감지까지 묶으면 scope 비대. **(A) 단독 유지** — list 확장 + rename 만.

## 다른 안 (대안 비교)

### 대안 1: 명시 patterns list 확장 (채택안) ← 본 release 적용

- 결정론적, 사용자가 본문에서 확인 가능
- skill 본문 + commands/worktree.md 변경만 (Python 코드 X)
- 새 플랫폼은 추가 patch — 단 평균 patch 비용 < 자동 감지의 false positive 비용

### 대안 2: `.gitignore` 기반 자동 감지 (defer to v2.1+)

루트 gitignored 파일 모두 자동 복사 + exclude list 결합. 플랫폼 무관.

**단점**:
- false positive (build output / IDE dir / `.DS_Store`) 잘못 catch 시 워크트리 오염
- exclude list 자체가 명시 list 비슷한 유지보수 필요
- (A) 와 결합 시 결과 동일하지만 복잡도 ↑

→ **defer to v2.1+**

### 대안 3: 사용자가 매번 `/worktree --env <files>` 명시 (거부)

기존 HARD-GATE "사용자에게 묻지 X" 정신 위배. zero-friction 워크트리 컨셉 손상.

→ **거부**

### 대안 4: 현상 유지 (거부)

`.env*` 한정 유지. 다른 플랫폼은 수동 복사.

→ **거부** — Android/iOS/desktop dogfood 마다 마찰 반복.
