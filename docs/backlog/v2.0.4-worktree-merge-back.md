# v2.0.4 후보 — `worktree-merge-back` 신규 skill (워크트리 → 상위 머지)

> 작성: 2026-05-16
> 우선순위: 중간 — 워크트리 작업 종료 시점의 반복 마찰 제거.

## 문제

js-super 의 `setting-up-worktrees` skill 로 워크트리에서 작업 끝나면, 사용자가 매번 수동으로:

1. 현재 워크트리에 commit
2. parent (main) 워크트리로 이동
3. feature 브랜치 merge
4. 충돌 나면 parent 쪽에서 해결 (위험 — main 더럽힘)

→ **충돌을 main 쪽에서 풀면 main 이 깨질 위험**. 매번 사용자가 신경 써야 함.

## 해결 — `worktree-merge-back` 신규 skill

### 핵심 패턴 (사용자 의도)

> "너는 워크트리에서 진행된 작업내용이야. 하던 일을 commit 하고 상위 워크트리에 머지했으면 해. 단 충돌 가능 — **먼저 상위 변경사항을 워크트리 쪽으로 가져와서 충돌 해결 후**, 다시 상위에 병합해."

→ "Merge down before merging up" 패턴. 충돌 해결은 **feature 워크트리 (sandbox)** 안에서만 발생. main 워크트리는 항상 깨끗.

### 흐름

**Step 1 — 현재 워크트리 commit (선택)**

- working tree 에 변경사항 있으면 사용자에게 commit 여부 확인 (AskUserQuestion)
- yes → 사용자가 commit 메시지 제시 후 commit
- no → "commit 안 한 채로는 머지 못 함" 알리고 중단

**Step 2 — parent 워크트리 추론**

- `git worktree list` 로 main 워크트리 path + branch 추출
- 현재 워크트리가 main 자체면 → "이 skill 은 워크트리 안에서만 사용 가능" 메시지 + 중단

**Step 3 — parent 변경사항 가져오기 (feature 쪽으로)**

- `git fetch origin` (parent branch 최신화)
- `git merge origin/<parent-branch>` (또는 사용자 선호에 따라 rebase)
- **충돌 발생 시**:
  - 충돌 파일 리스트 노출
  - 사용자에게 해결 요청 (메인 에이전트가 자동 해결 시도 X — 사용자 판단)
  - 사용자가 해결 후 commit 한 뒤 Step 4 진행
- 충돌 없으면 Step 4 자동 진행

**Step 4 — parent 워크트리로 이동 후 머지**

- parent 워크트리 path 로 `cd` (또는 `git -C <path>`)
- parent branch 가 머지 대상 (보통 `main`)
- `git merge <feature-branch>`
- **이 시점에 충돌 X 가 보장됨** (Step 3 에서 이미 흡수)
- merge commit 메시지 사용자 확인

**Step 5 — 사후 처리 (모두 선택, 명시 게이트)**

- 워크트리 제거? (`git worktree remove <path>`) — AskUserQuestion
- feature 브랜치 삭제? (`git branch -d <branch>`) — AskUserQuestion
- remote push? (`git push origin <parent-branch>`) — AskUserQuestion
- 모두 default `no` — 사용자가 명시 yes 만

### 안전성 보장 (자동 적용)

1. **워크트리 안에서만 동작** — main 워크트리에서 invoke 시 차단
2. **충돌은 항상 feature 쪽에서 해결** — main 워크트리는 절대 더럽히지 않음
3. **자동 충돌 해결 X** — 충돌 시 무조건 사용자 판단 (에이전트가 임의로 한쪽 선택 금지)
4. **사후 처리 (worktree 제거 / 브랜치 삭제 / push) 는 모두 별도 명시 게이트** — default no
5. **Step 별 실패 시 즉시 중단** — 다음 단계 자동 진행 X

### 유연성 (사용자 의도 보존)

- 머지 전략 (merge / squash / rebase) 사용자 선택 (AskUserQuestion)
- commit 메시지 사용자 입력
- 테스트 실행 여부 사용자 선택 (skill 본문에 강제 X)
- Step 별 사용자가 "직접 처리" 선택 가능 — skill 이 매 step 자율 진행 vs prompt 모드 분기

### 발동 트리거

명시 invoke 만 (`/worktree-merge-back` slash command 또는 skill 직접 호출). 자동 발동 경로 없음.

## Non-goals (이번 버전 안 함)

- 테스트 자동 실행 (사용자가 직접)
- pre-commit hook 우회 (절대 X)
- multi-parent 워크트리 (예: nested worktree) 지원
- merge conflict 자동 해결 (절대 X)
- force-push (parent push 는 일반 push 만)

## 결합 / 영향 범위

- `setting-up-worktrees` skill 변경 0
- `change-history` / `risk-annotation` 등 본 영향 0 (이 skill 은 git 조작만, MD 안 건드림)
- auto-* / og-* 영향 0
- preflight.py 영향 0 (필요 시 신규 helper 추가, 기존 4-skill 변경 X)

## 미정 사항

- 머지 전략 default? (merge commit / squash / rebase — 추천 squash 또는 merge commit)
- Step 1 commit 시 작성자가 메시지 직접 입력 vs 사용자가 다른 turn 에서 미리 commit 후 invoke 가정?
- parent 워크트리 추론 실패 시 (예: worktree list 비어있음, multi-parent) fallback 흐름
- 흐름 도중 abort 시 rollback 방식 (Step 3 머지 중간 abort → `git merge --abort`?)

## 검증

- 워크트리 안에서 실행 시 정상 흐름 1회 완주 fixture
- 충돌 발생 fixture (Step 3 에서 main 쪽 변경 + feature 쪽 변경 동일 파일)
- main 워크트리에서 invoke 시 차단 fixture
- 사후 처리 게이트 default no 확인
