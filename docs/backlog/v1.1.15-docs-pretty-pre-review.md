# v1.1.15 후보 — docs-pretty 발화 시점 통일 (post-approval → pre-review)

> 작성: 2026-05-10
> 우선순위: 높음 — 현재 설계 결함 (safety net 미발동) + UX 일관성. 비용 추가는 작음.

## 문제

`skills/docs-pretty/SKILL.md` 가 **doc 종류별로 발화 시점이 다름** — 일관성 결여 + safety net 미발동.

### 현재 시점 (`docs-pretty/SKILL.md:13~18`)

| 문서 종류 | 시점 | 회수 |
|---|---|---|
| **requirements.md / tech-design.md** | 사용자 승인 후, change-history 직전 | final-1회 |
| **implementation-plan.md** | verifying-spec 통과 후 + code-pretty 완료 후 + 사용자 리뷰 직전 | per-draft (재발화) |

### 결정적 결함

`docs-pretty/SKILL.md:8` 이 명시적으로:

> "This pass tightens visual hierarchy WITHOUT touching meaning, **so the user reviews a clean version**. The user's review is the safety net: **if the format pass ever drifts meaning, they catch it BEFORE it gets locked into change-history**."

→ **docs-pretty 본문이 "사용자가 pretty 된 버전을 리뷰한다"를 전제**. safety net = 사용자 리뷰가 의미 drift 검출.

그런데 현재 requirements/tech-design 흐름:

```
메인 작성 → 사용자 리뷰 (raw) → 사용자 승인 → docs-pretty → change-history
                                                    ↑
                                          사용자 검증 없음 (safety net 미발동)
```

→ docs-pretty 가 의미 drift 시켜도 잡을 사람 없음. 잘못 prettified 된 텍스트가 audit trail 로 그대로 박힘.

## 해결 — 모든 doc 종류에서 pre-review 통일 (Option B)

### TO-BE 흐름 (3 doc 모두 동일)

```
메인 작성 → docs-pretty → 사용자 리뷰 (prettified) → 승인 → change-history
                              ↑
                    사용자 검증 (safety net 발동)

수정 요청 시:
사용자 fix → 메인 revise → docs-pretty 재발화 → 다시 prettified 보여주기 (loop)
```

→ implementation-plan 이 이미 사용하는 per-draft 패턴 (`SKILL.md:16`). brainstorming / designing-direction 도 동일하게 맞춤.

### 안전성 — 이미 박혀있는 가드들로 충분

`docs-pretty/SKILL.md` 의 의미 drift 방지 가드:

| 가드 | 위치 | 효과 |
|---|---|---|
| "절대 의미 잃지 말 것" #1 priority | line 8 | 최우선 제약 |
| Sonnet 강제 (negative constraint 준수율) | line 44 | Opus/Haiku 회피 |
| Haiku 회피 (Korean prose rephrasing 위험) | line 50 | 모델 선택 |
| Forbidden list (reword/paraphrase/summarize/expand/"improve") | line 102~113 | prompt 룰 |
| 변경이력 footer byte-identical 강제 | line 110 | 감사추적 보호 |
| Post-dispatch sanity check (header count / frontmatter / footer) | line 158~169 | 자동 검증 |

→ 이 가드들 + 사용자 리뷰가 발동되면 의미 drift 사실상 0%.

## 영향 파일

- `skills/brainstorming/SKILL.md` — Step 6/7 순서 swap. Step 6 사용자 리뷰 전에 Step 7 docs-pretty 호출. 사용자 fix 시 re-pretty.
- `skills/designing-direction/SKILL.md` — Step 6/7 동일하게 swap. (현재 Step 6 = combined approval gate, Step 7 = docs-pretty)
- `skills/docs-pretty/SKILL.md` — frontmatter description 갱신: 모든 doc 타입에 대해 pre-review timing 으로 통일. Trigger timing 섹션 (`SKILL.md:13~18`) 단일화.
- `skills/docs-pretty/SKILL.md` Anti-Trigger 섹션 — 일부 룰 (예: "requirements.md / tech-design.md BEFORE user approval — wait for approval first" line 25) 제거.
- `skills/writing-plans/SKILL.md` — 이미 pre-review 패턴 사용 중. 변경 없음 (참조 정렬만).

## 위험 / 엣지케이스

- **dispatch 횟수 증가** — 사용자 수정 2-3회 가정 시 dispatch 1회 → 3-4회. Sonnet format-only 라 비용 적음 (수 초). implementation-plan 이 이미 같은 패턴이라 새 부담 아님.
- **메인이 첫 draft 에 pretty 적용 후 사용자 fix 받으면 raw 본문이 어디 있는지** — 메인의 in-memory 본문은 raw, 파일은 pretty. 사용자 fix 받으면 메인이 in-memory raw 갱신 후 다시 docs-pretty → 파일 덮어씀. 패턴 자체는 implementation-plan 에서 이미 검증됨.
- **change-history footer 가 pretty 후에도 byte-identical 인지** — `docs-pretty/SKILL.md:110` 룰로 강제. sanity check (line 167) 가 자동 검증.
