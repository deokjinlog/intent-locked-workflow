# v2.3.3 후보 — generating-html 의 핵심 내용 default-visible 강제

> 작성: 2026-05-22
> 우선순위: 높음 — v2.2.4 톤 자유 / 인터랙션 허용 룰이 핵심 PRD 본문을 가리는 회귀 발견.

## 문제 (사용자 신고)

ga4-chatbot-backend 프로젝트의 `canvas-merge-rule-requirements.html` (1260 lines) 검토 시 사용자 첫인상:

> "html에 원본기획서 알고 싶으면 저기로 가라고만 되어있네."

실제 검증 결과 html 본문에는 FR-1~5 + AC-1~7 + 배경 + 결정 비교 등 **PRD 전체 내용이 다 포함됨**. 그러나 **인터랙티브 UI 요소가 핵심 내용을 가림**:

- FR 카드가 `accordion` 패턴 (`.fr-body { display:none }` default, 첫 카드만 `DOMContentLoaded` 에서 auto-open)
- AC 체크리스트가 toggle 패턴
- 첫 로드 시 viewport 에 보이는 건:
  - Hero
  - 3 notices (For agentic workers / 연결 티켓 / **원본 기획서: ~/Downloads/...**)
  - 배경/목적 본문
  - **닫힌 FR-1 헤더 + 닫힌 FR-2~5 카드** ← 핵심 내용 안 보임
- 사용자는 스크롤 / 클릭 해야 비로소 내용 발견 → "안내만 있네" 라는 첫인상

### 회귀 원인 — v2.2.4 룰의 부작용

v2.2.4 (`generating-html 디자인 톤 자유 + 인터랙션 허용`) 에서:
- "정적 페이지만" 룰 해제 → `<details>` / tab / accordion 허용
- "Wow first" — variety = feature
- 디자인 톤 6종 자유 선택

→ 의도: 사람용 `.html` 의 시각 풍부도 ↑
→ 부작용: subagent 가 인터랙션을 적극 사용하면서 **핵심 PRD 본문도 hide-by-default** 패턴에 박힘.

`.md` 의 정보 hierarchy:
- 변경이력 footer, 메타 notices → secondary
- 배경 / FR / AC / 결정 / 범위 밖 → **primary**

`.html` 의 인터랙션 UI 패턴은 primary 와 secondary 를 구분 못 하고 **모든 항목을 동등 collapse 후보로 본다**.

## 해결 — `html-companion-prompt.md` 본문에 "primary content default-visible" 룰 박기

### 추가 룰 (3 항목)

#### 1. Primary content default-visible 강제

`.md` 의 다음 콘텐츠는 **첫 페이지 로드 시 시각적으로 노출** 되어야 한다:
- 배경 / 목적 / 문제 정의 본문
- 모든 FR (Functional Requirement) 항목 본문
- 모든 NFR 항목 본문
- 모든 결정 사항 본문 (alternatives 비교 포함)
- 모든 AC (Acceptance Criteria) 항목 본문
- 범위 밖 / Non-goals 본문
- 미정 사항 본문

→ 위 콘텐츠는 accordion / collapsed `<details>` / hidden tab / overlay modal 안에 **숨기지 마라**.

#### 2. Secondary content 만 진보적 공개 허용

다음 콘텐츠만 collapse / tab / `<details>` 사용 OK:
- 변경이력 (`## 변경이력`) footer entry — append-only audit log, 첫 로드 시 펼칠 필요 없음
- TOC / nav — sticky / collapsible OK
- code block 안의 긴 raw output — 첫 N줄 보이고 expand
- 다이어그램 raw source (Mermaid 코드 자체) — 렌더 결과는 보이고 코드는 collapse OK
- 메타 notices (티켓 ID / 작성자 / 외부 링크) — 노출하되 핵심 내용보다 작게

#### 3. "Wow first" 의 의미 재정의

v2.2.4 의 "Wow first" 는 **첫 viewport** 에서 발생해야 한다. 즉:
- Hero / 시각 임팩트는 OK
- 그러나 사용자가 스크롤 / 클릭 없이 **viewport 첫 1~2 화면 안에서 핵심 PRD 의 모양과 무게감이 전달**되어야 한다
- accordion 으로 모든 카드를 닫아둔 채 "스크롤하면 클릭하라" 는 패턴은 anti-Wow

## Anti-Patterns (prompt 본문에 추가)

| 안티 패턴 | 이유 |
|---|---|
| 모든 FR / AC 카드 default-closed accordion | 핵심 내용 가림. 사용자가 "안내만 있다" 라고 첫인상 받음. |
| 첫 카드만 auto-open, 나머지 closed | 동일. PRD 의 모든 FR 은 등가. 일부만 default-open 은 자의적. |
| `<details>` 안에 FR / AC / 결정 본문 wrap | `<details>` 는 secondary 전용. primary 내용에 사용 X. |
| Hero / decoration 만 첫 viewport 채우고 본문은 스크롤 아래 | "Wow first" 의 오용. 첫 viewport 에 핵심 내용 일부라도 보여야 함. |
| Tab 으로 FR 들을 가르고 한 번에 한 FR 만 보이기 | tab 은 secondary 전용 (예: "raw vs rendered"). primary 본문에 X. |
| AC 체크리스트가 toggle 으로 hide | AC 는 PRD 의 최종 검증 기준 = primary. 한눈에 다 보여야 함. |

## 검증

- 같은 PRD 로 v2.2.4 결과 vs v2.3.3 결과 비교 fixture — FR / AC 본문이 default-visible 인지 확인
- 첫 viewport (`window.innerHeight` 만큼) 스크린샷 비교 — 핵심 내용 일부라도 노출
- accordion / `<details>` / hidden tab grep — primary 콘텐츠 ID 에 매칭되면 fail
- 톤 자유 (v2.2.4) 검증 fixture — variety 가 유지되는지 별도 확인 (본 룰은 인터랙션만 제한)

## Non-goals (이번 버전 안 함)

- 인터랙션 자체 금지 (v2.2.4 톤 자유 룰 무화 X)

- 디자인 톤 default 강제 (6종 자유 그대로)
- 첫 viewport 의 정확한 px / 항목 수 강제 (LLM judgment)
