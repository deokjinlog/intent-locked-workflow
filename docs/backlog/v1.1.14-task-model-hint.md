# v1.1.14 후보 — Task별 모델 힌트 필드 (plan template + dispatch 자동화)

> 작성: 2026-05-10
> 우선순위: 중간 — v1.1.13 model pin의 진짜 응용. 큰 피처 dogfood 시 dispatch 비용 누적 체감 후 결정.

## 문제 (v1.1.13에서 부분만 응용된 상태)

v1.1.13에서 `js-super-subagent-driven-development/SKILL.md`에 Model Selection 섹션 추가하고 implementer/spec-reviewer prompt에 `model: "sonnet"` 기본 핀까지는 했는데, **upstream 4-조건 룰의 실제 응용은 미완**.

upstream 원본 (`skills/subagent-driven-development/SKILL.md:91, 98`):

> Mechanical implementation tasks (isolated functions, clear specs, 1-2 files) → **fast, cheap model**

→ 코드 중심 1-2 파일 mechanical task는 **haiku** 권장. 그런데 현재 흐름은 메인이 dispatch 직전에 task 성격 평가해서 haiku로 override해야 하고, 자동화 안 돼있어서 실전에서는 그냥 sonnet으로 뽑힐 가능성 높음 → upstream 룰의 cost saving 절반만 누림.

## 해결 — 옵션 A (plan 템플릿에 task별 모델 힌트)

### 변경 흐름

1. **`writing-plans/SKILL.md` task schema에 `model_hint` 필드 추가**

   현재 task block 예시:
   ```markdown
   ## Task N: <task name>
   - **Files**: <list>
   - **Tests**: <test strategy>
   - **Steps**: ...
   ```

   v1.1.14 추가:
   ```markdown
   ## Task N: <task name>
   - **Files**: <list>
   - **Model**: haiku | sonnet | opus  ← 신규
   - **Tests**: <test strategy>
   - **Steps**: ...
   ```

2. **plan 작성 시점에 task 성격 평가 룰**
   - 코드 중심 + 1-2 파일 + 명확 spec → `haiku`
   - 코드 다중 파일 / 디버깅 / 패턴 매칭 → `sonnet`
   - Korean prose 조작 (skill 본문 / MD 편집 task) → `sonnet` (Haiku rephrasing 리스크)
   - 설계 / 광범위 이해 / 리뷰 → `opus`

3. **dispatch 흐름** (`js-super-subagent-driven-development/SKILL.md` Per-task Sequence)
   - 메인이 task block에서 `Model:` 필드 읽음
   - dispatch 시 그 값을 implementer-prompt 의 `model:` 필드에 그대로 전달
   - `Model:` 필드 없으면 `sonnet` 기본값 (현재와 동일, backward compat)

### Spec reviewer 모델 결정

핵심 관찰: **spec review = 명세 텍스트 ↔ 코드 line-by-line 대조 + 누락/초과 분류**. 본질적으로 패턴 매칭에 가깝고 multi-step reasoning 거의 불필요. 즉 **implementer 가 mechanical task로 분류된 경우, spec review도 mechanical**.

- 단순 mechanical task의 spec review → **haiku 충분** (cost saving 큼, dispatch 비용 task당 2회 → 둘 다 haiku로 절감)
- 다중 파일 통합 / 미묘한 누락 가능성 / 패턴 매칭 복잡 → sonnet
- 보수적 디폴트는 sonnet (false ✅ = 안전망 무력화 → 인라인보다 위험)

→ **권장 정책**: spec-reviewer는 implementer hint를 상속받되 한 단계 위 안전장치 적용.
- implementer = haiku → spec-reviewer = haiku (대칭)
- implementer = sonnet → spec-reviewer = sonnet
- implementer = opus → spec-reviewer = opus 또는 sonnet (opus는 reviewer 비용 과함, sonnet도 충분)

prompt 매개화 또는 dispatch 시 매핑 룰로 처리. v1.1.14 PRD 단계에서 "한 단계 위 안전장치 적용" 룰 자체를 다시 검증.

(idea 출처: 사용자 — "스펙 리뷰는 fast model로 하면 되지않나". 정확한 직관 — implementer가 haiku로 충분하면 reviewer도 같은 task spec 기반이라 haiku 충분.)

## 위험 / 엣지케이스

- **plan에 `Model:` 잘못 적힌 경우** (e.g., haiku 박았는데 실제로는 multi-file integration task) → spec reviewer가 결과 검증하니 안전망 있음. 다만 dispatch 비용은 낭비될 수 있음.
- **메인이 plan 읽고 무시하는 경우** → plan 명세가 SOT라는 원칙 깨짐. dispatch 흐름에 명시적 STOP 게이트 추가 ("Model 필드 없으면 sonnet으로 진행, 사용자에게 한 줄 notice").
- **Korean prose 조작 task 식별 실패** → "이 task가 prose 조작인지 코드 작성인지" 룰을 writing-plans 본문에 명확히 박아둬야 함. 모호한 케이스는 sonnet 디폴트로.
