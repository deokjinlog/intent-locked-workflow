# v2.2.2 후보 — `.md` / `.html` pretty 명령 family 완성 + 스킬 리네임

> 작성: 2026-05-16
> 우선순위: 중간 — v2.2.1 의 후속 정합 + 사용자 친화 명령 family 정리.
> 전제: v2.2.1 (`docs-pretty` 를 `.html` 전용 fire-and-forget 으로 전환) 구현 완료.

## 배경

v2.2.1 에서 `docs-pretty` 가 `.md` format-only pass 책임을 잃고 `.html` 만 생성하게 됨. 그 결과:

1. 사용자가 가끔 `.md` 자체를 깔끔하게 정형해서 외부 공유하고 싶은 경우 (PR 본문 paste, 포트폴리오, 인쇄 등) 가공 수단이 사라짐 — 손으로 다듬어야 함
2. 기존 `docs-pretty` 라는 이름이 더 이상 책임을 대변 안 함 — `.html` 만 생성하는데 "docs pretty" 는 거짓말
3. `.html` 동기화 명령어 이름 `/regen-html` 이 "처음부터 다시" 뉘앙스라 매번 디자인이 다르게 나올 거라는 사용자 오해 발생 — 실제로는 디자인 보존 + 내용 sync 가 정상 사용
4. change-propagation 같은 `.md` 변경 흐름 끝에서 사용자가 `.html` stale 인지 못 함 — 손으로 `/sync-html` 호출 잊어버리기 쉬움

## 해결 — 5개 항목 atomic patch

### 항목 1 — `/pretty-md` 신규 커맨드

- `commands/pretty-md.md` 신규
- `/pretty-md <slug>` slash 트리거
- 인자: feature slug (e.g., `2026-05-16-foo`)
- 인자 누락 시 latest slug 추론
- 동작: `<slug>-{requirements,tech-design,implementation-plan}.md` 중 존재하는 모든 `.md` 에 대해 Sonnet subagent format-only pass dispatch
- 출력: 같은 `.md` 자리에 정형 본문 덮어쓰기
- 보존: `## 변경이력` footer **byte-equal** 보존
- 의미 보존 절대 (문장 / 항목 / 결정 사항 / 숫자 / 표 내용 1:1, 의역 / 요약 / 재구조화 X)
- opt-in 명시 호출만 — 자동 발동 경로 X

### 항목 2 — `/sync-html` 신규 커맨드

- `commands/sync-html.md` 신규
- `/sync-html <slug>` slash 트리거 + `--rebuild` 플래그
- 인자: feature slug
- 인자 누락 시 latest slug 추론
- 동작 (스마트 분기):
  - `.html` 없음 → 디자인 + 생성 (`generating-html` 스킬 신규 발동과 동등 경로)
  - `.html` 존재 → 기존 `.html` 을 subagent 에 같이 입력, **디자인 / 레이아웃 / 색·여백 보존 + 내용만 sync**
  - `--rebuild` 플래그 → `.html` 존재 무시, 디자인 처음부터 다시
- 실행 모드: **fire-and-forget subagent** — 메인 즉시 다음 turn

### 항목 3 — `docs-pretty` → `generating-html` 스킬 리네임

- `skills/docs-pretty/` → `skills/generating-html/` 폴더 이름 변경
- `skills/generating-html/SKILL.md` frontmatter `name` 필드 갱신
- 스킬 본문 자기 지칭 표현 (`docs-pretty` → `generating-html`) 전부 교체
- 실행 모드: **fire-and-forget subagent** (v2.2.1 기조 유지)
- 책임: v2.2.1 그대로 — `.html` 생성만, `.md` 가공 X
- 발동 boundary: 첫 change-history entry 까지 (v2.2.1 그대로)

### 항목 4 — `change-propagation` 마지막 단계 힌트

- `skills/change-propagation/SKILL.md` 본문 마지막 단계에 한 줄 추가
- 자동 호출 **X** — `.html` 자동 sync 안 함
- 출력: `"→ 이 변경으로 .html 이 stale 됐을 수 있음. 필요시 /sync-html <slug>"` 와 같은 안내 메시지
- 사용자 판단으로 호출 (stale 허용 철학 보존)

### 항목 5 — 명령어 참조 일괄 교체

`docs-pretty` / `regen-html` 단어가 참조된 모든 곳 교체:

- v2.2.1 본문의 `/regen-html` → `/sync-html`
- `.html` 푸터 stale 안내 문구 (`/regen-html` → `/sync-html`)
- CLAUDE.md 결합 메모 (`docs-pretty ↔ change-history`) 단어 갱신
- 다른 스킬 본문의 `docs-pretty` 참조 (executing-plans / writing-plans / designing-direction / brainstorming / auto-* / setting-up-worktrees 등) 일괄 교체
- og-* 스킬 영향 X (upstream mirror)

## 실행 모드 정리

| 컴포넌트 | 모드 | 메인 흐름 영향 |
|---|---|---|
| `generating-html` 스킬 | fire-and-forget subagent | 메인 즉시 다음 turn |
| `/sync-html` 커맨드 | fire-and-forget subagent | 메인 즉시 다음 turn |
| `/pretty-md` 커맨드 | fire-and-forget subagent | 메인 즉시 다음 turn (또는 user 가 결과 보고 결정하면 동기 — 미정) |

`.md` / `.html` 모든 pretty 처리는 **메인 흐름 비차단** 원칙.

## 영향 범위

- 신규 파일: `commands/pretty-md.md`, `commands/sync-html.md`, `skills/generating-html/SKILL.md` (구 docs-pretty 이전)
- 삭제: `skills/docs-pretty/` 폴더 (이전 후 삭제)
- 본문 변경: change-propagation SKILL.md (힌트 1줄 추가), CLAUDE.md (결합 메모 단어 교체), 기타 docs-pretty 참조 스킬 다수
- og-* / auto-* (auto-brainstorming/designing-direction/writing-plans/executing-plans) — 본문 영향 0
- preflight.py 영향 0 (docs-pretty 의 pre-flight 룰은 generating-html 로 그대로 이전)

## Non-goals (이번 버전 안 함)

- `/pretty-md` + `/sync-html` 결합 호출 (`/pretty-both` 같은 슬래시) — v2.2.3 후보
- `.md` format-only pass 의 자동 발동 부활 — 절대 X (v2.2.1 단순화 무화)
- `.html` 디자인 versioning (이전 디자인으로 롤백) — 안 함
- `.html` ↔ `.md` 양방향 sync — 절대 X, `.md` 가 source

## 검증

- `/pretty-md <slug>` 호출 시 `.md` format-only 적용 + 변경이력 footer byte-equal 보존 fixture
- `/sync-html <slug>` 첫 호출 (`.html` 없음) → 디자인 + 생성 fixture
- `/sync-html <slug>` 두번째 호출 (`.html` 있음) → 디자인 보존 + 내용만 sync fixture (시각 hash 비교)
- `/sync-html <slug> --rebuild` → 디자인 변경 가능 fixture
- change-propagation 호출 후 마지막 메시지에 `/sync-html` 안내 포함 fixture
- `docs-pretty` 단어 grep 결과 0 (og-* / 백로그 히스토리 제외) fixture
- 모든 발동이 fire-and-forget — 메인 turn elapsed 차이 < 100ms fixture

## 미정 사항

- `/pretty-md` 의 디바운스 정책 (사용자가 연속 호출 시) — 일단 디바운스 X, 매번 dispatch
- `/sync-html --rebuild` 가 정말 자주 쓰일지 dogfood 후 결정 (안 쓰이면 제거 후보)
- `/pretty-md` 결과를 사용자가 보고 적용 / 거부 게이트가 필요할지 (현재 안: 그냥 덮어쓰기, 사용자가 git diff 로 확인)
- 마이그레이션 — 기존 사용자가 `docs-pretty` / `regen-html` 호출 시 deprecated 메시지로 안내할지

## 비용

- `/pretty-md` 호출 1회당 Sonnet 1회 (기존 docs-pretty `.md` 가공과 동일)
- `/sync-html` 호출 1회당 Sonnet 1회 (v2.2.1 그대로)
- change-propagation 마지막 힌트 1줄 추가는 메인 turn 비용 거의 0

## 결합 / 회귀 catch

- 5개 항목 atomic patch — 한 곳만 누락 시 회귀
- 단어 일괄 교체 grep:
  ```
  grep -rn "docs-pretty\|regen-html" skills/ commands/ CLAUDE.md
  ```
  → og-* / 백로그 히스토리 제외하고 0 보장
