# A. 안내 톤 쉽게 (한국어 친화)

## 문제 (사용자 신고)

> "그리고 전체적으로 그 안내하는 말들이 너무 어렵게 말하는경향이 있어.."

회귀 패턴 — 메인이 사용자에게 안내 / 보고 / 결과 요약 시 다음 같은 톤:

- 영어 식별자 / 약어 다발 (`AskUserQuestion`, `byte-copy`, `plan_byte_check`, `dispatch`, `boilerplate`, `mirror`, `fire-and-forget` 등)
- 압축된 한 줄 / 슬래시 / 콜론 다발 (`5 task / 3 wave / 9 commits / pytest 61 PASS`)
- 한 문단에 정보 5+개 (배경 + 결정 + 영향 + 검증 + 다음 단계 한 줄)
- 메모 형식 (`OK 진행`, `진행 시작`, `- ✅`, `→ ...`)
- 한국어 / 영어 mix (`auto-flow chain`, `fire-and-forget dispatch`)

### 영향

- 사용자가 읽기 어려움 — 빠른 훑기 X, 매번 멈춰서 해석
- 초보자 / 새 합류자 — 진입 장벽 ↑
- 회의 / 보고 시 그대로 인용 어려움
- 인지 부담 ↑

## 해결 — "한국어 친화 안내 톤" 룰 (모든 skill body + 메인 응답)

### 룰 A-1: 사용자 안내는 짧은 한국어 문장

- 영어 식별자 / 약어는 **꼭 필요할 때만** + 처음 등장 시 짧은 한국어 설명 동반
- 한 문장에 정보 1~2개. 한 문단에 4 문장 이하
- 한 줄 압축 보고 (`5 task / 3 wave / ...`) 보다 표 / 불릿 + 줄 띄움

### 룰 A-2: 메모 / 슬래시 / 콜론 다발 금지

- "OK 진행", "→ ...", "✅ ..." 등 메모 패턴은 진행 노트 (한 줄) 만 OK
- 사용자 결과 보고는 완전 문장 ("...완료했습니다." / "...에서 회귀 발견했습니다.")
- 콜론 다발 (`결과: X / 다음: Y`) 보다 풀어쓰기

### 룰 A-3: 한국어 / 영어 mix 최소화

- 사용자 시야 표현은 한국어 우선
- 영어는 (a) 도구 / 함수 / 파일 이름 (식별자), (b) 짧은 약어 (CLI / API 등) 만
- "fire-and-forget" → "백그라운드 호출", "dispatch" → "호출", "byte-copy" → "원본 그대로 보존" 등

### 룰 A-4: 보고 = 사용자 친화 톤

- 사용자에게 보고할 때:
  - 무엇을 했는지 (1~2 문장)
  - 왜 그렇게 했는지 (필요 시 1 문장)
  - 다음에 무엇이 필요한지 (1 문장)
  - **그 외 디테일은 자세히 묻기 전 생략**

### 룰 A-5: skill body 안 한국어 영역 vs 메인 응답 영역 구분

- skill body 의 **식별자 / 코드 패턴 / 룰 본문** — 영어 OK (기존 답습, 메인 prompt 가공)
- skill body 의 **사용자 노출 문구** (안내 / Anti-Patterns 표 / Why 본문 등) — 한국어 친화
- 메인이 사용자에게 직접 보고하는 응답 — 한국어 친화 강제

## 영향 범위 (광범위)

본 룰 변경의 직접 대상:

- 모든 js-super skill body 의 사용자 노출 문구 (수십 개)

# B. generating-html 비동기 생성 신뢰성

## 문제 (사용자 신고)

> "그리고 비동기로 html 생성이 안되는경우가 있어 (처음 md 생성시)"

회귀 패턴 — `auto-brainstorming` / `auto-designing-direction` / `auto-writing-plans` Step 4.5/4.6 의 fire-and-forget dispatch 가:

- **처음** md 생성 시점 (변경이력 footer entry 0건) 에 가끔 실패
- 메인은 dispatch 후 즉시 다음 Step 진행 → background subagent fail 인지 못 함
- `.html` 결과물 부재 → 사용자가 검토 surface 없음

## 회귀 원인 (추정)

가능한 원인 3가지:

1. **race condition**: dispatch 직후 메인이 change-history skill 호출 → footer entry 추가 → background subagent 가 그 시점에 md 읽음 → footer 1건 인지 → generating-html "live doc 룰" (footer 1건+ → 발동 skip) 적용 → silent skip
2. **dispatch 자체 실패**: Agent tool background launch 가 가끔 fail (subagent id 안 받음). 메인이 dispatch return 만 보고 next step 진행
3. **subagent 내부 silent error**: B subagent 가 md 읽기 / 분석 / write 중 silent fail. 결과 보고만 보고 메인 인지 X

## 해결 (안)

### 룰 B-1: dispatch 후 retry / verify

- background dispatch 직후 **dispatch id 확인** (return 받았는지)
- 일정 시간 후 (예: 5분) `.html` 파일 존재 확인 → 없으면 메인이 직접 호출 (sync fallback)

### 룰 B-2: race condition 해결 — Step 4.5 ↔ change-history 순서 보장

- 현재: dispatch (fire-and-forget) → change-history (footer 1건) → race
- 변경: dispatch 후 short delay (예: 5초) → change-history. background subagent 가 footer 0건 시점에 md 읽도록 보장

### 룰 B-3: silent log 모니터

- `.js-super/html-regen.log` 에 dispatch / success / fail entry 기록
- 사용자가 `/sync-html --check` 또는 `tail .js-super/html-regen.log` 로 확인 가능

### 룰 B-4: 메인 응답에 dispatch 결과 명시

- "✅ requirements.html 백그라운드 생성 시작" → "✅ requirements.html 백그라운드 완료 (41KB)" 또는 "⚠️ 실패 — 5분 뒤 재시도 필요"
- transition notice 시점에 명시

## 영향 범위

- `skills/generating-html/SKILL.md` — fire-and-forget 룰 보정 (B-2)
- `skills/auto-brainstorming/SKILL.md` Step 4.5 — race 해결
- `skills/auto-designing-direction/SKILL.md` Step 4.5 — race 해결
- `skills/auto-writing-plans/SKILL.md` Step 4.6 — race 해결
- `commands/sync-html.md` — `--check` 옵션 추가 (B-3)
- 알람 시스템 (`.js-super/html-regen.log`) 도구 추가 가능 (옵셔널)

## 미정

- short delay 정확 값 (3초? 5초? 10초?)
- sync fallback 의 trigger 조건 (5분 후? 사용자 명시 호출만?)
- silent log entry 형식 / rotation 정책
- A 항목과 B 항목 release 순서 — 묶음 vs 분리 (저자 의견: 묶음 — v2.4 메이저 의도)
