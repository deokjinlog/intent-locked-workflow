# v2.0.3 후보 — js-super 신규 skill 의 모든 사용자 질문을 AskUserQuestion 도구로 강제

> 작성: 2026-05-16
> 우선순위: 높음 — 알람 누락으로 사용자가 게이트를 놓치는 실사용 마찰.

## 문제

js-super 가 신규로 만든 4 skill (`brainstorming` / `designing-direction` / `writing-plans` / `executing-plans`) 의 흐름 도중, **사용자에게 질문하는 지점이 다수** 존재. 주요 명시 게이트 (예: `Gate #11` 통합 승인, `Gate #14` 실행 모드 선택 등) 는 이미 `AskUserQuestion` 도구로 보장됨.

그러나 **흐름 중간의 즉흥 clarifying 질문 / 소크라테스식 질문 / 모호점 확인** 같은 비-게이트 질문은 **prose (산문 한 줄)** 로 던지는 경우가 남아 있음. 이때:

- macOS `Notification` 훅의 `elicitation_dialog` 매처가 발화 X
- → `~/.claude/scripts/repeat-alert.sh` (30초 반복 알람) 발화 X
- → 사용자가 다른 작업 중이면 **질문 자체를 놓침** → 흐름 중단 / 사용자 마찰

실사용 시나리오 (사용자 신고):
> "그 yes/no 받을때 뭔가.. 그 도구호출은 하지않고 그냥 'cycle3-tech-design.md is finalized. Proceed?' 이렇게 질문하니까 ... 사용자가 인지하기가 힘들어" (v1.1.8 신고 재발)

## 해결 — `AskUserQuestion` 강제 룰 4 skill 본문 박기

### 룰 (요약)

**4 skill 본문 (brainstorming / designing-direction / writing-plans / executing-plans) 어느 단계에서든 사용자에게 질문할 때는 반드시 `AskUserQuestion` 도구를 호출해야 한다.**

- 산문 질문 (`"X 할까요? — yes / no"`) 금지
- clarifying / Socratic / 모호점 확인 / 게이트 / 어떤 질문이든 적용
- 메인 에이전트가 자율 판단해서 prose 로 묻는 모든 케이스 차단

### 신규 boilerplate (4 skill 본문 상단에 박을 룰)

```markdown
## 사용자 질문 룰 (v2.0.3+) — 항상 AskUserQuestion

이 skill 흐름 안에서 사용자에게 질문할 일이 생기면 **반드시** `AskUserQuestion`
도구로 호출한다. 산문으로 "~ 할까요?" 한 줄 던지지 마라.

### Why

Notification 훅 (`elicitation_dialog` 매처) 이 알람을 발화하려면 도구 호출이
실제로 일어나야 함. 산문 질문은 훅이 못 잡아서 사용자가 놓침.

### How to apply

- clarifying / Socratic / 모호점 확인 / 게이트 / 모드 선택 — 모두 포함
- 단답 yes/no 도 prose X → `AskUserQuestion` choices `[yes, no]` 사용
- 다중 선택은 `multiSelect: true` 또는 enum choices
- 사용자가 평문 응답이 자연스러운 경우 (예: Socratic 자유 답변) 에만 prose 허용 — 단 그 경우에도 **첫 prompt 는 AskUserQuestion 로 트리거** + 본문에 "자유 응답 가능" 명시
```

### 영향 범위

- **brainstorming**: Step 0 entry router (PRD/Socratic 선택), Socratic Q 다수, clarifying Q, 최종 승인 게이트 — 모두 `AskUserQuestion` 화 (일부는 이미 적용, 누락분 채우기)
- **designing-direction**: 7-topic 토론 중 clarifying, adaptive 토픽 필요 여부 확인, alternatives 선택, verify 보고서 후 통합 승인 게이트 — 동일
- **writing-plans**: task 분해 중 clarifying, RISK 코드 지점 확인, verify 후 통합 승인 게이트, 실행 모드 선택 게이트 — 동일
- **executing-plans**: mode-check 게이트 (이미 적용), per-task 도중 모호점 확인, finishing handoff — 동일

### auto-* skill 영향

`auto-*` 4 skill 은 본 룰의 **예외** — auto-* 의 디자인 의도는 "사용자 입력 최소" 이므로 clarifying Q 자체를 거의 안 던짐. clarifying Q 가 꼭 필요한 케이스 (사용자가 fast-forward 모드를 명시 invoke 하지 않은 경우) 에만 prose 허용. 또는 본 룰을 똑같이 적용해 일관성 확보 — **결정 미정** (v2.0.3 작업 시 dogfood 후 결정).

### og-* skill 영향

upstream mirror 보존. 변경 X.

### fast-tasks (v2.1.0 후보) 영향

신규 skill 도 본 룰 따르도록 v2.1.0 작업 시 본문에 동일 boilerplate 박음.

## 검증

- 4 skill 본문 grep — 산문 질문 패턴 (`"\\? — yes / no"`, `"진행할까요"` 등) 잔재 0
- AskUserQuestion 호출 카운트 — 4 skill 본문 합산 N 회 (수치 dogfood 후 baseline 확정)
- 사용자 한 turn 안에 prose 로 던진 질문 0 (Socratic 자유 응답 모드 제외)

## 비용

- `AskUserQuestion` 호출당 elicitation_dialog 발화 → 알람 1회
- 호출 빈도 ↑ 시 알람 피로 가능 — repeat-alert.sh 의 카운터 / max=10 정책으로 흡수 (이미 설정됨)
- 첫 호출에 사용자 응답 늦으면 30s × N 알람 — 단 multi-question batching 으로 완화 가능 (한 호출에 questions[] 다수)

## 미정 사항

- multi-question batching 가이드 (한 turn 에 묶을지 분리할지)
- Socratic 자유 응답 모드의 첫 트리거 정확한 형식 (single Q AskUserQuestion + "자유롭게 답해주세요" label?)
- auto-* skill 의 룰 적용 여부 (전체 적용 vs 예외 유지)
