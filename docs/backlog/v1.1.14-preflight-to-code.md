# v1.1.14 후보 — Pre-flight check 를 LLM 에서 코드로 이관

> 작성: 2026-05-10
> 우선순위: 중간 — 비용/정확도 효과 명확. v1.1.14 묶음 작업 시 함께 진행 권장.

## 문제

여러 skill 이 본 작업 직전에 `Pre-flight check` 라는 사전 검증 단계를 두고 있고, 현재 **메인 에이전트 (보통 Claude Opus 4.7) 가 직접 LLM 으로 처리** 함. 그런데 검증 내용이 본질적으로 결정론적이라 LLM 이 할 일이 거의 없음.

### 실제 Pre-flight 작업 (예: docs-pretty)

`skills/docs-pretty/SKILL.md:54~62`:
1. target file 존재? → `ls` 한 줄
2. `## 변경이력` footer entry 0 개? → `grep` 한 줄
3. filename 이 `<slug>-{requirements,tech-design,implementation-plan}.md` 패턴? → 문자열 매칭

→ **bash 두세 줄로 끝나는 boolean check**. LLM 추론 불필요.

다른 skill 의 pre-flight 도 비슷한 성격:
- `executing-plans` mode-check: commit_policy 필드 + git rev-parse 가용성
- `js-super-subagent-driven-development` task entry: BASE_SHA 캡처 + commit_policy 검증
- `risk-annotation`: 3-checklist 적용 (이건 LLM 추론 필요 — 제외 대상)

## 해결 — 코드 helper 로 이관

js-super 가 이미 v1.1.7 에서 `scripts/changelog_buffer.py` + `scripts/tests/` 패턴을 도입했으므로 동일 패턴 따라감.

       ok: bool
       reason: str

       if not file_path.exists():
           return PreflightResult(False, "file not found")
       if _has_changelog_entries(file_path):
           return PreflightResult(False, "변경이력 footer not empty (doc is live)")
       if not _matches_feature_md_pattern(file_path):
           return PreflightResult(False, "filename doesn't match feature MD pattern")
       return PreflightResult(True, "ok")

   def execute_plan_mode_check(plan_path: Path) -> PreflightResult:

       ...

   def subagent_task_entry_check(...) -> PreflightResult:
       # BASE_SHA 캡처 + commit_policy 검증
       ...
   ```

2. 각 SKILL.md 의 Pre-flight 섹션을 `python -c "..."` 호출 한 줄로 대체

   AS-IS (`docs-pretty/SKILL.md:54~62`):
   ```
   ### Step 1 — Pre-flight check
   Before dispatching, the main agent MUST verify:
   1. The target file exists (Read or Glob)
   2. The file's `## 변경이력` footer has ZERO entries (Grep)
   3. The file is one of the three feature MDs
   If ANY check fails → DO NOT dispatch.
   ```

   TO-BE:
   ```
   ### Step 1 — Pre-flight check
   ```bash
   python -c "
   from pathlib import Path
   from scripts.preflight import docs_pretty_check
   result = docs_pretty_check(Path('<TARGET>'))
   print(f'ok={result.ok} reason={result.reason}')
   exit(0 if result.ok else 1)
   "
   ```
   → exit code 0 일 때만 dispatch. 실패 시 reason 메시지 사용자에게 노출.
   ```

3. `scripts/tests/test_preflight.py` 신규 — 모든 케이스 fixture 단위 테스트

## 효과 (예상)

| 항목 | 현재 (LLM) | 변경 후 (코드) |
|---|---|---|
| 정확도 | 99% (gerp 누락 / 패턴 오인 가능) | **100% (deterministic)** |
| 속도 | 수 초 (LLM 응답) | **밀리초** |
| 비용 | pre-flight 당 수백 토큰 | **0 토큰** |
| 디버깅 | 매번 다른 응답 | **재현 가능** |
| 휴먼 에러 | LLM 이 "대충 통과" 판단 가능 | **룰대로만 작동** |

피처당 pre-flight 총 호출 수 (전체 워크플로 기준):
- brainstorming → docs-pretty pre-flight (1)
- designing-direction → docs-pretty pre-flight (1)
- writing-plans → code-pretty + docs-pretty pre-flight (2)
- /execute-plan 진입 mode-check (1)
- subagent task entry × N (task 수만큼)

→ 평균 5~10 회 / 피처. 모두 LLM 영역에서 빠지면 **token 절감 + 응답 속도 개선** 누적 효과 큼.

## 위험 / 엣지케이스

- **scripts/preflight.py import 실패** (파이썬 환경 문제) → exit code != 0 → STOP. 명확한 실패 메시지로 사용자에게 환경 점검 안내.
- **pre-flight helper 자체 버그** → fixture pytest 로 잡힘. 회귀 방지.
- **3-checklist (risk-annotation) 는 LLM 영역 유지** — 이건 결정론적이지 않고 코드 의미 추론 필요. 코드로 빼면 안 됨. 본 PRD 범위에서 명시적으로 제외.
