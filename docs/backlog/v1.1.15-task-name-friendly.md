# v1.1.15 후보 — Task 목록 이름 사용자 친화화

> 작성: 2026-05-10
> 우선순위: 중간 — UX 마찰 직접 노출. 단순 rename 작업이지만 일관된 적용 필요.

## 문제

skill Checklist 본문에 박힌 **내부 용어 (Gate #, CH-id, skill 명)** 가 TaskCreate task 이름으로 그대로 노출됨. 사용자 시점에서 의미 파악 어려움.

### 사용자 catch (2026-05-10 세션)

> "태스크 목록에 나오는 명칭같은게 너무 사용자 친화적이질 못해;; 예를들면 P10. change-history CH-003 이렇게 떳거나 P11. Gate #14 + 실행 핸드오프 이런식으로.. 웬 게이트 ? 가 포함되었거나 등"

### 현재 jargon 노출 사례

`skills/brainstorming/SKILL.md:34`:
> 8. **Invoke change-history skill** — append first `[요구사항-수정]` entry

→ TaskCreate 시 "Invoke change-history skill" 이 그대로 사용자에게 보임. "change-history" / "skill" / "Invoke" 모두 내부 용어.

`skills/writing-plans/SKILL.md:330`:
> Gate #13 — plan + verify 결합 승인

→ "Gate #13" 이 사용자에게 무의미한 내부 식별자.

`skills/writing-plans/SKILL.md:360`:
> Gate #14 — 실행 모드 선택

→ 동일 문제.

### 패턴

| 내부 용어 | 사용자 친화 표현 (제안) |
|---|---|
| "Invoke change-history skill" | "변경이력 기록" |
| "Invoke docs-pretty skill" | "문서 포맷 정리" |
| "Invoke code-pretty skill" | "코드 블록 포맷 정리" |
| "Invoke verifying-spec" | "사양 정합성 검증" |
| "Gate #N — X" | "X" (Gate 번호 제거, 의미만 노출) |
| "change-history CH-NNN" | "변경이력 기록" |
| "Run mode-specific dialogue" | "요구사항 질문 진행" |
| "Self-review" | "자체 검토" 또는 그대로 |

## 해결 — 2단계 접근

### 단계 1: skill Checklist 본문 rename

각 skill 의 Checklist 섹션 (TaskCreate 가 직접 참조) 을 사용자 친화 이름으로 재작성. **단, 본문의 다른 부분 (Process Flow diagram, Detailed sections) 의 내부 용어는 그대로 유지** — 메인 에이전트가 정확히 어떤 skill 호출하는지 알아야 하므로.

예시 — `skills/brainstorming/SKILL.md` Checklist:

**AS-IS** (line 23~36):
```
1. **Explore project context** — files, docs, recent commits
2. **Confirm feature name + slug** — one question, then create folder
3. **Mode selection gate** — ask user PRD or Socratic
...
8. **Invoke change-history skill** — append first [요구사항-수정] entry
9. **Auto-proceed to designing-direction (v1.1.9+ — gate removed)** — ...
```

**TO-BE**:
```
1. **프로젝트 컨텍스트 탐색** — files, docs, recent commits
2. **피처 이름/슬러그 확정** — 한 번 묻고 폴더 생성
3. **모드 선택** — PRD 또는 소크라테스식
...
8. **변경이력 기록** — 요구사항 첫 entry append
9. **개발방향 단계 자동 진행** — 사용자 인터럽트 가능
```

→ 메인 에이전트는 본문 내 detailed step 으로 정확한 동작 알 수 있음. Checklist 는 사용자 시야 (TaskCreate UI) 라 친화적 표현만.

### 단계 2: TaskCreate 작성 시 가이드 추가

각 skill 본문 상단에 한 줄 가이드:

> **TaskCreate 명칭 룰**: 사용자 시야에 노출되는 task 이름은 한국어 + 친화적 표현. 내부 용어 (skill 명, Gate #, CH-id) 는 task 본문 description 또는 details 에만.

또는 글로벌 룰을 `skills/using-superpowers/SKILL.md` 또는 `CLAUDE.md` 에 박기 (모든 skill 에 일괄 적용).

## 위험 / 엣지케이스

- **메인 에이전트가 Checklist 만 보고 정확한 skill 호출 못할 위험** — Checklist 는 친화적 이름이라도 본문 내 Detailed Step 또는 Process Flow 다이어그램에 정확한 skill 명 명시. 메인은 본문 전체 읽으므로 매칭 가능.
- **rename 일관성** — task-name-friendly 룰을 한 곳 (CLAUDE.md) 에 정의하고 각 skill 에서 참조. 신규 skill 작성 시도 그 룰 따르도록.
- **upstream 패턴과 충돌** — upstream 의 og-* skill 들은 손대지 않음 (upstream verbatim). js-super 자체 skill 만 적용.
- **변경이력 footer 의 entry tag** (`[요구사항-수정]` 등) 는 schema 매직 키워드라 유지. task 이름과 별개.
