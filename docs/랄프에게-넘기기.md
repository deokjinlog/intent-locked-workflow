# 랄프에게 넘기기 — ① 만 만들고 ②③④ 맡기기

**당신은 요구사항까지만 만듭니다. 나머지는 자고 일어나면 돼 있습니다.**

```
당신 │ 기획 자료 → /brainstorming → ① requirements.md → 승인   ← 여기까지만
     ▼
랄프 │ ② 개발방향 → ③ 구현계획서 → ④ 코드                     ← 무인
```

`ralph-loop` 는 **Stop 훅으로 종료를 막고 같은 프롬프트를 다시 먹이는** 플러그인입니다. 그래서 매 반복 **기억이 없고**, 진척은 **파일에만** 남습니다.

---

## 먼저 — 뭘 잃는지 알고 시작하세요

| 잃는 것 | 뭔 뜻인가 |
|---|---|
| **② 의 기술 선택권** | `auto-tech-design` 은 *"alternatives 비교 → **recommendation 자동 선택**"* 입니다. 비교는 하지만 **고르는 것도 자기가** 합니다. 라이브러리·아키텍처를 직접 고르고 싶으면 이 방식은 아닙니다. |
| **③ 을 검토할 사람** | 랄프가 쓴 ③ 을 랄프가 실행합니다. **자기가 쓴 걸 의심하지 않아요.** ③ 의 완료 조건이 틀려도 그대로 쫓아가서 "다 됐다"고 합니다. |

> **실제로 있었던 일** — 사람이 쓰고 승인까지 받은 ③ 에서, 나중에 대조하다 **완료 조건의 테스트 건수가 22 인데 실제는 25** 인 걸 발견했습니다 (한 task 의 테스트 3건이 집계에서 누락). 랄프였다면 22개만 통과해도 완료를 선언했을 겁니다.

**그래서 ① 이 좋아야 합니다.** 아무도 뒤에서 안 고쳐줍니다.

## 전제 — ① 이 **완성**돼 있어야 함

랄프를 붙이기 전에 이게 참인지 확인하세요.

```bash
F=docs/features/2026-07-15-내기능        # 당신 피처 폴더
sed -n '/^## 변경이력/,$p' "$F"/*-requirements.md | grep -c '^- \*\*id\*\*:'
# 1 이상이어야 함 — 0 이면 아직 승인 전이다
```

**변경이력 엔트리가 그 문서의 완성 표식**입니다. 이 도구 자신이 그 신호를 씁니다:

> *"Look for an existing `## 변경이력` footer with one or more entries. **If ANY entry exists, this is NOT initial creation.**"*
> — `skills/generating-html/SKILL.md`

## `PROMPT.md` — 복사해서 쓰세요

프로젝트 루트에 `PROMPT.md` 로 저장하고, `<날짜>-<슬러그>` 와 §4 완료 조건만 바꾸면 됩니다.

````markdown
# <프로젝트 이름> — 무인 실행 지시서

> 이 파일은 랄프 루프가 **매 반복 읽는** 지시서다. 프롬프트는 반복 사이에 **안 바뀐다.**
> 너는 매 반복 **기억 없이** 시작한다. 진척은 **파일에만** 있다.

## 0. 지금 어느 단계인가 — 먼저 판정해라

추측하지 마라. **읽어서 확인해라.**

```bash
F=docs/features/<날짜>-<슬러그>

for d in requirements tech-design implementation-plan; do
  f=$(ls "$F"/*-$d.md 2>/dev/null)
  if [ -z "$f" ]; then echo "$d: 없음"
  else echo "$d: 변경이력 $(sed -n '/^## 변경이력/,$p' "$f" | grep -c '^- \*\*id\*\*:')건"; fi
done

git log --oneline -10
cat BLOCKED.md 2>/dev/null || echo "(막힌 것 없음)"
```

**판정 규칙 — 변경이력 엔트리가 1건 이상이면 그 문서는 완성이다.**
파일만 있고 엔트리가 0 이면 **쓰다 만 것**이다. 다시 써라.

| 지금 상태 | 이번 반복에 할 일 |
|---|---|
| ① 엔트리 0 | **멈춰라.** `BLOCKED.md` 에 "① 이 미승인" 이라 적고 아무것도 하지 마라 |
| ② 없음 / 엔트리 0 | **② 를 쓴다** → 커밋 → **멈춘다** |
| ② 완성 · ③ 없음/엔트리 0 | **③ 을 쓴다** → 커밋 → **멈춘다** |
| ③ 완성 | **④ — 체크 안 된 첫 task 실행** |

## 1. ★ 한 반복 = 한 단계. 이어붙이지 마라 ★

`auto-tech-design` 은 끝나면 `auto-writing-plans` 를 **자동으로 부른다** (Step 7).
**그걸 하지 마라.** ② 를 쓰고, 커밋하고, **멈춰라.**

**이유**: ②③ 을 한 반복에 다 하면 컨텍스트가 다 차서 ④ 를 시작할 여유가 없다.
게다가 반쯤 쓰다 죽으면 다음 반복이 그걸 완성본으로 오해한다.
**루프가 다음 반복을 준다. 서두를 이유가 없다.**

**단계를 끝냈으면 반드시 커밋해라.** 다음 반복의 너는 git 이력으로만 진척을 본다.

## 2. 단계별 지시

### ② 개발방향
`auto-tech-design` 을 호출한다. 단 **Step 7 (다음 단계 자동 호출) 은 하지 마라.**
change-history 엔트리까지 박고 커밋한 뒤 멈춘다.

### ③ 구현계획서
`auto-writing-plans` 를 호출한다. 단 **Step 7 은 하지 마라.**

**③ 을 쓸 때 반드시 지켜라** — 다음 반복의 네가 이것만 보고 실행한다:
- 모든 step 을 `- [ ]` 체크박스로. 이게 진척 장부다.
- **완료 조건을 §4 에 명시**하고, 전부 **기계가 판정 가능**해야 한다.
  "눈으로 확인" 은 금지. 브라우저 렌더처럼 판정 불가한 건 `BLOCKED.md` 로 뺀다.
- **`commit_policy: per-task`** — 안 그러면 다음 반복이 진척을 못 읽는다.
- 완료 조건에 숫자(테스트 건수 등)를 적었으면 **본문과 대조해서 맞는지 확인해라.**

### ④ 실행
체크 안 된 첫 task 를 계획서대로 실행한다. 끝나면 **3개 다** 해야 끝이다:
1. 체크박스 `- [x]`
2. 계획서 하단에 `[코드-수정]` 엔트리 (task 당 1건)
3. 코드 + 계획서를 **한 커밋으로**

## 3. 사람이 없다 — 묻지 마라

이 워크플로는 **"막히면 멈추고 물어봐"** 로 설계됐다. 랄프는 **"막히면 계속 시도"** 다. **정반대다.**

**여기엔 답할 사람이 없다.** 질문하고 기다리면 — 너는 멈추고, 루프가 같은 프롬프트를 다시 먹이고, 너는 또 질문한다. **무한 루프다.**

> **질문하지 마라. `BLOCKED.md` 에 적고, 건너뛰고, 다음으로 가라.**

| 원래 규칙 (사람에게 물어봄) | 여기선 |
|---|---|
| 계획에 없는 파일을 만져야 함 | `BLOCKED.md` → **건너뜀** |
| 파괴적 작업 (`rm -rf` · `reset --hard` · force-push) | **절대 금지** → `BLOCKED.md` |
| 자가 복구 3회 실패 | `BLOCKED.md` → **건너뜀** |
| 외부 호출 (`git push` · PR · 외부 API) | **절대 금지.** 로컬 커밋까지만 |
| 계획에 없는 새 의존성 | `BLOCKED.md` → **건너뜀** |
| **① 요구사항이 틀린 것 같음** | **`BLOCKED.md` → 멈춤.** ① 은 사람 것이다. **절대 고치지 마라** |

마지막 줄이 제일 중요하다. **① 을 고치면 이 도구를 쓰는 의미가 사라진다.**

### `BLOCKED.md` 형식

없으면 만들고, 있으면 **해당 섹션에 추가**해라 (덮어쓰지 마라).

```markdown
# BLOCKED — 사람이 봐야 할 것

## 사람이 확인해야 할 것 (예정된 것)
### <항목>
<확인 절차를 구체적으로>

## 막혀서 못 한 것 (예정 밖)
### <어디서>
- **하려던 것**:
- **막은 것**: <에러 원문>
- **시도 횟수**: N회 / 시도한 것:
- **사람이 할 일**:
```

## 4. 완료 조건

**③ 의 §4 가 정본이다.** 거기 적힌 검증을 **전부** 돌려서 **전부 통과**하면 완료다.

전부 참일 때만 마지막 메시지에 정확히 출력해라:

```
<promise><완료문구></promise>
```

### 거짓 약속 금지

- 하나라도 실패하면 → **출력하지 마라**
- 막힌 게 남았으면 → **출력하지 마라**
- "이 정도면 됐다" 싶어도 → **출력하지 마라**
- 루프를 벗어나고 싶어서 → **절대 출력하지 마라**

막힌 채로 반복 상한에 도달하면 — **그게 정상 결과다.** `BLOCKED.md` 가 사람에게 설명해준다.
거짓말로 빠져나가는 것보다 **상한에 부딪히는 게 낫다.**

## 5. 절대 하지 마라

| 금지 | 왜 |
|---|---|
| 사람에게 묻고 기다림 | 답할 사람이 없다. 무한 루프가 된다 |
| **① requirements.md 수정** | **① 은 사람 것이다.** 고치면 도구를 쓰는 의미가 없다 |
| 참이 아닌 `<promise>` | 루프의 유일한 안전장치를 무력화한다 |
| `git push` · PR · 외부 API | 되돌릴 수 없다. 로컬 커밋까지만 |
| `rm -rf` · `reset --hard` · force-push | 되돌릴 수 없다 |
| 한 반복에 두 단계 | 컨텍스트가 터진다. 하나 하고 멈춰라 |
| 커밋 없이 다음으로 | 다음 반복의 네가 진척을 못 읽는다 |
| 테스트를 통과시키려고 테스트를 고침 | 테스트가 곧 수용 기준이다. 구현을 고쳐라 |
````

## 실행

**새 세션에서** 프로젝트 폴더를 열고 (랄프는 **실행한 세션을 그대로 잡습니다**):

```bash
cd <프로젝트> && claude
```

```
/ralph-loop "PROMPT.md 를 읽고 그대로 따라라. 매 반복 그 파일이 지시서다." \
  --completion-promise "<완료문구>" --max-iterations 30
```

**`--max-iterations` 를 꼭 주세요.** 플러그인 문서가 명시합니다 — *"`--completion-promise` 는 정확히 하나만 매칭되니 **`--max-iterations` 를 주 안전장치로** 삼아라."*
②③ 에 2~3 반복 + task 수 + 재시도 여유로 잡으면 됩니다.

## 이 방식이 안 맞는 경우

- **기술 선택이 중요한 프로젝트** → ②③ 은 게이트를 열어두고 ④ 만 랄프에게 (`/writing-plans` 까지 사람이)
- **① 이 모호한 프로젝트** → 랄프가 ② 에서 추측으로 메웁니다. ① 부터 다시.
- **되돌리기 어려운 작업이 포함** (DB 마이그레이션 · 배포) → 랄프는 파괴적 작업을 금지당해서 전부 `BLOCKED.md` 로 갑니다. 사람이 하세요.
