1. 아키텍처 개요

외부 채널(푸시/이메일)은 느리고 자주 죽습니다. 그래서 원래 요청과 발송을 끊어놓고, DB에 먼저 남긴 뒤 큐로 넘깁니다.

의도("놓치지 않게, 그러나 귀찮게 하지 않게")를 구조로 옮기면:

의도의 절반구조적 장치
놓치지 않게DB 먼저 기록(유실 방지) · 백오프 재시도 · 실패도 보존
귀찮게 하지 않게발송 전 설정 확인 · dedup_key 로 중복 차단

2. 영향 받는 컴포넌트/파일

컴포넌트변경
models/notification.py신규 — 알림 엔티티 + 상태
models/notification_pref.py신규 — 사용자 채널 설정
services/notification_service.py신규 — 설정 확인 + 알림 생성
workers/notification_worker.py신규 — 큐 소비 + 발송 + 재시도
channels/base.py · channels/push.py · channels/email.py신규 — 어댑터 인터페이스
api/notifications.py신규 — 조회 · 읽음 · 설정
migrations/신규 — 2개 테이블

3. 데이터 모델/스키마 변경

schema 
notifications
  id             PK
  user_id        FK, index
  event_type     문자열 (comment | mention | ...)
  channel        push | email
  status         대기 | 발송중 | 완료 | 실패
  dedup_key      unique          ← 중복 차단 (NFR-2)
  attempts       정수, 기본 0
  next_retry_at  timestamp, nullable
  created_at     timestamp

notification_prefs
  user_id + event_type + channel   복합 PK
  enabled                          불리언

dedup_key = {user_id}:{event_type}:{event_id}:{channel} — 같은 사람에게 같은 일로 같은 채널은 한 번뿐.

5. 핵심 결정 + 대안 비교

결정 1 — 발송 방식: 큐 기반 비동기

채택

큐 기반 비동기

장점
외부 채널 장애가 원 요청에 영향 없음, 재시도 자연스러움
단점
인프라 1개 추가, 지연 존재
✕ 미채택

동기 발송

장점
단순, 즉시성
단점
이메일 서버가 느리면 원 요청이 같이 느려짐 — NFR-1 위반
✕ 미채택

크론 배치

장점
인프라 최소
단점
최대 지연이 배치 주기 — NFR-4(5초) 위반

근거: NFR-1(원 요청이 막히면 안 됨)이 단순함보다 위입니다.

결정 2 — 중복 차단: DB unique 제약

채택

DB unique 제약

장점
워커가 몇 대든 DB가 최종 심판
단점
중복 시 예외 처리 필요
✕ 미채택

애플리케이션 사전 검사

장점
예외 없이 깔끔
단점
워커 2대가 동시에 검사하면 둘 다 통과 — NFR-2 위반 (CH-002 에서 뒤집힘)
✕ 미채택

큐 자체의 중복 제거

장점
코드 0
단점
큐 종류에 종속, 재시도 건이 새 메시지라 안 걸림

근거: NFR-2(워커 여러 대여도 중복 금지)를 애플리케이션 검사로는 보장할 수 없습니다. 검사와 삽입 사이에 다른 워커가 끼어듭니다.

결정 3 — 기록 순서: DB 먼저, 큐 나중

채택

DB(대기) → 큐

장점
큐 투입 전에 죽어도 대기 로 남아 복구 가능
단점
쓰기 1회 추가
✕ 미채택

큐 → 워커가 DB 기록

장점
쓰기 1회
단점
큐가 유실되면 흔적조차 없음 — NFR-3 위반

근거: 의도의 전반부("놓치지 않게")는 흔적이 남는 것부터 시작합니다.

6. 위험/사이드이펙트 (preliminary)

breaking

재시도가 중복 발송이 됨

재시도는 같은 알림 행을 재사용 (새 행 X) — dedup_key 유지

side-effect

큐 투입 전 프로세스 종료 → 알림 유실

DB 대기 선기록 후 큐 투입 (결정 3)

race

워커 2대가 같은 알림을 동시에 집음

대기 → 발송중 상태 전이를 조건부 UPDATE 로 (한 대만 성공)

breaking

설정 조회 실패 시 기본값

조회 실패 = 발송 안 함 (스팸보다 누락이 안전)

7. 테스트 전략

  • 단위: 설정 꺼짐 → 생성 안 함, dedup_key 생성 규칙, 백오프 간격 계산
  • 통합: 이벤트 → 생성 → 발송 → 완료 왕복
  • 동시성: 워커 2대가 같은 건 집기 → 한 대만 발송
  • 멱등성: 같은 이벤트 2회 → 알림 1건
  • 실패 경로: 채널 3회 실패 → 실패 상태로 보존 (조용히 사라지지 않음)
