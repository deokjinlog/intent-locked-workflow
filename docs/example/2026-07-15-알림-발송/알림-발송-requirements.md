# 요구사항: 알림 발송 (notification delivery)

> **intent-locked-workflow 예시 산출물** — `/brainstorming 알림 발송 기능` 이 만드는 **① 기획** 단계 문서입니다.
> 데모용 가상 기능이며 실제 서비스가 아닙니다. 실제 문서도 이 형식 그대로, 분량만 더 깁니다.

## 1. 배경/목적

댓글·멘션·처리 완료 같은 일이 생겨도 사용자가 앱에 들어와야만 알 수 있습니다. 놓치는 일이 늘고 있습니다.

**여기서 잠그는 의도** — *"놓치지 않게. 그러나 귀찮게 하지 않게."*

이 문장에는 **긴장**이 있습니다. 한쪽으로 기울면 스팸이 되고, 반대로 기울면 알림이 무의미해집니다. 이후 모든 게이트는 산출물이 이 **균형**에서 벗어나지 않았는지를 봅니다.

> 드리프트가 실제로 일어나는 지점: "재시도"를 넣다 보면 중복 발송이 생기고 — 기술적으로는 맞지만 **의도의 후반부("귀찮게 하지 않게")를 배신**합니다.

## 2. 사용자 스토리 / 시나리오

- 사용자로서, 내 글에 댓글이 달리면 알림을 받고 싶다.
- 사용자로서, 어떤 종류의 알림을 어느 채널로 받을지 **직접 끄고 켜고** 싶다.
- 사용자로서, 같은 일로 **두 번 알림받고 싶지 않다.**
- 사용자로서, 앱에서 지난 알림을 모아보고 읽음 처리하고 싶다.
- 운영자로서, 발송 실패한 알림이 조용히 사라지지 않기를 바란다.

## 3. 기능 요구사항 (FR)

| ID | 요구사항 |
|---|---|
| FR-1 | 이벤트가 발생하면 수신자의 **설정을 확인한 뒤** 알림을 만든다. |
| FR-2 | 사용자는 알림 종류별로 채널(푸시 / 이메일)을 켜고 끌 수 있다. |
| FR-3 | 같은 이벤트로 같은 사용자에게 **두 번 발송하지 않는다.** |
| FR-4 | 발송에 실패하면 **간격을 늘려가며 최대 3회** 재시도한다. |
| FR-5 | 3회 모두 실패하면 `실패` 로 남기고 조용히 버리지 않는다. |
| FR-6 | 사용자는 자신의 알림 목록을 조회하고 읽음 처리할 수 있다. |

## 4. 비기능 요구사항 (NFR)

| ID | 요구사항 |
|---|---|
| NFR-1 | 외부 채널(푸시/이메일)이 느리거나 죽어도 **원래 요청이 막히지 않는다.** |
| NFR-2 | 워커가 여러 대여도 같은 알림이 **중복 발송되지 않는다.** |
| NFR-3 | 큐에 넣기 전에 죽어도 알림이 **유실되지 않는다.** |
| NFR-4 | 이벤트 발생부터 발송까지 정상 상황에서 5초 이내. |

## 5. 범위 밖 (Out of Scope)

- 실제 푸시/이메일 공급자 연동 — 어댑터 인터페이스까지만
- 알림 문구 다국어 번역
- SMS · 카카오톡 채널
- 실시간 인앱 배지 카운트
- 방해 금지 시간대(야간 발송 억제)

## 6. 수용 기준 (Acceptance Criteria)

- [ ] 댓글 이벤트 발생 → 설정이 켜진 채널로만 알림 생성
- [ ] 해당 채널을 끈 사용자 → 알림 생성 안 됨
- [ ] 같은 이벤트를 두 번 흘려보냄 → 알림 1건만 생성
- [ ] 채널 발송 실패 → 재시도 간격이 늘어나며 최대 3회, 이후 `실패`
- [ ] 워커 2대가 동시에 같은 건을 집음 → 한 대만 발송
- [ ] 큐 투입 직전에 죽음 → 알림은 `대기` 로 남아 복구 가능
