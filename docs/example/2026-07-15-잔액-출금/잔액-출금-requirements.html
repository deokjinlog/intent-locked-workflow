사용자 잔액 출금

balance withdrawal · 2026-07-15 · intent-locked-workflow 예시 산출물

🔒 여기서 잠그는 의도

“사용자가 스스로, 안전하게, 되돌릴 수 있게 뺀다.”

이 한 문장이 설계 · 계획 · 코드까지 변형 없이 이어집니다. 이후 모든 게이트는 산출물이 이 문장과 어긋나지 않았는지를 봅니다.

1. 배경 / 목적

적립만 가능하고 인출 경로가 없어 잔액이 계속 쌓이고 있습니다. 사용자가 스스로 잔액을 인출할 수 있어야 합니다.

2. 사용자 스토리

  • 사용자로서, 잔액 중 일부를 등록된 계좌로 출금 신청하고 싶다.
  • 사용자로서, 신청 후 처리 상태(대기 / 완료 / 실패)를 확인하고 싶다.
  • 사용자로서, 아직 처리되기 전이라면 신청을 취소하고 싶다.
  • 운영자로서, 비정상 출금 시도를 사후에 추적하고 싶다.

3. 기능 요구사항

ID요구사항
FR-1사용자는 잔액 이하의 금액을 출금 신청할 수 있다.
FR-2신청 시점에 잔액을 즉시 차감하고 대기 상태의 출금 건을 만든다.
FR-3사용자는 대기 상태의 신청을 취소할 수 있고, 취소하면 잔액이 복원된다.
FR-4사용자는 자신의 출금 내역과 상태를 조회할 수 있다.
FR-5모든 출금 상태 변화는 감사 로그로 남는다.

4. 비기능 요구사항

ID요구사항
NFR-1동시 요청에도 잔액이 음수가 되지 않는다.
NFR-2같은 신청이 중복 처리되지 않는다.
NFR-3출금 신청 응답은 1초 이내.

5. 범위 밖

  • 계좌 등록 · 본인 인증
  • 실제 송금 연동
  • 관리자 수동 승인 화면
  • 출금 수수료 정책

6. 수용 기준

  • 잔액 10,000원에서 3,000원 신청 → 잔액 7,000원 + 대기 건 1개
  • 잔액을 넘는 금액 신청 → 거부, 잔액 그대로
  • 대기 건 취소 → 잔액 복원, 상태 취소
  • 이미 완료된 건 취소 시도 → 거부
  • 같은 잔액으로 동시에 2건 신청 → 하나만 성공, 잔액 음수 없음
