# 개발방향: 사용자 잔액 출금

> **intent-locked-workflow 예시 산출물** — `/tech-design` 이 만드는 **② 설계** 단계 문서입니다.
> 상위 문서: [잔액-출금-requirements.md](잔액-출금-requirements.md)

## 1. 아키텍처 개요

출금은 **잔액 차감**과 **출금 건 생성**이 항상 같이 성공하거나 같이 실패해야 합니다. 둘을 한 트랜잭션으로 묶고, 잔액 행에 잠금을 걸어 동시 요청을 직렬화합니다.

```
[요청] → 잔액 행 잠금 → 잔액 확인 → 차감 + 출금 건(대기) 생성 → 커밋 → [응답]
                                └ 부족하면 전체 롤백
```

의도(*"안전하게, 되돌릴 수 있게"*)를 구조로 옮기면 — **안전** = 한 트랜잭션 + 행 잠금, **되돌림** = 상태 전이(대기→취소)로 복원.

## 3. 데이터 모델/스키마 변경

```
withdrawals
  id            PK
  user_id       FK, index
  amount        정수 (원 단위, 음수 불가)
  status        대기 | 완료 | 실패 | 취소
  requested_at  timestamp
  settled_at    timestamp, nullable
  idem_key      unique          ← 중복 신청 차단 (NFR-2)
```

금액은 **정수(원)** 로 둡니다. 부동소수는 반올림 오차로 잔액이 틀어질 수 있습니다.

## 5. 핵심 결정 + 대안 비교

### 결정 1 — 동시성 제어: 비관적 잠금

| 대안 | 장점 | 단점 | 판정 |
|---|---|---|---|
| **비관적 잠금** (`SELECT FOR UPDATE`) | 잔액 음수 원천 차단, 구현 단순 | 같은 사용자 요청 직렬화 | **채택** |
| 낙관적 잠금 (version) | 잠금 없음, 처리량 높음 | 충돌 시 재시도 로직 필요 | 미채택 — 잔액은 재시도보다 정확성 우선 |
| 애플리케이션 잠금 | DB 무관 | 다중 인스턴스에서 깨짐 | 미채택 |

**근거**: 출금은 초당 수천 건이 아닙니다. NFR-1(잔액 음수 금지)이 처리량보다 위입니다.

### 결정 2 — 취소 방식: 상태 전이

| 대안 | 장점 | 단점 | 판정 |
|---|---|---|---|
| **상태 전이** (대기→취소, 잔액 복원) | 이력 보존, 감사 가능(FR-5) | 행이 남음 | **채택** |
| 행 삭제 | 단순 | 이력 소실 — FR-5 위반 | 미채택 |

**근거**: 요구사항의 *"되돌릴 수 있게"* 는 **흔적 없이 지우기가 아니라 되돌린 기록이 남기**입니다.

## 6. 위험/사이드이펙트 (preliminary)

| 위험 | 카테고리 | 대응 |
|---|---|---|
| 잔액 차감 후 예외 → 돈만 사라짐 | side-effect | 한 트랜잭션으로 묶음 |
| 같은 요청 두 번 → 이중 차감 | breaking | `idem_key` unique |
| 동시 신청 → 잔액 음수 | race | `SELECT FOR UPDATE` |
| 잔액 메서드가 다른 기능에서도 쓰임 | breaking | 호출자 조사 후 시그니처 유지 |

## 7. 테스트 전략

- **단위**: 잔액 차감/복원, 상태 전이 규칙(완료 건 취소 거부)
- **통합**: 신청 → 취소 → 잔액 복원 왕복
- **동시성**: 같은 잔액으로 2건 동시 신청 → 하나만 성공
- **멱등성**: 같은 `idem_key` 2회 → 1건만 생성
