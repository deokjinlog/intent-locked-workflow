### D2: 다크 프리미엄 1 톤 default + variety 보존 (B-1)

v2.2.4 의 6 톤 자유 룰을 완전 삭제하면 디자인 다양성 (variety = feature) 가치 손상. → **baseline 톤만 다크 프리미엄으로 통일**, 그 안에서 variety 보존:

- 악센트 색상 팔레트 (보라+시안 / 시안+핑크 / 청록+에메랄드 매번 다르게)
- 카드 / overlay / glow 강도 (subtle ↔ pronounced)
- 헤더 typography / spacing
- 다이어그램 스타일 (flat / glassmorphism / neon outline)

다크 프리미엄 = "Premium dashboard / 고급 분석 도구" 톤. raw 색 예시:

- 배경: `#0a0b0f` / `#0d1117` / `#111218` 류
- 텍스트: `#e6edf3` / `#f0f6fc` / `#e8eaf0` (muted 도 `#8b949e` 보다 어둡지 않게)
- 악센트: 보라/시안/핑크 그라데이션 + accent glow
- 카드: 반투명 흰색 overlay (`rgba(255,255,255,0.04~0.07)`)

### D3: 4-요소 옵셔널 (B-2) — LLM 자율 판단

| 요소 | 적용 케이스 |
|---|---|
| 데이터 차트 / 그래프 | 숫자 비교 / 분포 / 시계열 / 비율 |

본문 콘텐츠가 위에 안 맞으면 사용 X. 강제 4 요소 의무화는 anti-pattern.

### D4: WCAG AA 4.5:1 baseline (B Anti-Pattern)

다크 배경에서도 muted 텍스트 contrast 4.5:1 이상 보장. 자동 검증 도구는 v2.3.x 후속 후보 (현 버전 LLM self-check).

### D5: line-level unified diff only (C-1) — semantic / word-level 금지

word-level / character-level diff 는 의역 위험. semantic diff (변경 의도 요약 텍스트) 는 의미 1:1 룰 위반. line-level 단순 비교만.

### D6: context ±N (기본 3) collapse OK (C-2) — primary visibility 예외

v2.3.3 "primary default-visible" 룰은 본문 콘텐츠 적용. 코드 raw 의 unchanged 영역은 secondary 성격 → collapse 허용 (`<details>`). 단 변경 라인 (+ / -) 자체는 항상 default-visible.

### D7: 신규 / 삭제 파일 fallback (C-4)

진짜 before/after 쌍일 때만 unified diff:

- 신규 파일 (원본 X / 수정본 only) → 초록 배경 단순 코드 블록
- 삭제 파일 (원본 only / 수정본 X) → 빨강 배경 단순 코드 블록

억지 unified diff 형식 강요 X.

## 우려 / 해결

### Q1: v2.2.4 톤 자유 (variety) 가치 손상?

→ 다크 프리미엄 baseline 안에서 variety 보존. 6 톤 inspiration 자체 삭제 X (다크 프리미엄의 sub-variation 으로 흡수).

### Q2: 사용자가 라이트 톤 명시 원할 때?

→ `prefers-color-scheme: light` 분기 옵셔널 제공 (OS 설정 따라감). 명시적 override 슬래시 옵션 (e.g. `/sync-html --theme=light`) 은 v2.3.x 후속 후보 (Non-goal).

### Q3: 4 요소 옵셔널 — subagent B 가 강제 4 요소 의무화 시도?

→ Anti-Pattern 명시 ("4 요소 모두 의무 사용 — 콘텐츠와 안 맞으면 사용 X"). LLM 자율 판단 강제 X.

### Q4: WCAG AA contrast 자동 검증?

→ 현 버전 미도입 (subagent self-check + Anti-Pattern 명시). 자동 검증 도구 도입은 후속 후보.

### Q5: C 항목 diff 가 v2.0.0 byte-copy 룰과 충돌?

→ X. `.md` 원본의 `**원본**` + `**수정본**` 두 블록은 무변형 (byte-copy 보존). `.html` 렌더링 단계 표시 방식만 변경. writing-plans / executing-plans / js-super-sub-driven / plan_byte_check.py 본문 영향 0.

### Q6: C 항목 다중 hunk (한 task 안 코드 블록 떨어진 위치 2+ 곳 변경)?

→ unified diff 안에 hunk 분리 표시 (전통 git diff 형식). 별개 diff 블록 분리 안 함 (단일 코드 블록 안의 변경이므로). LLM 자율 표현.

### Q7: C 항목 다른 산출물 (req / tech-design) 도 같은 룰 적용?

→ impl-plan 한정. req / tech-design 에 before/after 코드 블록 쌍이 있다면 LLM 자율 판단 (강제 X). 명시 룰은 impl-plan 한정.

### Q8: 변경이력 footer 생략 시 generating-html 발동 조건 (footer 0건 → 초안 → 발동) 영향?

→ X. footer 0건 / 1건+ 판정은 `.md` 본문 기준 (`.html` 표시 방식과 무관). `.md` 의 변경이력 footer 는 그대로 (생략 X). `.html` 만 옵셔널 생략.
