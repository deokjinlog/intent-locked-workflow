# v2.2.1 — `docs-pretty` 를 `.html` 전용 fire-and-forget 으로 전환

> **Mode:** Backlog promotion (auto-flow, brainstorming 생략 — 백로그 `docs/backlog/v2.2.1-docs-pretty-html-only.md` 가 이미 brainstorming 수준으로 정리됨). v2.2.0 후속 단순화.
> 작성: 2026-05-19

## 한 줄 요약

v2.2.0 의 `.md` format-only pass A 를 **제거**하고, `.html` 생성 subagent B 를 **fire-and-forget** 으로 dispatch. 메인 흐름은 docs-pretty 결과 대기 안 함. 신규 `/regen-html <slug>` slash command 로 live doc 후 수동/자동 재생성 가능.

## 배경 (v2.2.0 과의 관계)

v2.2.0 (이미 release 완료) 은 docs-pretty 발동 시 Sonnet subagent **2개 병렬** dispatch:
- A — `.md` format-only pass (기존)
- B — `.html` 사람 전용 시각화 사본 (신규)

dogfood 가설:
- AI 는 어차피 `.md` 만 읽음 → A 의 `.md` format-only pass 가치는 **사람용**
- 그런데 B 의 `.html` 가 사람용 가독성 사본이 되면 A 의 `.md` 가공은 **중복**
- 그러면 docs-pretty 가 메인 흐름 동기 블록할 이유가 사라짐 (사람용 자료는 백그라운드로 처리 OK)

→ v2.2.1 = **A 제거 + B fire-and-forget**.

## 핵심 결정

### D1. A (`.md` format-only pass) **제거**

- `.md` 는 메인이 작성한 RAW 본문 그대로 사용자 리뷰
- 사람용 가독성은 B 의 `.html` 가 전담
- AI 흐름은 어차피 `.md` 만 읽으므로 영향 0

**근거**: A 의 가치는 사람 가독성인데 B 가 그 책임을 흡수. 중복 제거.

### D2. B (`.html` 생성) **fire-and-forget**

```
RAW .md 작성
  → docs-pretty 발동 — .html subagent dispatch (run_in_background=true)
  → 메인 즉시 다음 turn 진행 (user review RAW .md)
  → (배경에서 .html 사이드카로 떨어짐)
```

- 메인은 subagent 결과 **대기 X**
- 완료 notify 받으면 **silent log** (사용자에게 push X)
- 실패 시 silent log + 사용자가 다음에 `.html` 열려고 할 때 부재 인지 → `/regen-html` 수동

**근거**: 메인 흐름 동기 블록 해제 → latency ↓ + 비용 동일 (subagent 1회).

### D3. 신규 `/regen-html <slug>` slash command

- 인자: feature slug (e.g., `2026-05-19-foo`)
- 동작: `<slug>-{requirements,tech-design,implementation-plan}.md` 셋 중 존재하는 모든 `.md` 에 대해 `.html` 재생성 subagent dispatch
- 인자 누락 시 latest slug 추론 (`scripts/auto_flow.find_latest_slug`)
- **skill 이 아니라 command** — 사용자 명시 호출용 1회성 액션 (자동 description 매칭 차단)

**근거**: `/fast-tasks` (v2.1.0) 와 같은 패턴 — 명시 호출 only.

### D4. `change-propagation` SKILL.md 본문에 `/regen-html` 자동 호출

- live doc 진입 후 `.md` 변경 시 `.html` stale 위험
- `change-propagation` 마지막 단계에 `/regen-html <slug>` 자동 호출 추가
- "의미 있는 변경" (change-propagation) 만 자동, 그 외는 stale 허용 + 사용자 수동 `/regen-html`

**근거**: `change-propagation` 은 의미 변경 시점의 분기점. 그 외 (오타 fix 등) 는 stale 허용 + 사용자 판단.

### D5. `.html` 푸터 stale 표시

`.html` 푸터에 generation 기준 시점 박음:

```html
<footer>
  이 파일은 <slug>-requirements.md 의 [CH-YYYYMMDD-NNN @ YYYY-MM-DD HH:MM] 기준입니다.
  이후 .md 수정 사항은 반영되지 않을 수 있습니다 — 필요 시 /regen-html <slug>.
</footer>
```

**근거**: 사용자가 시각적으로 stale 인지 가능. 자동 감지 없이도 안전.

### D6. 디바운스 — 연속 fix 시 마지막만 regen

- 메인이 마지막 fix turn 마다 이전 subagent 가 살아있으면 cancel + 새 dispatch
- 디바운스 윈도우 정확 값 → U1 (designing-direction 에서 결정)

**근거**: 사용자가 연속 fix 던질 때 매번 regen 하면 비용 ↑ + 마지막만 의미 있음.

### D7. 동기화 boundary — v2.2.0 답습

기존 docs-pretty 의 boundary 그대로:
- 초기 작성 iteration loop 에서만 발동 (매 user fix turn 마다)
- **첫 change-history entry logging 순간 STOP**
- live doc 진입 후 `.html` 동기화는 두 경로 (D4 자동 / D3 수동)

### D8. `.html` 자체 룰 — v2.2.0 보존

| 항목 | 룰 |
|---|---|
| 의미 보존 | `.md` 의 문장 / 항목 / 결정 사항 / 숫자 / 표 내용 1:1. 의역 X. |
| 시각화 | 헤더 hierarchy, 표 정렬·줄무늬·sticky, DAG → Mermaid 또는 inline SVG, syntax highlight |
| Self-contained | 외부 CDN / JS / CSS 의존 X. inline 빌드 |
| 다크/라이트 | `prefers-color-scheme` 자동 |
| AI 노출 | **X** — 모든 AI 흐름은 `.md` 만 읽음 |

`skills/docs-pretty/html-companion-prompt.md` (v2.2.0 신규) 그대로 사용.

## 미정 사항 (designing-direction 에서 결정)

- html-renderer: 책임 명확, 향후 룰 분리 가능
- recommendation candidate: general-purpose (DRY + 신규 agent 정의 부담 X)

### U3. 실패 silent log 위치

- `/tmp/cc-html-regen.log`
- `.js-super/html-regen.log` (subagent changelog buffer 옆)

### U4. `.html` gitignore 여부 — v2.2.0 미정 항목 상속

- v2.2.0 에서 이미 결정: **gitignore** (`docs/features/**/*.html`). v2.2.1 은 답습.
- recommendation candidate: 답습 (변경 X)

### U5. v2.2.0 거치고 v2.2.1 vs v2.2.0 건너뛰기

- 백로그 저자 의견: "v2.2.0 단계 건너뛰고 v2.2.1 안을 v2.2.0 으로 채택 (후자가 깔끔)"
- 실제 상황: v2.2.0 이미 release 완료 + push 됨 (2026-05-19) — 답 정해짐
- recommendation: v2.2.0 위에 patch (단순화). v2.2.0 의 A 제거 + B 를 fire-and-forget 으로 변환

## 비목표 (Non-goals)

- 메인이 `.html` subagent 결과 대기 → fire-and-forget 의미 무화
- `.html` 을 source of truth 로 사용 → 절대 X (`.md` only)
- `.md` format-only pass (A) 부활 시도 → v2.2.1 의 단순화 무화
- 모든 Edit 도구 호출 감지해서 자동 `.html` regen → 불완전 + 노이즈
- change-propagation 외 자동 hook 추가 → 직접 편집은 수동 `/regen-html` 만
- `.html` 인터랙티브 (탭/접힘/검색) — 정적 페이지만 (v2.2.0 답습)
- `code-pretty` 의 `.html` 동봉 — 본 변경 범위 외

## 우려 / 해결

### O1. 메인 흐름 latency 측정 — fire-and-forget 검증

- 메인이 docs-pretty 발동 전후 turn elapsed 차이 < 100ms 보장
- 측정 fixture (다음 외부 dogfood 시 발현)

**Mitigation**: H18 fixture 시나리오 — 메인 turn timestamp 비교.

### O2. 메인 turn 진행 후 subagent 결과 도달 — race

- 사용자가 `.html` 열려는 시점에 아직 subagent 완료 안 됨 → 빈 파일 또는 stale `.html`
- 사용자 수동 인지 + `/regen-html` 호출

**Mitigation**: D5 푸터 stale 표시 + `.html` 부재 시 사용자 안내.

### O3. 디바운스 cancel 시 작업 손실

- 이전 subagent 가 80% 완료 시점에 cancel → 결과 폐기
- 마지막 fix 만 의미 있으므로 손실 의도된 동작

**Mitigation**: D6 명시 — 디바운스는 마지막만 의미. cancel 시 silent log.

### O4. `change-propagation` 호출 외 `.md` 직접 편집

- 사용자가 오타 한두 글자 등 직접 편집 → `.html` stale
- 자동 hook 추가 X (D4 명시) → 사용자 명시 `/regen-html` 만

**Mitigation**: D5 푸터 stale 표시 + 사용자 안내. 자동 감지 시도 X.

### O5. silent log 위치 폴루션

- `/tmp/cc-html-regen.log` → 시스템 임시 폴더 (재부팅 시 사라짐)
- `.js-super/html-regen.log` → 프로젝트 안 (gitignored)

**Mitigation**: U3 결정 (recommendation: `.js-super/html-regen.log`).

### O6. v2.2.0 의 A+B 병렬 dispatch 룰이 v2.2.1 에서 깨짐

- v2.2.0 의 SKILL.md / CLAUDE.md / Anti-Patterns 표 / 결합 메모 모두 갱신 필요
- 결합 메모 4 파일 atomic patch 룰 (v2.2.0 CLAUDE.md 명시) 와 충돌

**Mitigation**: v2.2.1 변경은 v2.2.0 결합 메모 갱신을 포함. Anti-Patterns 표 의 "A 와 B 순차 dispatch" 룰 → "fire-and-forget" 으로 변경.

### O7. AskUserQuestion 강제 룰 — `docs-pretty` 는 적용 X

- `docs-pretty` 는 8 skill 룰 적용 대상 **아님** (보조 skill)
- 본 변경에서도 `docs-pretty` 본문 / `/regen-html` command 본문에 AskUserQuestion 강제 boilerplate 박지 않음

**Mitigation**: v2.2.0 동일 룰 보존.
