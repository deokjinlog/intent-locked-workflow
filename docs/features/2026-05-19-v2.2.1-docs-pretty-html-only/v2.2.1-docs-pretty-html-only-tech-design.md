# 개발방향: docs-pretty html-only fire-and-forget

> **For agentic workers:** Technical spec for `docs-pretty` v2.2.1 single-task simplification — A (`.md` format-only) 제거 + B (`.html`) fire-and-forget. Anchored to `v2.2.1-docs-pretty-html-only-requirements.md`. Consumed by `v2.2.1-docs-pretty-html-only-implementation-plan.md`. NEXT STEP: invoke `auto-writing-plans` (or `/auto-write-plan`).

## 1. 아키텍처 개요

v2.2.0 의 두 subagent 병렬 dispatch → **B 단일 fire-and-forget dispatch** 로 축소. A (`.md` format-only) 책임 제거. 신규 `/regen-html` slash command + `change-propagation` 자동 호출 + 디바운스.

흐름:

```
RAW .md 작성
   │
   ▼
[Pre-flight] file 존재 + 변경이력 footer 비어있음 + filename pattern (v2.2.0 답습)
   │
   ▼
[Step 2 신규] fire-and-forget B dispatch
   ├─ Task tool 호출 (run_in_background=true)
   ├─ subagent_type=general-purpose, model=sonnet, prompt=html-companion-prompt.md
   ├─ 이전 subagent 살아있으면 cancel (디바운스 3초 안)
   └─ 메인 즉시 return — Step 3 진행
   │
   ▼
[Step 3 신규] 메인 다음 turn (user review RAW .md)
   │
   ▼ (배경)
[Subagent] .html 사이드카 작성 → .js-super/html-regen.log silent log
   │
   ▼ (live doc 진입 후)
[change-propagation 마지막 단계] /regen-html <slug> 자동 호출 (D4)
   │
   또는
   │
[사용자 수동 /regen-html <slug>] 명시 호출 (D3)
```

핵심 분리:
- **A 제거** — `.md` format-only pass 없음. AI 흐름은 RAW `.md` 만 읽음 (영향 0)
- **B fire-and-forget** — 메인 latency 거의 0 (Task dispatch 비용만)
- **`/regen-html`** — live doc 후 사용자 수동 + `change-propagation` 자동
- **디바운스 3초** — 연속 fix 시 마지막만 regen
- **silent log** — `.js-super/html-regen.log` (gitignored, 사용자 push X)

## 5. 핵심 결정 + 대안 비교

### D-T1. A (`.md` format-only pass) 제거 — U5 결정 답습 (v2.2.0 위 patch)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **A 제거** ← 채택 | 중복 제거 (B 가 사람 가독성 흡수) / 비용 ↓ / 메인 latency ↓ / AI 흐름 영향 0 (RAW `.md` 만 읽음) | RAW `.md` 가 약간 거칠 수 있음 (사용자 리뷰 시) |
| A 보존 + B 만 fire-and-forget | v2.2.0 호환 / 사람용 `.md` 도 깔끔 | A 의 가치가 B 와 중복 (사람 가독성) — 비용 정당화 어려움 |
| A 만 보존 (v2.2.0 의 B 제거) | `.html` 비용 0 / 단순 | 백로그 핵심 가치 (사람 가독성 ↑) 손실 |

**근거**: 백로그 D1 핵심. AI 흐름이 `.md` 만 읽으므로 A 의 가치는 사람 전용. B 가 사람 가독성을 더 잘 흡수 (Mermaid / 시각화) → A 중복.

### D-T2. fire-and-forget dispatch — Task tool `run_in_background=true`

| 옵션 | 장점 | 단점 |
|---|---|---|
| **Task tool `run_in_background=true`** ← 채택 | Claude Code 표준 / notify hook 활용 / cancel 가능 | run_in_background 미지원 harness 호환성 (drop in 가능 — sync fallback) |
| Bash background process (`&`) | 단순 | 추적 어려움 / cancel 어려움 / model dispatch 불가 |
| Sync dispatch (메인 대기) | 결과 즉시 확인 | 메인 latency ↑ (수십초 ~ 분), fire-and-forget 의도 무화 |

**근거**: D2 명시. Claude Code Task tool 의 `run_in_background` 가 표준 패턴.

### D-T3. 디바운스 윈도우 — 3초 (U1 결정)

| 옵션 | 장점 | 단점 |
|---|---|---|
| 1초 | 빠른 반응 | 사용자 cancel 인지 거의 X / 연속 fix 시 cancel 빈도 ↑ |
| **3초** ← 채택 | 사용자 fix 1~2개 흡수 (자연스러운 연속 입력 단위) / cancel 인지 가능 | 평균 latency 3초 추가 (의미 X — fire-and-forget 이므로 메인 대기 X) |
| 5초 | 더 많은 fix 흡수 | 사용자 대기 인지 ↑ ("왜 안 하지?" 혼란) |
| 디바운스 없음 (매 fix 즉시 dispatch) | 단순 | 연속 fix 시 비용 ↑ (각 dispatch Sonnet 비용 누적) |

**근거**: 사용자 평균 fix 간격 분석 (전체 패턴 — release 별 dogfood) 시 1~2초 ↔ 5~10초 분기점이 3초 부근. 균형점.

### D-T4. subagent_type — `general-purpose` (U2 결정)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **general-purpose** ← 채택 | DRY (기존 인프라 활용) / 신규 agent 정의 부담 X / prompt 만으로 책임 분리 | agent description 분리 X (B 의 책임이 prompt 안에만 명시) |
| 신규 `html-renderer` 전용 agent | 책임 명확 / 향후 룰 분리 가능 / agent description 검색 가능 | 신규 agent 정의 비용 / `.claude-plugin/plugin.json` agents 항목 추가 / 등록 + reload 부담 |

**근거**: v2.2.0 도 `general-purpose` 사용. 일관성 + 신규 agent 정의 비용 회피. 향후 분리 필요 시 별도 patch.
