# v2.5.1 — Worktree Cleanup 자동화 (merge-back 자동화 + 환경 파일 동기화 + worktree-remove 신규)

**Mode**: socratic-free-form
**Slug**: v2.5.1-worktree-cleanup-auto
**작성일**: 2026-05-25

---

## 배경

v2.5.0 까지의 `/worktree-merge-back` 은 안전성 우선으로 슬림화 (v2.0.5) 되어 있지만, 실제 사용 시 마찰 지점이 남아 있다.

**사용자 신고**:

1. 머지 대상이 `origin/$MAIN_BRANCH` (remote) 라 parent 워크트리의 로컬 브랜치 변경이 반영 안 됨. 실제 의도는 parent 워크트리의 로컬 브랜치 머지.
2. HARD-GATE + Step 3 충돌 게이트 때문에 자동 흐름이 자주 멈춤. 사용자가 원하는 건 "워크트리에서 작업 → parent 로 자동 머지 → 충돌 자동 해결 → 다시 워크트리로 복귀" 의 자동화.
3. 환경 변수 파일 (`.env*`) 같은 gitignored 영역은 git 머지로 못 옮김. 워크트리에서 새 키 추가하면 parent 가 stale 상태로 남음.
4. 워크트리 정리 (`git worktree remove` + `git branch -d`) 가 안내만 있고 명시 명령이 없음. 사용자가 매번 수동 입력.

위 4가지를 v2.5.1 patch 로 해결한다.

---

## 핵심 결정

### D1. 머지 대상 = parent 워크트리의 로컬 브랜치 (origin X)

- 현재 Step 3: `git fetch origin && git merge origin/$MAIN_BRANCH`
- 변경: `git merge $MAIN_BRANCH` (local 만). `git fetch origin` 제거.
- 이유: 사용자 의도는 로컬 작업 동기화. remote 흡수는 사용자가 필요할 때 직접 `git fetch` + 머지.
- 영향: 사용자가 parent 워크트리에서 진행한 로컬 commit 이 머지 흐름에 자연스럽게 반영됨.

### D2. 충돌 처리 = 재귀 머지 자동 시도 + 실패만 사용자 개입

- 현재 Step 3: 충돌 시 항상 `AskUserQuestion` 게이트.
- 변경: git default 동작 (재귀 머지) 자동 시도 후, semantic conflict (실제 conflict marker 발생) 만 사용자 개입.
- `--strategy ours/theirs` 자동 적용은 데이터 손실 위험이라 도입 X. 사용자가 명시 옵션으로만.
- HARD-GATE worktree-only 자체는 유지 (main 워크트리에서 호출 차단). 안전성 핵심.

### D3. 환경 파일 동기화 = LLM 변경 의미 판단 + 선택적 cp

- 워크트리에서 변경된 "로컬 빌드 환경 파일" 목록을 감지 (`.env*`, 기타 plugin/플랫폼별 파일).
- 각 파일의 변경 의미를 LLM 이 판단:
  - 새 환경 변수 키 추가 → parent 로 cp (의미 있음)
  - 임시 디버그 값 / 주석만 변경 → 제외
  - 기존 키의 값 변경 → 사용자에게 짧게 보고 (자동 판단 보류)
- gitignored 영역이라 git 머지 불가. cp 만 가능.
- 사용자 의도 보존: cp 전에 변경 요약 prose 출력. 사용자가 stop 가능.

### D4. `worktree-remove` 신규 슬래시 명령 (worktree-merge-back 와 독립)

- 신규: `commands/worktree-remove.md` + `skills/worktree-remove/SKILL.md`.
- 동작: 현재 워크트리 1개 정리 — `git worktree remove <current-path>` + `git branch -d <current-branch>`.
- chain X. `worktree-merge-back` 와 자동 연결 안 함. 사용자가 명시 호출.
- 기본 브랜치 삭제: safe (`-d`) — 머지 안 됐으면 차단. `-D` 자동은 데이터 손실 위험.
- HARD-GATE: feature 워크트리 안에서만 호출 가능 (`worktree-merge-back` 와 동일 패턴).
- 호출 후 현재 디렉토리가 사라지므로, 종료 메시지에 사용자가 어디로 이동해야 하는지 안내.

## 우려 / 해결

### R1. CLAUDE.md v2.0.4+ Anti-Pattern 표 약화 — 안전성 회귀 위험

CLAUDE.md 의 `## worktree-merge-back skill 결합 (v2.0.4+)` 의 Anti-Pattern 표는 "자동 충돌 해결 도입 → 데이터 손실 위험" 명시. D2 가 이걸 일부 약화.

**해결**: 표 자체는 보존 + "재귀 머지 시도는 안전 (git default 동작이라 실제 충돌 발생 시 conflict marker 만 남기고 stop)" 사례를 추가. `--strategy ours/theirs` 자동 적용은 여전히 차단.

### R2. 환경 파일 cp 의 LLM 오판단 위험

D3 의 LLM 판단이 잘못되면 의도 외 변수 노출 / 의도 변수 누락.

**해결**: 모든 cp 결정을 사용자에게 prose 로 보고 (각 파일별 1줄 요약). 사용자가 stop / 명시 거부 가능. silent cp 금지.

### R3. parent 워크트리의 로컬 브랜치가 stale (다른 곳에서 push 됨) → 머지 후 충돌

D1 이 origin 흡수를 제거했기 때문에, parent 의 로컬 브랜치가 remote 보다 뒤처져 있으면 머지 후 사용자가 push 시점에 충돌.

**해결**: Step 5 종료 메시지에 "remote 와 동기화가 필요하면 `git -C <main-path> pull` 또는 별도 fetch 후 push" 안내 추가. 명시 안내만, 자동 fetch 는 X (사용자 의도 보존).

### R4. worktree-remove 호출 후 cwd 사라짐

D4 의 `git worktree remove $(pwd)` 호출 후 현재 디렉토리가 사라져서 후속 명령 X.

**해결**: 종료 메시지에 "디렉토리 이동: `cd <parent-path>`" 명시. skill 내부에서 자동 cd X (사용자 shell 영향).
