## §1. Architecture overview

### worktree-merge-back 흐름 (수정)

1. HARD-GATE (worktree-only 검사) — 유지
2. Step 1 — Working tree 검사 (자동, dirty 시 종료) — 유지
3. Step 2 — Parent 추론 (자동, 단일 parent 만) — 유지
4. **Step 3 (변경)** — parent 의 로컬 브랜치 흡수 (`git merge $MAIN_BRANCH`, origin 흡수 제거)
   - 충돌 시 git default 재귀 머지 자동 시도
   - semantic conflict (실제 conflict marker 발생) 만 사용자 개입 안내
5. Step 4 — Parent 워크트리로 머지 (`git -C $MAIN_PATH merge ...`) — 유지
6. **Step 4.5 (신규)** — 환경 파일 동기화 (LLM 변경 의미 판단 + prose 보고 + 선택적 cp)
7. **Step 5 (보강)** — 종료 메시지에 `/worktree-remove` 호출 안내 추가 (자동 chain 없음)

### worktree-remove 흐름 (신규 슬래시 명령)

1. HARD-GATE — feature 워크트리 안에서만 호출 (main 워크트리 차단)
2. Working tree dirty 검사 (자동, 변경 있으면 종료)
3. parent 추론 + 현재 워크트리 / 브랜치 캡쳐
4. `git -C $MAIN_PATH worktree remove $CURRENT_PATH`
5. `git -C $MAIN_PATH branch -d $CURRENT_BRANCH` (safe, 머지 안 됐으면 차단)
6. 종료 메시지 — parent 디렉토리 이동 안내 (`cd <parent-path>`)

두 skill 은 독립. 자동 chain 없음.

## §5. Key decisions

### D-1. 머지 대상 = parent 워크트리의 로컬 브랜치 (origin 제거)

**Alternatives**:

- A. `git fetch origin && git merge origin/$MAIN_BRANCH` (현재 v2.0.5) — remote 최신 흡수
- B. `git merge $MAIN_BRANCH` (로컬만) — parent 워크트리의 로컬 commit 만 흡수
- C. 둘 다 옵션 (`--remote` 플래그) — 사용자 명시 시만 origin

**Recommendation**: B. 사용자 의도 (로컬 브랜치 동기화) 그대로 반영. remote 동기화는 사용자가 진입 전 직접 `git fetch` + pull 또는 별도 `/sync-remote` 같은 명령으로 분리.

### D-2. 충돌 처리 = git default 재귀 머지 자동 시도 + 실제 conflict marker 만 사용자 개입

**Alternatives**:

- A. 현재 v2.0.5 — 모든 충돌 게이트 (`AskUserQuestion`)
- B. git default 재귀 머지 자동 시도 + 실제 conflict marker 발생만 prose 안내 + 사용자 수동 해결
- C. `--strategy ours/theirs` 자동 — 한쪽 임의 채택, 데이터 손실 위험

**Recommendation**: B. git default 자체가 안전 (3-way merge 알고리즘, 충돌 시 marker 만 남기고 stop). 사용자가 명시 `--strategy` 플래그 안 주면 ours/theirs 자동 적용 절대 없음.

### D-3. 환경 파일 동기화 = LLM 변경 의미 판단 + prose 보고 + 선택적 cp

**대상 파일 감지**:

- glob 후보: `.env*`, `local.properties`, `gradle-wrapper.properties`, 기타 플랫폼별 로컬 빌드 환경 파일 (`setting-up-worktrees` 의 LLM-judged Procedure와 동일 룰)
- 실제 비교: 워크트리의 git untracked 또는 modified 파일 중 parent 와 내용이 다른 것

**판단 기준 (LLM, 각 파일별 1줄 prose 보고)**:

- 새 환경 변수 키 추가 → cp 권장
- 임시 디버그 값 / 주석만 변경 → 제외
- 기존 키의 값 변경 → 보고 + 사용자 선택 보존 (silent cp 금지)

**cp 방식**:

- `cp -p <feature-file> <parent-file>` (permission 보존, overwrite)
- symlink 처리: `-P` (symlink 보존) default — symlink 발견 시 별도 prose 보고
- 사용자가 prose 응답으로 stop / 명시 거부 가능

### D-4. worktree-remove 신규 슬래시 명령 (chain 없음)

**Alternatives**:

- A. `worktree-merge-back` 의 Step 5 자동 chain
- B. 독립 슬래시 명령 (사용자 명시 호출만)
- C. `--cleanup` 옵트인 플래그로 chain

**Recommendation**: B. 사용자가 "스킬스가 아니라 커맨드 개념" 명시. `worktree-merge-back` 의 종료 메시지에 `/worktree-remove` 호출 안내만 추가 (자동 호출 없음).

### D-5. HARD-GATE worktree-only 유지

**Alternatives**:

- A. 유지 — main 워크트리에서 호출 차단
- B. 완화 — main 워크트리에서도 호출 가능 (사용자 의도 신호로 해석)

**Recommendation**: A. main 워크트리에서 `git worktree remove` 호출 시 자기 자신 정리 — 데이터 손실 위험. 안전성 핵심 보존.

### D-6. 브랜치 삭제 default = safe (-d)

**Alternatives**:

- A. safe (-d) default — 머지 안 됐으면 차단
- B. force (-D) default — 무조건 삭제
- C. safe default + 옵트인 플래그 `--force` 시만 force

**Recommendation**: C. default safe (-d). 사용자가 머지 안 된 브랜치 강제 삭제 원하면 `--force` 명시. 사용자 발언 ("알아서하기엔 좀 위험해서") 그대로 반영.

## §6. Preliminary risks

### R-1. CLAUDE.md v2.0.4+ Anti-Pattern 표 약화 회귀

D-2 가 "자동 충돌 해결 도입 → 데이터 손실 위험" 룰을 일부 완화.

**완화**: 표 자체 유지 + 명시 분기 추가 — "재귀 머지 자동 시도는 안전 (git default 동작, conflict marker 발생 시 자동 stop, ours/theirs 자동 적용 절대 X)". `--strategy ours/theirs` 자동 적용은 여전히 차단 사례로 명시.

### R-2. LLM env 파일 판단 오판 → 의도 외 변수 노출 / 누락

D-3 의 LLM 판단이 잘못되면 secret 노출 또는 누락.

**완화**: silent cp 금지. 각 파일별 변경 의미 1줄 prose 보고. 사용자가 stop / 명시 거부 가능. 보고 형식 예:

```
.env.local 에서 새 키 NEW_API_KEY 추가됨 → parent 로 cp 권장
.env.development 에서 DEBUG_MODE=true 임시 변경 → 제외 권장
```

### R-3. parent 로컬 브랜치 stale → push 시점 충돌

D-1 이 origin 흡수 제거 → parent 의 로컬이 remote 보다 뒤처지면 사용자 push 시점에 충돌.

**완화**: Step 5 종료 메시지에 "remote 동기화 필요 시 `git -C <main-path> pull` 별도 실행" 안내 추가. 자동 fetch 는 안 함 (사용자 의도 보존).

### R-4. worktree-remove 호출 후 cwd 사라짐

**완화**: 종료 메시지에 "cd <parent-path>" 명시. skill 내부 자동 cd 금지 (사용자 shell 영향).

### R-5. 환경 파일 symlink 처리

`.env*` 가 symlink 일 수 있음. cp 시 symlink 깨질 위험.

**완화**: D-3 의 cp 방식에 `-P` (symlink 보존) default 명시. symlink 발견 시 별도 prose 보고 후 사용자 선택.
