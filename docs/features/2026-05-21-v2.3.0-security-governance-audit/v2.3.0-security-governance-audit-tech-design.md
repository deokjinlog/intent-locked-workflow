**호출 path**:

```

  ↓
메인이 한 메시지에 5개 Task 호출 (Sonnet, run_in_background:false)
  ├─ Subagent A (API Cost) — 자체 grep + context7/WebSearch
  ├─ Subagent B (PII)
  ├─ Subagent C (Sensitive Logic)
  ├─ Subagent D (Agent — pre-check 후 PASS 가능)
  └─ Subagent E (Governance)
  ↓ (5개 결과 wait — block 대기)
메인이 raw 리포트 5개 종합 → Subagent F dispatch (별도 1회)
  ↓
F (commands/audit-report-prompt.md 로드 + 5 영역 raw 입력)
  → Write `docs/audit/<timestamp>-audit-risk.html`
  ↓
메인이 사용자에게 1줄 요약 + HTML 경로 출력
```

## §4. 외부 통합 — context7 / WebSearch / Task 도구

### Subagent A 의 `/context7` 호출

- 목적: 외부 SDK pricing 페이지 조회 (openai / anthropic / stripe / twilio / aws-sdk 등)
- 동작: A subagent prompt 안에서 `mcp__plugin_context7_context7__resolve-library-id` + `mcp__plugin_context7_context7__query-docs` 호출
- fallback: context7 결과 부족 시 WebSearch (`<sdk-name> pricing`)
- 두 모두 실패 시: "추정 불가" 라벨 + 호출 위치만 보고 (C8 mitigation)

### Subagent F 의 입력 schema

```
{
  "project_meta": {
    "name": "<from git remote or package.json>",
    "stack": "<detected stacks — Python / Node / Java / etc.>",
    "size": "<rough loc count>"
  },
  "areas": {
    "api_cost": {"findings": [...], "score": <0-100>, "summary": "..."},
    "pii": {"findings": [...], "score": <0-100>, "summary": "..."},
    "sensitive_logic": {"findings": [...], "score": <0-100>, "summary": "..."},
    "agent": {"status": "PASS|ACTIVE", "findings": [...], "score": <0-100>, "summary": "..."},
    "governance": {"findings": [...], "score": <0-100>, "summary": "..."}
  },
  "timestamp": "YYYY-MM-DD-HHMMSS"
}
```

각 `findings[]` element 구조:
```
{
  "file": "src/foo.ts",
  "line_range": "42-58",
  "severity": "Critical|High|Medium|Low",
  "category": "<영역별 sub-category>",
  "title": "<짧은 제목>",
  "description": "<상세>",
  "recommendation": "<권장 조치>",
  "confidence": "high|medium|low"   ← Subagent A 비용 추정에 주로 사용
}
```

### Task 도구 dispatch 패턴

- 5 read-only subagent: `subagent_type: "general-purpose"`, `model: "sonnet"`, `run_in_background: false`
- F (write): 동일하나 `description: "audit-risk HTML report generator"`
- 메인이 한 메시지에 5개 Task 호출 (parallel) → 5 결과 모두 wait → 6번째 Task 호출 (F)

## §5. 핵심 결정 + 대안 비교

### D1. 5+1 subagent 패턴 채택

| 옵션 | 장점 | 단점 | 결정 |
|---|---|---|---|
| **A1. 5 read-only + 1 write (백로그 그대로)** | fresh-context per-area + 일관 종합 + 메인 컨텍스트 절약 | 6 Sonnet 호출 비용 | ✅ 채택 |
| A2. 단일 subagent 5 영역 분담 | 비용 ↓ (1 호출) | 컨텍스트 폭발 + 영역별 깊이 ↓ + fresh-context 깨짐 | ❌ |
| A3. 영역별 독립 커맨드 (`/audit-pii` 등) | 사용자 부분 호출 가능 | 5 HTML 따로 → 일관 종합 안 됨 + UX 분산 | ❌ |

→ **A1**. 비용은 1회성 + 사용자가 의사결정 trigger 라 명확.

### D2. Subagent F (HTML 생성) prompt 분리

| 옵션 | 장점 | 단점 | 결정 |
|---|---|---|---|
| **B1. `commands/audit-report-prompt.md` 별도 파일** | generating-html/html-companion-prompt.md 패턴 답습 + 결합 룰 catch 명확 + diff 추적 쉬움 | 메인이 Read 1회 + inject 단계 추가 | ✅ 채택 |
| B2. `commands/audit-risk.md` 안에 inline | Read 1회 절약 | 본문 비대 + 결합 변경 시 같은 파일 충돌 | ❌ |
| B3. F subagent prompt 를 메인이 동적 생성 | flexibility | reproducibility ↓ + 디버깅 어려움 | ❌ |

→ **B1**. v2.2.4 의 `html-companion-prompt.md` 형제 패턴.

### D3. HTML 보고서 톤 — v2.2.4 mirror (자유)

| 옵션 | 장점 | 단점 | 결정 |
|---|---|---|---|
| **C1. v2.2.4 generating-html mirror (매 호출 자유)** | 일관 디자인 철학 + variety = feature + dogfood 데이터 축적 | 매 호출 다른 톤 → 일부 사용자가 "이상해" 느낄 가능성 | ✅ 채택 |
| C2. dashboard 고정 (가독성 우선) | 예측 가능 | 회귀 — v2.2.4 룰과 충돌 | ❌ |
| C3. 사용자가 톤 인자로 선택 (`/audit-risk --tone=dashboard`) | UX 유연 | scope 폭발 + 인자 schema 복잡 | ❌ (v2.3.x 후보) |

→ **C1**. 단 보고서가 audit 성격이므로 prompt 에 "정보 hierarchy 우선 + 위험도 색깔 절대 명확" 추가 제약 박아둔다.

### D4. 비용 추정 신뢰도 표기 — best/worst + 라벨

| 옵션 | 장점 | 단점 | 결정 |
|---|---|---|---|
| **D1. best/worst case + 신뢰도 라벨 (high/medium/low) + 근거 한 줄** | 사용자 투명 + LLM 추정 한계 명시 | 표시 공간 ↑ | ✅ 채택 |
| D2. 단일 숫자 + 라벨 | 간결 | over-precision 위험 (사용자가 실제 청구서 X 라고 오해) | ❌ |
| D3. 추정 없음 (호출 빈도만) | 안전 | 백로그 목적 안 충족 (사용자가 비용 추정 원함) | ❌ |

→ **D1**.

### D5. 메인 vs F 의 책임 분리

- 메인: 5 subagent dispatch + 결과 수집 + F 입력 schema 조립 + 사용자 1줄 출력
- F: HTML 생성 + write 1 파일. **메인 컨텍스트에 raw 리포트 5개 load 후 즉시 F 로 inject → 메인 컨텍스트 절약** (F 가 raw 리포트 보유, 메인은 요약만 유지)

### D6. Subagent D pre-check fallback

- pre-check 패턴 명시 (백로그 답습): `openai` / `anthropic` / `langchain` / `crewai` / `autogen` / `system prompt` 변수 / tool calling 코드
- pre-check fail → "PASS — 에이전트 위험 없음" 1줄 + 즉시 종료
- pre-check pass → 5개 detect (민감 데이터 접근 / prompt injection / system prompt 노출 / tool 권한 / output 검증)

### D7. 비활성 토픽 — DB schema (§3 N/A)

- 코드 변경 X, 산출물 HTML 1 파일 + 폴더 1개. DB / migration / schema 영역 모두 N/A.

## §6. 위험 — risk-annotation 3-checklist 매핑

| R-id | 카테고리 | 위치 | 시나리오 | 완화 |
|---|---|---|---|---|
| **R-1** | side-effect | 5 subagent 병렬 호출 | 5 Sonnet 동시 호출 → 청구 spike | 사용자 명시 호출만, 자동 발동 경로 X. 보고서 상단 비용 disclaimer. |
| **R-2** | side-effect | Subagent A 의 context7 / WebSearch | 외부 호출 실패 시 추정 누락 | fallback "추정 불가" 라벨. 호출 위치만 보고 (C8). |
| **R-3** | breaking | 신규 `commands/audit-risk.md` slash | 기존 매칭과 충돌 가능성 | unique slug + 자동 발동 X. README "주요 커맨드" 표 등록 시 중복 확인. |
| **R-4** | side-effect | F subagent 가 `docs/audit/` write | 사용자가 깜빡하고 commit → repo 노이즈 | `.gitignore` 차단 + 사용자 명시 add 만 진입 가능 (C6). |
| **R-5** | breaking | `.gitignore` append | 기존 룰 silently 무력화 위험 | append-only (기존 라인 보존), grep 0 검증 후 commit. |
| **R-6** | race | 메인이 5 subagent 결과 wait | 1개 fail → 종합 실패 | F prompt 에 `area.status: "failed"` 분기 — 일부 실패해도 부분 보고서 생성. |
| **R-7** | side-effect | F 가 self-contained HTML 보장 못 함 (외부 CDN 참조 등) | offline 깨짐 / 보안 leak | F prompt 의 v2.2.4 mirror 룰 명시 + verification ("외부 URL 0 검증") 박음. |
| **R-8** | breaking | CLAUDE.md 결합 메모 누락 | 후속 변경 시 atomic patch 깨짐 | 결합 메모 obligatory + 회귀 catch grep 박음 (다른 결합 섹션 패턴 mirror). |
| **R-9** | side-effect | Subagent E 의 secrets hardcoding 검출 → 보고서에 secret 노출 | HTML 에 평문 secret 표시 → 누가 보면 leak | F prompt 에 "secret 의심 패턴은 redact (`****`) 후 file:line 만 보고" 명시. |
| **R-10** | side-effect | Subagent A~E 가 false-positive 폭주 (모든 grep noise 를 critical 처리) | 사용자 noise 폭주 → 무시 패턴 | 각 prompt 에 위험도 매트릭스 엄격 적용 + dead code 는 Low. HTML 에 위험도 filter UI. |

## §7. 성능 / 비용 — 1회성 트레이드오프

- **호출**: Sonnet × 5 (병렬) + Sonnet × 1 (F) = **6 Sonnet calls** + 약간의 context7 / WebSearch (Subagent A 안). 1회 호출당.
- **메인 컨텍스트**: 5 raw report 를 메인이 잠깐 보유 → F dispatch 후 폐기 (D5). 영구 누적 X.
- **시간**: 5 subagent wall-clock ≈ 가장 느린 1개 + F ≈ 30~120s 추정 (프로젝트 규모 따라). 1회성이라 사용자 wait 허용.
- **빈도**: 자동 발동 X. 사용자 명시 호출만. 누적 비용 미미.
- **opt-out**: 사용자가 호출 안 하면 0 비용. 디폴트 비활성.
