### Task 1 — `commands/audit-risk.md` 신규 (메인 dispatch 본문)

**Files (Create)**: `commands/audit-risk.md`
**Model**: sonnet
**Deps**: (없음)

`/audit-risk` 슬래시 커맨드 본문 작성. 메인이 5+1 subagent dispatch 흐름 따라가도록 단계별 안내. 구조:

1. **헤더 + 사용 안내 (한 줄)** — "프로젝트 전체 보안 + 거버넌스 + 비용 1회성 감사. 코드 변경 X, HTML 보고서 1개 산출."
2. **출력 폴더 보장** — `mkdir -p docs/audit/` 실행 단계
3. **5 read-only subagent 병렬 dispatch** — 한 메시지에 5 `Task` 호출 (`subagent_type: "general-purpose"`, `model: "sonnet"`, `run_in_background: false`). 각 prompt 본문 inline 포함 (A/B/C/D/E 영역별 책임 + 검출 패턴 + 위험도 매트릭스 + 출력 schema)
4. **결과 수집** — 5 결과 모두 wait. 1건 fail 시 `area.status: "failed"` 마킹 + 나머지로 계속 (R-6 mitigation)
5. **Subagent F dispatch** — `commands/audit-report-prompt.md` Read → fresh-context inject + 5 raw 리포트 + 프로젝트 메타 inject. 1 Task 호출 (sonnet)
6. **사용자 1줄 요약 출력** — `✅ 감사 완료: docs/audit/<ts>-audit-risk.html` + 영역별 발견 카운트 + 점수
7. **Disclaimer** — 비용 추정 best-effort / 외부 보안 도구와 보완 관계 / secret redaction 적용

**TDD**:
1. fixture H23 의 모의 프로젝트 (의도된 hardcoded secret + 외부 API loop + PII log) 에 `/audit-risk` 호출 시 5 영역 모두 Critical 1+ 검출 + HTML offline 렌더 OK
2. 빈 프로젝트 (소스 0) → 5 영역 "위험 없음" 보고서 + Subagent D "PASS" 1줄

**RISK**: side-effect (5 Sonnet 병렬 호출 비용 spike → 자동 발동 X + 보고서 상단 disclaimer)

**Acceptance**:
- `commands/audit-risk.md` 가 위 7 단계를 명확히 안내
- 5 subagent prompt schema (검출 대상 / 위험도 매트릭스 / 출력 형식) 가 모두 inline 명시
- Subagent F 의 prompt 파일 경로 (`commands/audit-report-prompt.md`) 명시
- 자동 발동 안 됨 (slash 명시 호출만)

### Task 2 — `commands/audit-report-prompt.md` 신규 (F subagent prompt)

**Files (Create)**: `commands/audit-report-prompt.md`
**Model**: sonnet
**Deps**: (없음 — Task 1 과 file-disjoint 병렬 가능)

Subagent F 의 fresh-context prompt. v2.2.4 generating-html mirror + audit-risk 특수 룰. 구조:

1. **헤더 — "감사 보고서 HTML 생성기"** + 목적
2. **입력 schema** — `project_meta` + `areas` (A/B/C/D/E 각각 findings[] + score + summary) + `timestamp`. tech-design §4 의 입력 schema 그대로 답습
3. **Output target** — `docs/audit/<timestamp>-audit-risk.html` (timestamp = `YYYY-MM-DD-HHMMSS`)
4. **Design Philosophy (v2.2.4 mirror)** — "Wow first" / 톤 inspiration 6종 / variety = feature. 단 audit 성격이라 정보 hierarchy + 위험도 색깔 명확 추가 제약.
5. **적극 권장 시각 요소** — hero header + 영역별 score donut/radar + 위험도 카드 grid + filter UI (Critical/High/Medium/Low) + sticky TOC + dark-mode toggle + tab/accordion + sort/search
6. **인터랙션** — `<details>` / tab / scroll-spy / copy 버튼 (file path) / filter / sort / search (전부 inline JS)
7. **Self-contained rules** — inline CSS/SVG/JS 만, 외부 CDN 0 (offline 렌더)
8. **Secret redaction rule (R-9)** — secret 의심 패턴 (hardcoded API key / password / JWT secret) 은 보고서에서 `****` 마스킹, file:line 만 표시
9. **FORBIDDEN** — raw secret value 노출 / 외부 URL 참조 / `.md` 본문 (이 경우 raw report) 의역 / Korean ↔ English 번역 / 5 영역 외 임의 추가
10. **Disclaimer 상단 출력** — "best-effort LLM 감사. 정확한 청구서 X. 외부 보안 도구 (Snyk / SonarQube / Semgrep) 와 보완 관계. CVE 자동 매칭 X."
11. **Verification before writing** — `findings[]` 모두 file:line 있는지 / 외부 URL 0 / secret 마스킹 적용 / score 0~100 범위 / timestamp 형식 valid
12. **How to apply** — 5 단계 (Read 입력 → 톤 선택 → HTML 생성 → write → 보고)

**TDD**:
1. fixture H23 의 모의 raw 리포트 (5 영역 + project_meta) 입력 → F 가 self-contained HTML 1 파일 write + 외부 URL 0
2. 의도된 secret 포함 raw 리포트 → HTML 에 `****` 마스킹 적용 (raw secret value 미노출)
3. Subagent D "PASS" 상태 raw 리포트 → HTML 에 "에이전트 위험 없음 — 본 영역 비활성" 1줄 + 다른 4 영역 정상 표시

**RISK**: side-effect (F 가 self-contained 보장 못 함 → R-7 mitigation, verification 단계 grep)

**Acceptance**:
- Prompt 본문에 입력 schema / 톤 룰 / secret redaction / verification 모두 명시
- `commands/audit-risk.md` 의 Subagent F dispatch 단계와 prompt 경로 일치
- v2.2.4 mirror — 사무 톤 금지 + variety = feature 명시
- FORBIDDEN 룰 명시 (raw secret 노출 / 외부 CDN / 의역 / 5 영역 외 추가)
