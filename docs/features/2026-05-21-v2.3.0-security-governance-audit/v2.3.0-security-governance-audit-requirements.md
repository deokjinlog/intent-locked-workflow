# v2.3.0 — 프로젝트 전체 보안 & 거버넌스 감사 커맨드 `/audit-risk`

Mode: Socratic (auto-flow)

## 배경

js-super 가 이미 코드 컨텍스트와 도메인 지식을 알고 있는 상태인데, 사용자 입장에서 "이 프로젝트 전체에 산발적으로 누적된 위험" 을 한눈에 파악할 도구가 없다. 외부 보안 스캐너 (Snyk / SonarQube / Semgrep) 는 도메인 모르고, grep 으로 직접 훑기는 양이 많다. 5 카테고리의 누적 위험:

1. **API 비용 폭주** — 외부 LLM / payment / SaaS API 를 loop 안 호출 / 무한 retry / cache 미적용
2. **개인정보 (PII) 노출** — log / response / 외부 송신 / 평문 저장에서 누출
3. **민감 로직 결함** — 사용량 카운터 / 비용 산정 / 결제 처리 의 race / off-by-one / 검증 누락
4. **에이전트 위험** — LLM 에이전트가 민감 데이터 접근 + prompt injection 취약
5. **거버넌스 회피** — 인증 우회 / RBAC 누락 / audit log 결여 / hardcoded secrets / SQL injection / XSS / CSRF

→ js-super 가 LLM 으로 한 번 훑어주는 게 가장 빠름. 1회성, 코드 수정 X, HTML 리포트 1개.

## 핵심 결정

### 채택 approach — A1: 5+1 subagent 패턴 (백로그 그대로)

| 단계 | 책임 | 모델 | 병렬성 |
|---|---|---|---|
| Subagent A | 외부 API 사용처 + 비용 분석 + `/context7` + WebSearch 단가 조회 | Sonnet | 병렬 |
| Subagent B | PII 처리 (필드 keyword grep / 평문 저장 / log / 외부 송신) | Sonnet | 병렬 |
| Subagent C | 사용량·비용·결제 로직 결함 (race / off-by-one / 트랜잭션 / idempotency) | Sonnet | 병렬 |
| Subagent D | 에이전트 위험 (pre-check 후 LLM 코드 없으면 PASS 1줄 + 즉시 종료) | Sonnet | 병렬 |
| Subagent E | 인증 / RBAC / audit log / secrets / SQL injection / XSS / CSRF | Sonnet | 병렬 |
| Subagent F | A~E raw 리포트 + 프로젝트 메타 → `docs/audit/<timestamp>-audit-risk.html` 1 파일 생성 | Sonnet | 직렬 (5 결과 wait 후) |

**왜 A1**: 백로그가 명시. 5 영역 + 1 종합 = 6 subagent 1회 호출. 영역별 독립 fresh-context 보장 + 종합 단계에서 일관 디자인. read-only 5 + write 1 안전.

**대안 A2 (단일 subagent + 5 영역 자체 분담)** — 거부: 한 subagent 컨텍스트에 5 영역 모두 로드 시 폭발 + 영역별 깊이 ↓. fresh-context per-area 보장 안 됨.

**대안 A3 (영역별 독립 커맨드 — `/audit-pii` / `/audit-api-cost` 등)** — 거부: 5번 따로 호출 + 5 HTML 따로 → 일관 리포트 안 됨. 사용자 시점 한 번에 보고 싶음.

## 우려 / 해결

| 우려 | 해결 |
|---|---|
| **C1** Subagent A 의 비용 추정 부정확 → 사용자 오해 (실제 청구서와 차이) | best/worst case + 신뢰도 라벨 명시 + 추정 근거 한 줄. "정확한 청구서 X, 추정 가이드" 라고 보고서 상단 disclaimer. |
| **C2** false-positive 폭주 (5 영역 모두 grep 기반 → noise ↑) | 각 subagent prompt 에 "위험도 분류 (Critical/High/Medium/Low) 엄격 적용 + dead code / 잠재 위험 은 Low" 명시. HTML 에서 위험도 filter UI 제공 (사용자가 Critical 만 먼저 보기 가능). |
| **C3** 5 subagent 병렬 비용 (Sonnet × 5 + F × 1 = 6 Sonnet) | 백로그 명시 답습 (1회성 비용 명확). 호출 빈도 ↓ → 누적 비용 미미. 사용자가 직접 의사결정용 trigger. |
| **C4** Subagent D 의 pre-check false-negative (LLM 코드 있는데 못 잡고 PASS) | pre-check 패턴 명시 (openai / anthropic / langchain / crewai / autogen / `system prompt` / tool calling 코드). pre-check fail 시 fallback — 메인이 의심되면 D 재dispatch 옵션 (v2.3.x). |
| **C5** Subagent F 의 HTML 디자인이 v2.2.4 generating-html 룰과 drift | F prompt 가 별도 파일 (`commands/audit-report-prompt.md`) 분리 — generating-html 의 v2.2.4 톤 룰 mirror. 변경 시 결합 룰 catch grep. |
| **C6** `docs/audit/` 폴더 무제한 누적 → 디스크 ↑ (사용자 잊어버림) | gitignored + 사용자 직접 정리 책임 명시 (보고서 상단 안내). 향후 보존 정책 도입 시 `.audit-config` 같은 별도 파일. |
| **C7** 외부 보안 도구와 결과 차이 (Snyk 등) → 사용자 혼란 | 보고서 상단 disclaimer — "이 보고서는 도메인 인지 LLM 감사, 외부 도구 (Snyk / SonarQube / Semgrep) 와 보완 관계. CVE 자동 매칭 X, 컴플라이언스 인증 X". |
| **C8** Subagent A 의 `/context7` + WebSearch 결과 부족 시 비용 추정 실패 | fallback: 단가 unknown 시 "추정 불가" 라벨 + 호출 빈도 + 호출 위치만 보고. 사용자가 pricing 직접 조회 권장. |
