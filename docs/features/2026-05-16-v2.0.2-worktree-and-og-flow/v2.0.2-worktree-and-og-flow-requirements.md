## 배경

v2.0.2 release 는 **두 개의 결이 가까운 workflow polish 를 묶어 처리**한다. 두 건 모두 v2.0.0/v2.0.1 dogfood 직후 사용자 catch.

### 1. worktree skill 의 `.env*` 한정 한계 (dogfood finding)

`skills/setting-up-worktrees/SKILL.md` 의 auto-copy glob 룰이 `.env*` 한정 → web/Node 전용. Android (`local.properties`, keystore), iOS (`xcconfig`, `GoogleService-Info.plist`), desktop 등 빌드 환경에서는 워크트리 생성 후 수동 복사 필요. 워크트리의 zero-friction 가치 손실.

같은 패턴 다른 플랫폼:

| 플랫폼 | 누락되는 파일 (현재) |
|---|---|
| Android | `local.properties` / `keystore.properties` / `*.keystore` / `*.jks` |
| iOS | `*.xcconfig` (local) / `GoogleService-Info.plist` (일부) / `Secrets.swift` |
| Desktop / 일반 | `secrets.properties` / `secrets.json` / `config.local.*` |

### 2. og-flow ↔ js-super subagent 누수 의심 (dogfood catch)

사용자가 `/og-brainstorm` → `/og-design` → `/og-write-plan` 흐름 끝, Execution Handoff 게이트에서 "Subagent-Driven" 선택 시 — og-writing-plans SKILL.md 가 명시적으로 `superpowers:subagent-driven-development` (upstream untouched) 를 가리킴에도 불구하고 — 메인 에이전트가 `js-super:js-super-subagent-driven-development` (wave-parallel + RISK + 변경이력) 을 발화하는 현상 목격.

og 흐름은 upstream 원본 스타일 (per-task fresh subagent + 2-stage review + commit-per-task) 이어야 하는데 js-super 의 wave-parallel + governance 가 끼어들면 og 컨셉 자체가 손상.

**의심되는 원인**: skill 이름 매치 충돌 — `subagent-driven-development` 가 두 skill 이름에 모두 들어있고 (`subagent-driven-development` 와 `js-super-subagent-driven-development`), 후자 이름의 `js-super-` prefix 가 더 두드러져 매치 우위. 메인이 `superpowers:` prefix 를 흘려보내고 `js-super:js-super-subagent-driven-development` 매칭.

## 핵심 결정

### 1. worktree 일반화 — "레시피" 형태 LLM-judged copy

skill 본문 `.env*` hardcoded glob → **"로컬 빌드 환경 파일"** 개념 rename. 본문에 플랫폼별 예시 patterns 박지만 **deterministic glob 으로 강제하지 않음** — LLM 이 프로젝트 컨텍스트 (Android / iOS / Node / Flutter / Rust / etc.) 보고 적절한 후보 판단 + 복사.

- 본문에 박히는 것: **레시피 / 가이드** (플랫폼별 known 예시 + "로컬 빌드 환경 파일" 컨셉 설명)
- 본문에 박히지 않는 것: hardcoded glob 강제 룰 / Python helper 자동 탐지
- 결과: 새 플랫폼 (Flutter / RN / Rust 등) 등장 시 skill body patch 불필요. LLM 이 레시피 컨셉에 매핑.

#### 거부된 안 — Approach B (.gitignore 기반 자동 감지)

루트 gitignored 파일 모두 자동 복사. false positive (build output / IDE dir / `.DS_Store`) 잘못 catch 시 워크트리 오염. **거부 — v2.1+ 후속 검토.**

#### 거부된 안 — Approach A (명시 patterns list 확장, deterministic glob)

기존 backlog 본문의 채택안이었으나 **거부됨** — 사용자 명시 의도가 "레시피 컨셉으로 본문 박고 LLM 판단". 새 플랫폼 추가 시 patch 비용 없는 LLM-judged 패턴 우위.

## 우려 / 해결

### O-1. rename breaking 비용 — 기존 plan 의 hint / commands / CLAUDE.md / scripts 일제 갱신

해결: tech-design 단계에서 grep 으로 모든 참조 catch + 변경 list 명시 (예상 영향: CLAUDE.md / commands/auto-*.md / auto-executing-plans SKILL.md / preflight.py 호출자 / 기존 features 폴더 plan 의 `**Model**:` 안내 본문). 기존 features plan 본문은 immutable (변경 X) — history 보존.

### O-2. worktree LLM-judged 의 비결정성 — 세션마다 다른 파일 복사 가능

해결: skill 본문에 "로컬 빌드 환경 파일" 정의 + 플랫폼별 예시 patterns 명시 → LLM 의 판단 폭 제한. 모호 케이스 (`google-services.json` committed vs gitignored) 본문에 케이스별 룰 명시.

### O-3. og-flow B (handoff 강화) 만으로는 매칭 모호성 잔존 가능

해결: A (rename) 이 root cause fix. B 는 future-proof 안전망. 둘 다 채택 → 이중 안전망.

### O-4. 보안 우려 — `*.keystore` / `*.jks` 워크트리 복사

해결: `.worktrees/` 가 gitignored 라 git 노출 0. 같은 머신 안 복사라 외부 노출 확대 없음. 기존 `.env` 복사 정책 ("이미 디스크에 있는 파일, 같은 머신 안 복사") 그대로 유지. 사용자 명시 exclude 옵션은 기존 HARD-GATE 그대로.

### O-5. iOS Firebase / Android google-services 등 ambiguous 케이스

해결: 일부 프로젝트는 committed (git checkout 으로 자동 복원), 일부는 gitignored (복사 필요). skill 본문에 케이스별 룰 명시 — "committed 인 경우 git checkout 으로 자동 복원, gitignored 인 경우만 복사". 또는 무조건 복사 (덮어쓰기 0 + 같은 머신 안 동일 파일이라 정확성 유지).

### O-6. fragile 파일 — `local.properties` 의 머신 절대 경로

해결: 워크트리는 same-machine 운영이라 깨짐 없음. 다른 머신 동기 의도면 별도 issue. 본 release scope 밖.
