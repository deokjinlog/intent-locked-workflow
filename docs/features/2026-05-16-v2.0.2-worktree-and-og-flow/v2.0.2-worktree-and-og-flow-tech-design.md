**채택**: A 단독으로 root cause 해소 가능하지만 B 추가 → 이중 안전망.

**대안 비교**:

| 안 | 장점 | 단점 | 채택 |
|---|---|---|---|
| A 단독 (rename `js-super-sub-driven`) | root cause fix / 영구 / matching 충돌 0 | breaking — 14 파일 갱신 | ✅ (B 와 결합) |
| B 단독 (handoff 강화만) | 비파괴 / 작은 변경 | rename 안 하면 매칭 모호성 잔존 | ❌ |
| A + B (채택) | root cause + future-proof | 비용 ↑ (단 1회 작업) | ✅ |
| C (`/og-execute-plan` 신규) | 명시 라우팅 | A 하면 unnecessary / 이미 commands/og-execute-plan.md 존재 catch | ❌ |
| D (검증만) | 비용 0 | root cause 미해결 | ❌ |

**Catch**: tech-design 단계 grep 결과 `commands/og-execute-plan.md` **이미 존재** (전 사용자 인지). 요구사항 본문의 "C 신규" 표현은 부정확 — 실제로는 og-execute-plan 이 존재하지만 subagent path 가 upstream `subagent-driven-development` 를 가리키지 않음 (inline 만). C 는 거부 유지 — rename 자체로 해결됨.

### D-T3. worktree 일반화 — 레시피 형태 LLM-judged (deterministic glob X)

**채택**: skill body 에 플랫폼별 레시피 + "로컬 빌드 환경 파일" 컨셉 → LLM 이 프로젝트 컨텍스트 보고 후보 판단 + 사용자 노출 + 복사.

**대안 비교**:

| 안 | 장점 | 단점 | 채택 |
|---|---|---|---|
| Approach A (deterministic glob list) | 결정론적 / 동일 결과 보장 | 새 플랫폼 patch 필요 / Flutter/Rust 등 미적용 | ❌ |
| Approach B (.gitignore 기반 자동 감지) | 플랫폼 무관 / 새 플랫폼 자동 cover | false positive (build output / IDE dir / .DS_Store) | ❌ (v2.1+ defer) |
| 레시피 + LLM-judged (채택) | 새 플랫폼 patch 불필요 / 컨텍스트 적응 | 비결정성 (세션마다 다를 수 있음) | ✅ |

**비결정성 mitigation**: skill body 에 "로컬 빌드 환경 파일" 정의 + 플랫폼별 known 레시피 명시 (Android / iOS / Node / Desktop) → LLM 판단 폭 제한. 모호 케이스 (Firebase config committed vs gitignored) 본문 케이스별 룰.

### D-T4. worktree 레시피 본문 구조 — "Procedure" 섹션 prose + 레시피 sub-section

**채택**: Step 2 의 bash one-liner 폐지 → Procedure 본문 prose ("메인 에이전트가 프로젝트 컨텍스트 read → 후보 list 사용자 노출 → 복사"). 본문 하위 레시피 sub-section 으로 플랫폼별 예시.

**대안 비교**:

| 안 | 장점 | 단점 | 채택 |
|---|---|---|---|
| bash one-liner + 레시피 prose 병기 | hybrid (자동 + LLM) | 모순 — bash 가 hardcoded 면 LLM-judged 의미 X | ❌ |
| Procedure prose 단독 (채택) | LLM-judged 일관 / 본문 정독 시 명확 | bash 자동화 손실 (단 zero-friction 가치 LLM 가 회복) | ✅ |
| Python helper 자동 탐지 | 결정론적 | hardcode glob 본질로 회귀 (Approach A 의 변형) | ❌ |
