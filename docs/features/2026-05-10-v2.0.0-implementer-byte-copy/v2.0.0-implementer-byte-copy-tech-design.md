## 1. 아키텍처 개요

### 변경 전 (v1.1.x)

```
Wave 안 task:
  Implementer (sonnet, plan 의 **Model**: hint 사용)
    → LLM transcription via Edit/Write
    → Status: DONE 또는 spec-reviewer fix loop
  Spec Reviewer (sonnet)
    → line-by-line 검증
```

### 변경 후 (v2.0.0+)

```
Wave 안 task:
  Implementer (haiku 고정, byte-copy only)
    → Edit old=원본/new=수정후 byte-identical
    ├── 성공 → Spec Reviewer
    └── BLOCKED (원본 mismatch) → Reorder Stage 발화
  Reorder (sonnet, conditional)
    → 사용자 수정 + plan 변경 reconcile
    ├── 성공 → Spec Reviewer
    └── NEEDS_USER → 메인 AskUserQuestion 발화 → 사용자 지시 → 메인 직접 reconcile → Spec Reviewer
  Spec Reviewer (sonnet, 변경 X)
    → line-by-line 검증
```

3-stage 분담 핵심: 각 stage 가 단일 책임 (byte-copy / reconcile / 검증). 모델 + prompt 분리로 silent 위험 제거.

### Plan-side 정밀도 강화

`scripts/plan_byte_check.py` 신규 helper. writing-plans + auto-writing-plans 의 self-review 단계에서 호출. plan 의 모든 `**원본**` 블록이 실제 파일과 byte-equal 인지 자동 검증, 미스매치 시 plan 저장 거부 (fail-fast).

본 release 안 wave 1 = v1.1.16 fix:

### D-T2. Implementer haiku 고정 (Model 힌트 무시)

byte-copy 는 LLM 추론 거의 불요 (Edit tool 의 old/new 인자 = byte-copy). haiku 가 충분.

| Alternatives | 평가 |
|---|---|
| **haiku 고정** ✅ | 토큰 ~50% 절감 / 응답 ↑ / byte-copy 정확도 충분 |
| sonnet 유지 | 비용 ↑, byte-copy 정확도 차이 미미 |
| Model 힌트 존중 | 사용자가 byte-copy 모드 인식해야 함 (마찰) |

→ **haiku 고정 채택** (PRD D11). plan 의 `**Model**:` 필드는 일반 `/execute-plan` (inline) 에서만 사용.

### D-T3. Reorder prompt — Status NEEDS_USER 형식

reorder 가 NEEDS_USER escalate 시 메인에게 전달할 형식. 메인은 `AskUserQuestion` 으로 사용자 게이트.

```
Status: NEEDS_USER
Reason: <한 줄 — 무엇이 충돌했는지>
Conflict:
  <plan 의 변경 의도 한 줄>
  vs
  <사용자 수정 한 줄>
Files: <영향 받는 file:line>

  - <메인이 사용자에게 물을 만한 질문 1>
  - <질문 2>
```

| Alternatives | 평가 |
|---|---|
| **structured Status: NEEDS_USER 본문** ✅ | 메인이 즉시 AskUserQuestion 으로 변환 가능 |
| free-form 한 줄 | 메인이 다시 파싱 — 토큰 낭비 |
| reorder 가 직접 사용자 게이트 호출 | 서브에이전트가 사용자와 직접 대화 X — js-super 안전 룰 위반 |

→ **structured Status 채택**.

### D-T4. plan_byte_check API

```python
@dataclass(frozen=True)
class Mismatch:
    plan_path: Path
    block_index: int          # plan 안 N번째 **원본** 블록
    file_path: Path           # 참조된 file (plan 의 (file:line) annotation 에서 추출)
    expected: str             # plan 의 **원본** 블록 내용
    actual: str               # 파일의 실제 해당 위치 내용
    reason: str               # human-readable

def verify_plan_block_byte_equal(
    plan_path: Path,
    repo_root: Path
) -> List[Mismatch]:
    """plan 의 모든 **원본** 블록이 실제 파일과 byte-equal 인지 검증.
    
    plan 안 task block 에서:
    - **원본** (`<file>:<line-range>`):
      ```<lang>
      <expected content>
      ```
    
    `<file>` 을 repo_root 기준 resolve 후 `<line-range>` 추출.
    expected vs actual byte-equal 비교. 미스매치 시 Mismatch 추가.
    
    빈 List → all pass.
    """
```

| Alternatives | 평가 |
|---|---|
| **List[Mismatch] return** ✅ | self-review 가 모든 미스매치 한 번에 사용자에게 노출 |
| 첫 미스매치 raise | self-review fail-fast 좋지만 다중 미스매치 batch 한꺼번에 못 보여줌 |
| ok/fail bool | 디버깅 정보 손실 |

→ **List[Mismatch]** 채택. caller (writing-plans / auto-writing-plans) 가 비어있지 않으면 plan 저장 거부 + Mismatch 사용자에게 노출.

### D-T5. SKILL.md Per-wave Sequence W-2 분기

기존 W-2 (Pair-parallel dispatch) 는 implementer + spec-reviewer 페어. v2.0.0 패치:

W-2. Pair-parallel dispatch

### D-T7. v1.1.16 fix 1 — `scripts/dag_builder.build_waves` deps ordering

**버그**: Kahn topological sort + file-overlap conflict 감지 결합 시 deps 우선순위 위반. 예: T13 deps=[T12] 인데 file overlap 으로 T13 이 T12 보다 먼저 wave 배치됨.

**Fix 방향**: Kahn 의 indegree 0 후보 중 file overlap 적용 시 strict deps 우선. 즉:
1. Kahn 알고리즘으로 indegree 0 후보 추출
2. 후보 중 file overlap 없는 task 들만 같은 wave 에 묶기
3. file overlap 있는 task 는 다음 wave 로 deferred — 단 deferred 결정 시 deps 우선순위 보장 (deps 가 더 뒤에 있는 task 먼저 wave 배치 X)

**Test fixture**: `scripts/tests/test_dag_builder.py` 신규 케이스
```python
def test_strict_deps_ordering_with_file_overlap():
    """T1, T2 둘 다 동일 file 만짐. T2 deps=[T1].
    Kahn indegree 0 후보 = [T1]. T2 deps 만족 후에만 wave 배치.
    file overlap 으로 T1, T2 동일 wave 안 배치 X.
    """
```

### D-T8. v1.1.16 fix 2 — `scripts/preflight._has_changelog_entries` regex

**버그**: markdown code block 안 리터럴 `## 변경이력` 도 footer 로 매치. 예: docs-pretty 가 plan 본문 안 code block 의 `## 변경이력` 보고 "이미 logged" 잘못 판정.

**Fix 방향**: `text.rsplit("## 변경이력", 1)` 로 마지막 occurrence 만 검사. footer 는 항상 파일 끝에 있으므로 last occurrence = footer.

**Test fixture**: `scripts/tests/test_preflight.py` 신규 케이스

    """plan 본문 안 ```markdown ... ## 변경이력 ... ``` 가 footer 로 매치되지 않음."""
