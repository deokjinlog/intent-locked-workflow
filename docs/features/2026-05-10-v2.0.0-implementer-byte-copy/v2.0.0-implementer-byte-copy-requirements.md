> **STRICT BYTE-COPY (v2.0.0+)**: Edit tool's `old_string` MUST be byte-identical to `**원본**` block content. `new_string` MUST be byte-identical to `**수정 후**` block content. ### D4. BLOCKED → Reorder dispatch (조건부)

## 3. 우려 / 해결

### 우려 1 — plan-side 정밀도 부담 ↑ (가장 큰 trade-off)

`**원본**` 블록이 실제 파일과 byte-equal 보장 필요. 현재 룰은 권장 정도, byte-copy 모드에서는 미스매치 = false-fail.

**해결**: `scripts/plan_byte_check.py` (D6) 가 plan 작성 self-review 단계에서 catch. plan 저장 거부 (fail-fast).

### 우려 2 — markdown 들여쓰기 / 트레일링 공백 false-fail

plan code block 의 들여쓰기가 살짝 다른 경우 byte-equal 깨짐.

**해결**: plan_byte_check 가 작성 단계에서 catch. implementer 단계 normalize 룰 추가 X (drift 위험 — backlog 거부 권장).

### 우려 3 — 다중 Edit task 부분 실패

task 가 5건 sequential Edit 호출, 3번째 fail 시 1~2번 이미 적용. byte-copy 는 LLM 판단 X.

**해결**: implementer BLOCKED 시 메인이 wave finalization 단계에서 `git checkout -- <task.files>` rollback. 기존 failure isolation 패턴 재활용.

### 우려 4 — 사용자가 LLM 자율 보정 선호

drift 제거 = AI 자율 보정 기능 제거. 일부 사용자 PEP 8 자동 적용 등 선호 가능.

**해결**: byte-copy 는 subagent 모드만 (D7). inline executing-plans 는 LLM 패턴 유지. 사용자 선택 보존.

### 우려 5 — v1.1.16 + v2.0.0 묶음 scope 비대칭

v1.1.16 fix 2건 = micro-patch, v2.0.0 byte-copy = 8 파일 변경. 비대칭 scope.

**해결**: release 안 task 분리 명확 (D9). wave 1 = bugfix, wave 2~ = byte-copy. dogfood 단계에서 v1.1.16 fix 효과 자연스럽게 검증.

### 우려 6 — plan-runner Python tool 도입 (선택사항)

byte-copy 가 deterministic 하다면 LLM 거치지 않아도 됨. plan 파싱 → Edit/Write 자동 실행 가능.

**해결**: 본 release 미포함. v1.1.X 후속 검토 (LLM self-review 손실 trade-off 평가 필요).
