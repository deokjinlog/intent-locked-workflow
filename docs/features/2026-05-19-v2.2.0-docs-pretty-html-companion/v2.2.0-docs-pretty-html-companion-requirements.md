# v2.2.0 — `docs-pretty` `.html` 동봉본 (사람용 가독성 사본)

> **Mode:** Backlog promotion (auto-flow, brainstorming 생략 — 백로그 `docs/backlog/v2.2.0-docs-pretty-html-companion.md` 가 이미 brainstorming 수준으로 정리됨)
> 작성: 2026-05-19

## 한 줄 요약

`docs-pretty` 발동 시 **두 subagent 를 병렬 dispatch** — A (기존, `.md` format-only) + **B (신규, `.html` 시각화 사본)**. AI 흐름은 항상 `.md` 만 읽음 (영향 0), `.html` 은 사람이 브라우저로 빠르게 훑는 용도. Mermaid / 풍부 표 / 인쇄 가능 / 다크모드 자동.

## 배경

js-super 의 3-MD (`<slug>-requirements.md` / `<slug>-tech-design.md` / `<slug>-implementation-plan.md`) 는 **AI 가 읽기 좋은 포맷**으로 작성됨. 표·헤더·코드블록 활용도 `docs-pretty` 가 잡아주지만:

- 사람이 긴 MD 를 stream 으로 훑을 때 **구조 파악 + 관계 시각화** 가 약함
- 다이어그램 / 풍부한 표 / 시각 hierarchy 필요한 정보 (DAG, 데이터 흐름, 컴포넌트 관계) 가 텍스트로만 노출
- 결과: 사용자가 `docs-pretty` 처리된 MD 를 받아도 **빠른 한눈 이해가 어려움**

## 핵심 결정

### D1. 산출물 형태 — `.md` (AI + 사람) + `.html` (사람 전용)

기존 `.md` 와 같은 폴더, 같은 basename, 다른 확장자:

```
docs/features/2026-05-19-foo/
├── foo-requirements.md      ← AI + 사람
├── foo-requirements.html    ← 사람 전용 (신규)
├── foo-tech-design.md
├── foo-tech-design.html
├── foo-implementation-plan.md
└── foo-implementation-plan.html
```

**근거**: AI 흐름 변경 0 보장. `.md` 가 single source of truth, `.html` 은 derived view.

### D2. dispatch 패턴 — 두 subagent 병렬 (`docs-pretty` Procedure 수정)

| Subagent | 산출물 | 대상 독자 |
|---|---|---|
| A (기존) | `.md` (format-only pass) | AI + 사람 |
| B (신규) | `.html` (시각화 사본) | **사람만** |

**병렬 dispatch 명세** (`docs-pretty` SKILL.md 의 Procedure 섹션 수정):

- Run two Task tool calls in a single message (parallel)
- Subagent B receives: prettified `.md` (A 출력 또는 RAW `.md` — 메인 reconcile)
- Strict prompt: "preserve meaning byte-equivalently at the semantic level, enhance visual layout only"
- 메인은 둘 다 완료 대기 후 validate (footer 보존 / `.html` 존재 / basename 매칭)

**근거**: 단일 subagent 분기보다 병렬이 latency ↓ + 책임 분리 명확.

### D3. 의미 보존 절대 — semantic 1:1 매핑

- `.md` 의 문장 / 항목 / 결정 사항 / 숫자 / 표 내용 1:1
- 의미 누락 X / 의역 X / 재구조화 X
- spot-check: `.html` semantic drift 의심 시 메인이 제목·항목 수 매칭

**근거**: `.html` 은 derived view. source-of-truth 와 diverge 시 사용자 혼란 + AI/사람 reading mismatch.

### D4. self-contained — 외부 의존 0

- inline CSS + inline SVG
- Mermaid (필요 시 inline JS 임베드 또는 미리 SVG 컴파일 — D11 미정)
- 외부 CDN (jQuery / 외부 CSS) → **금지**
- 인쇄 / offline (브라우저 더블클릭) 가능

**근거**: cross-machine 배포 + offline 리뷰 시 의존성 깨짐 방지.

### D5. 다크/라이트 자동 — `prefers-color-scheme`

OS 설정에 따라 자동 전환. JS 토글 X (D4 의 self-contained 와 충돌하지 않는 범위).

### D6. 동기화 boundary — `docs-pretty` 와 동일

- **초기 작성 iteration loop 에서만** 발동 — `.md` 와 `.html` 둘 다 매 draft 마다 재생성
- **첫 `change-history` entry 가 logging 되는 순간 STOP** — live doc 진입. 이후 부분 수정 시 `.html` 자동 재생성 X
- live doc 진입 후 `.html` stale → 무시 (`.md` 가 source). 사용자 원하면 명시 invoke 로 재생성

**근거**: 기존 `docs-pretty` boundary 그대로 답습. 비용 controllable.

### D7. 실패 모드 처리

| 상황 | 처리 |
|---|---|
| A 성공 / B 실패 | `.md` 만 갱신 + 사용자에게 "`.html` 생성 실패, 수동 retry 가능" 알림 |
| A 실패 / B 성공 | 둘 다 폐기 (A 가 핵심 — `.md` 없으면 의미 X) |
| A·B 둘 다 실패 | `docs-pretty` 자체 abort, RAW `.md` 유지 |
| `.html` semantic drift 의심 | B 결과 메인이 spot-check (제목·항목 수 매칭) — drift 발견 시 폐기 |

**근거**: `.md` priority. `.html` 은 best-effort.

### D8. 검증 fixture

`skills/docs-pretty/tests/H17-html-companion/` 신규 (README.md 만, 메타 dogfood):

- `.md` 와 `.html` basename 1:1 매칭
- `.md` 본문 문장 수 vs `.html` 문장-level node 수 차이 5% 이내 spot-check
- offline 검증: `.html` 더블클릭 → 표·다이어그램 렌더링 OK
- 다크모드 자동 전환 OK
- live doc 진입 후 추가 `docs-pretty` 발동 → `.html` 재생성 X (boundary 보존)

## 미정 사항 (designing-direction 에서 결정)

다음 5건은 `<slug>-tech-design.md` 의 §5 결정+대안 비교 단계에서 alternatives + recommendation 으로 결정:

### U1. Subagent B 모델 — Sonnet vs Haiku

- Sonnet: 시각화 풍부, Mermaid / 표 디자인 품질 ↑, 비용 ↑
- Haiku: 비용 ↓, 단순 .html 변환 만 — 시각화 풍부도 ↓
- recommendation candidate: Sonnet (사람 가독성 핵심 + `docs-pretty` 발동 빈도 낮음)

### U2. `.html` gitignore 여부

- commit (사람 리뷰용 cross-machine 공유) vs gitignore (AI 가 쓰지 않으므로 노이즈)
- recommendation candidate: gitignore + 사용자 옵션 (필요 시 commit)

### U3. Mermaid 렌더 전략

- inline JS 임베드 (mermaid.min.js 인라인) vs 미리 SVG 컴파일 (Subagent B 가 SVG 직접 생성)
- recommendation candidate: inline JS (B 의 시각화 자유도 ↑)

### U4. fix 매 턴 재생성 비용

- 매 fix iteration `.html` 재생성 (Sonnet 비용 2N) vs 명시 invoke 시만 `.html` 재생성 (D6 boundary 변형)
- recommendation candidate: 매 fix 재생성 (boundary 일관성) + 향후 cost monitoring 시 옵션 분기

### U5. 시각화 휴리스틱

- Subagent B 가 어느 표가 다이어그램 (Mermaid) 로 가야 할지 — 명시 룰 (B prompt 에 룰 박음) vs LLM 자율
- recommendation candidate: 명시 룰 최소 (DAG / 의존성 / 흐름 / 컴포넌트 관계 → Mermaid) + LLM 자율 보완

## 비목표 (Non-goals)

- 실시간 `.html` 미리보기 서버 / VSCode preview 통합
- `.html` 다국어 (영문/한글 자동 분기) — `.md` 가 그대로 한국어면 `.html` 도 그대로
- `.html` 인터랙티브 (탭/접힘/검색) — 정적 페이지만
- `.html` ↔ `.md` 양방향 sync (`.html` 수정 → `.md` 반영) — `.md` 만 source
- `code-pretty` 의 코드 산출물 `.html` 동봉 — 본 변경 범위 외 (별도 release)
- 기존 `.md` 산출물 의 `.html` retroactive 백필 — 새 발동만 적용

## 우려 / 해결

### O1. Subagent B 실패 시 무한 retry

- 1회 시도 후 실패 → `.md` 만 갱신 + 사용자 알림 (D7)
- 자동 retry X. 사용자 명시 invoke 만.

**Mitigation**: D7 실패 모드 매트릭스로 흡수.

### O2. `.html` semantic drift (의미 손상)

- B 의 prompt 가 "format-only" 명시 + 메인 spot-check
- drift 발견 시 B 결과 폐기 → `.md` 만 갱신

**Mitigation**: D3 의미 보존 룰 + D7 drift 폐기.

### O3. 외부 CDN / inline JS 의존

- D4 의 self-contained 룰로 차단
- Mermaid inline JS 임베드 시 (U3 옵션) 도 self-contained 빌드 (CDN 참조 X)

**Mitigation**: B prompt 에 명시 "외부 URL 참조 금지".

### O4. `.html` 비용 누적

- 초기 작성 iteration N 회 × 2 subagent — Sonnet × 2N 호출
- 사용자 fix 마다 추가 누적
- U4 의 옵션 분기로 cost monitoring 후 흡수 가능

**Mitigation**: D6 boundary (live doc 진입 후 STOP) + U4 옵션 분기.

### O5. AI 가 `.html` 읽음

- AI 흐름 모든 skill 은 `.md` 만 읽음 (영향 범위 0 보장)
- `.html` 은 사람 전용

**Mitigation**: 영향 범위 섹션 명시 + B prompt 에 "AI 흐름과 무관" 박음.

### O6. live doc 진입 후 `.html` stale

- D6 boundary 의 trade-off — 의도된 동작
- 사용자가 원하면 명시 invoke 로 재생성

**Mitigation**: 안내문 한 줄 (`docs-pretty` 발동 시점 안내).

### O7. v2.0.3 AskUserQuestion 강제 룰 적용

- `docs-pretty` 는 8 skill 룰 적용 대상 **아님** (`code-pretty` 와 같이 보조 skill)
- 본 변경에서도 `docs-pretty` 본문에 AskUserQuestion 강제 boilerplate 박지 않음

**Mitigation**: 영향 범위 명시. 사용자 게이트 추가 X (자동 발동 흐름).
