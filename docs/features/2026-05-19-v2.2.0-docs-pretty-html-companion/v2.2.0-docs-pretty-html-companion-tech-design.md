## 1. 아키텍처 개요

`docs-pretty` 본문의 Procedure Step 5 (Dispatch) 를 **단일 subagent → 두 subagent 병렬 dispatch** 로 변경. 신규 prompt 파일 1개 + fixture 1개 + manifest bump.

흐름:

```
docs-pretty 발동 (기존 Pre-flight 통과 후)
   │
   ▼
[Step 5] 두 subagent 병렬 dispatch (한 메시지 안 Task tool 2회)
   ├─ Subagent A (Sonnet) : .md format-only pass (기존 동작)
   └─ Subagent B (Sonnet) : .html 시각화 사본 생성 (신규)
   │
   ▼
[Step 5.5] 메인 reconcile + validate
   ├─ A 결과 → .md 갱신 (footer 보존 검사)
   ├─ B 결과 → .html 생성 (basename 매칭 + spot-check)
   └─ 실패 매트릭스 (D-T7) 적용
   │
   ▼
[Step 6] 사용자 리뷰 (기존 동작)
```

핵심 분리:
- **A 와 B 는 독립 dispatch** (병렬, 의존 0). 한쪽 실패가 다른 쪽 진행 막지 않음
- **A 가 single source of truth** (`.md` 우선) — A 실패 시 B 폐기 (D7)
- **B 는 derived view** (`.html`) — B 실패 시 `.md` 만 갱신
- **boundary 보존** — 기존 `docs-pretty` 의 "초기 작성 iteration 만 발동 + change-history 첫 entry 후 STOP" 그대로

## 5. 핵심 결정 + 대안 비교

### D-T1. Subagent B 모델 — Sonnet (U1 결정)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **Sonnet** ← 채택 | 시각화 풍부도 ↑ (Mermaid / 표 디자인 / typography 결정) / 의미 보존 + 시각화 동시 추론 가능 | 비용 ↑ (Haiku 대비 ≈ 5x) |
| Haiku | 비용 ↓ | 단순 `.md` → `.html` 변환만 — 시각화 디자인 판단력 부족. Mermaid 적합 표 판단 어려움 |
| Opus | 시각화 최고 품질 | overkill (`.html` 1 회당 비용 부담 ↑) |

**근거**: `docs-pretty` 발동 빈도 낮음 (per-feature × few iterations) → 비용 부담 controllable. Sonnet 의 추론력이 "어느 표를 Mermaid 로 변환할지" 같은 휴리스틱 판단에 필수.

### D-T2. `.html` gitignore 여부 — gitignore (U2 결정)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **gitignore** ← 채택 | `.md` 가 single source / 매 fix iteration 재생성 → commit 시 변경이력 폭주 / repo 무게 ↓ | cross-machine 공유 어려움 (사용자가 zip / drive 등 다른 채널) |
| commit | cross-machine 공유 자연 / PR 리뷰 시 immediate | 매 iteration 재생성 → git churn 폭주 / `.md` 와 정보 중복 |
| 사용자 옵션 (frontmatter 토글) | 유연 | 결정 부담 ↑ / 기본값 명시 필요 |

**근거**: `.html` 은 derived view (D3 의미 보존 1:1) — `.md` 가 있으면 재생성 가능. commit 가치 ↓. 필요한 사용자는 `.gitignore` exception 으로 cherry-pick.

### D-T3. Mermaid 렌더 전략 — inline JS (U3 결정)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **inline JS (mermaid.min.js 자체 임베드)** ← 채택 | self-contained (CDN X) / Subagent B 가 mermaid 코드만 생성 (출력 안정) / 인터랙티브 가능 (zoom / pan) | `.html` 파일 크기 ↑ (mermaid.min.js ≈ 2-3MB) |
| 미리 SVG 컴파일 | 파일 크기 ↓ / 의존성 0 | Subagent B 가 SVG 직접 생성 → 출력 품질 변동성 ↑ / 인터랙티브 X |
| 외부 CDN 참조 (unpkg) | 파일 크기 ↓ | D4 self-contained 위반 / offline 깨짐 |

**근거**: D4 self-contained 룰 준수 + Subagent B 의 출력 안정성 우선. 파일 크기 부담은 `.html` gitignore (D-T2) 로 흡수 (repo 무게 ↑ X).

### D-T4. fix 매 턴 재생성 비용 — 매 fix 재생성 (U4 결정)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **매 fix 재생성** ← 채택 | D6 boundary 일관성 / `.html` 항상 최신 / 사용자 별도 명령 X | Sonnet × 2N 비용 누적 (per-feature ≈ 3 × $0.003 ≈ $0.01) |
| 명시 invoke 시만 재생성 | 비용 ↓ | boundary 변형 / `.html` stale 위험 / 사용자가 별도 명령 기억 부담 |

**근거**: 평균 fix iteration 2-3회 → 누적 비용 controllable. 사용자 cost monitoring 후 v2.2.x patch 에서 옵션 분기 여지 (향후 백로그 후보).

### D-T5. 시각화 휴리스틱 — 명시 룰 최소 + LLM 자율 (U5 결정)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **명시 룰 최소 + LLM 자율** ← 채택 | 핵심 패턴 (DAG / 흐름) 만 명시 → 일관성 / 나머지 (표 디자인 / 색) LLM 자유 → 유연 | prompt 명시 룰 작성 필요 |
| 명시 룰 완전 (모든 시각화 케이스 룰 박음) | 결정적 / 출력 예측 가능 | 룰 폭주 / 새 패턴 대응 어려움 |
| LLM 완전 자율 | prompt 단순 | 출력 변동성 ↑ / DAG 같은 핵심 패턴 누락 위험 |

**명시 룰** (Subagent B prompt 의 "Visual heuristics" 섹션):

- DAG / 의존성 / 흐름도 / 컴포넌트 관계 → Mermaid 다이어그램
- 표 → 정렬·줄무늬·sticky header
- 코드 블록 → syntax highlight (Prism.js inline 또는 inline `<pre><code>` 클래스)
- 헤더 hierarchy → 색·여백·typography 차별화
- cross-reference (CH-id → 영향범위 등) → anchor 링크

**LLM 자율**: 색 팔레트 / 다크모드 색 조정 / 표 정렬 디테일 / 폰트 선택.

### D-T6. Subagent dispatch 방식 — 병렬 두 호출 (한 메시지 안 Task tool 2회)

| 옵션 | 장점 | 단점 |
|---|---|---|
| **병렬 두 Task tool 호출** ← 채택 | latency ↓ (A 와 B 동시) / 책임 분리 명확 / 한쪽 실패 다른 쪽 무관 | 메인이 두 결과 reconcile 필요 |
| 순차 (A 끝나고 B 시작) | 단순 / B 가 prettified `.md` 받음 (drift 위험 ↓) | latency × 2 / A 실패 시 B 진입 차단 |
| 단일 subagent 가 `.md` + `.html` 둘 다 생성 | dispatch 1회 | 책임 비대 / 한쪽 실패 시 둘 다 폐기 / model 선택 어려움 |

**근거**: A 와 B 는 의미상 독립 (D3 의미 보존 룰로 B 가 RAW `.md` 받아도 충분). 병렬이 latency / 책임 분리 모두 우월.

**Reconcile 룰**: 메인이 B 에 입력으로 prettified `.md` 또는 RAW `.md` 전달. 일반적으로 RAW `.md` (A 와 동시 시작이라 A 출력 대기 X). drift 위험 ↓ — D3 룰이 prompt 에 박혀있어 RAW 와 prettified 의 의미 동일.

### D-T7. 실패 모드 처리 매트릭스 — requirements D7 그대로

| 상황 | 처리 |
|---|---|
| A 성공 / B 실패 | `.md` 만 갱신 + 사용자에게 ".html 생성 실패, 수동 retry 가능" 알림 |
| A 실패 / B 성공 | 둘 다 폐기 (A 가 핵심) |
| A·B 둘 다 실패 | `docs-pretty` 자체 abort, RAW `.md` 유지 |
| semantic drift 의심 | B 결과 메인이 spot-check (제목·항목 수 매칭) — drift 발견 시 폐기 |

**spot-check 알고리즘** (메인 수행):
- `.md` 의 H1/H2/H3 헤더 개수 vs `.html` 의 `<h1>/<h2>/<h3>` 개수 — ±0 매칭 강제
- `.md` 의 코드 블록 개수 vs `.html` 의 `<pre><code>` 개수 — ±0 매칭 강제
- 문장 수 (line 단위 wc -l) 차이 5% 이내

## 6. 위험/사이드이펙트 (preliminary)

| R | 카테고리 | 위치 / 트리거 | Mitigation |
|---|---|---|---|
| R1 | side-effect | Subagent B 가 `.md` 의미 손상 (의역 / 누락) | D3 룰 + B prompt 명시 + 메인 spot-check (D-T7) |
| R2 | breaking | A 실패 시 B 가 stale `.html` 만 생성 → `.md` 와 mismatch | D-T7 매트릭스: A 실패 → B 폐기 강제 |
| R3 | side-effect | Mermaid inline JS 파일 크기 (≈ 2-3MB) | `.html` gitignore (D-T2) 로 repo 무게 ↑ 차단. 사용자 disk 누적은 매 fix 재생성이라 최신만 유지 (덮어쓰기). |
| R4 | side-effect | live doc 진입 후 `.html` stale | D6 boundary 의 의도된 동작. 안내문 한 줄. 사용자 명시 invoke 로 재생성 가능. |
| R5 | breaking | 외부 CDN 의존성 의도치 않게 도입 | B prompt 에 "외부 URL 참조 금지" 명시 + Anti-Patterns 표 + 메인 grep 검증 (선택적) |
| R6 | side-effect | AI 가 실수로 `.html` 읽음 → 의미 drift 흐름 | 영향 범위 명시 + B prompt 에 "AI 흐름과 무관" 박음. 다른 skill 본문에 `.html` 참조 0 확인. |
| R7 | breaking | `.html` cross-machine 공유 어려움 (gitignore) | 안내문 (`docs-pretty` 발동 후 한 줄): "`.html` 은 로컬 전용. PR 리뷰용 공유는 zip / drive 권장" |
| R8 | side-effect | Subagent B 비용 누적 | D-T4 의 매 fix 재생성 + 향후 옵션 분기 여지. v2.2.x patch 후보. |
| R9 | race | A 와 B 가 동시 같은 파일 write 시 충돌 | A 는 `.md`, B 는 `.html` — 파일 disjoint. 충돌 없음. |
| R10 | side-effect | spot-check 가 false-positive drift (정상 출력을 drift 로 오판) | spot-check 룰 보수적 (헤더 / 코드블록 count ±0 만 강제, 문장 수 5% tolerance) + 폐기 시 `.md` 는 보존 (A 결과). 사용자 손해 X. |
