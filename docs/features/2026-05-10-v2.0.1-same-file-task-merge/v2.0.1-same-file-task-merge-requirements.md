# 요구사항: v2.0.1 same-file mechanical task — plan 작성 시 1 task 묶음 룰

> **Mode:** Socratic (auto-flow)
> **For agentic workers:** This document is the PRD. NEXT STEP: invoke `auto-designing-direction` skill (or `js-super:designing-direction`) to produce `<slug>-tech-design.md`.

## 1. 배경

### dogfood finding

v2.0.0 release 직후 사용자가 별도 외부 피처 (Android Compose chat 화면) 에서 `/auto-write-plan` 발화. 결과 plan 8 task 중 T5~T8 모두 `ChatScreen.kt` 한 파일만 만짐:

| Task | Files | 내용 |
|---|---|---|
| T5 | `ChatScreen.kt` | Scaffold contentWindowInsets + imePadding |
| T6 | `ChatScreen.kt` | 자동 스크롤 (`LaunchedEffect(messages.size)`) |
| T7 | `ChatScreen.kt` | BackHandler (drawer-우선) |
| T8 | `ChatScreen.kt` | 빈 화면 placeholder + 비서 톤 + 예시 prompt 3 |

### 문제

- file overlap 으로 같은 wave 진입 X → **4 wave 직렬**.
- v2.0.0 의 wave-parallel 가치 (wall-clock 단축) **0** — `js-super-subagent-driven-development` 핵심 이점 손실.
- **4 commit fragmentation** — PR review 단계에서 squash 강요됨.
- TDD 분리 의미도 낮음 — UI 한 Screen 의 modifier / handler 추가는 mechanical 수준. 별도 테스트 경계 없음 (composable preview / 통합 UI 테스트는 한 번에 진행).

### Root cause

`writing-plans` skill 본문 task 분해 가이드:

> Each step is one action (2-5 minutes)
> Tasks bigger than ~30 minutes → Decompose further

이 룰이 "단일 modifier 추가도 별도 task" 까지 적용 → over-decomposition. 진짜 의도는 **TDD cycle 단위 분해** 인데, 같은 cycle 안 mechanical 변경까지 분리.

## 2. 핵심 결정 (D1~D8)

### D1. 룰 = backlog 의 3 조건 AND 그대로 채택

`writing-plans` / `auto-writing-plans` skill 본문에 아래 룰 추가:

> **Same-file mechanical 묶음 룰 (v2.0.1+)**: 둘 이상의 logical change 가 다음 **세 조건 모두** 만족하면 1 task 의 multi-step 으로 묶는다.
>
> 1. **같은 파일** — Files 목록 동일
> 2. **테스트 경계 없음** — 한 통합 test 또는 UI preview 로 같이 검증 가능
> 3. **mechanical** — modifier 추가 / handler 등록 / Scaffold 옵션 / placeholder text 등 (= 알고리즘 변경 X)
>
> 셋 중 하나라도 어기면 분리.

**왜 (a) 채택**:

- 안전판 3중 — 하나 약해도 나머지 둘이 catch
- 보수적 default → 명백 케이스만 묶음, 모호 시 분리 = v2.0.0 동작 유지 → **회귀 0**
- backward compat — 기존 plan 영향 0
- fixture H12 정의 deterministic

**거부된 안**:

- "1 task = 1 file" 단순화 — TDD cycle 깨짐 (mechanical + algorithmic 섞임 / multi-test 비대화). 깨끗하지만 false positive 비용 ↑.
- "(1)+(2) 만 (mechanical drop)" — mechanical 안전판 사라져 algorithmic 변경 묶임 위험. test 경계 모호성 그쪽으로 쏠림.

### D2. Self-Review catch 단계 추가

`writing-plans` Self-Review 항목 + `auto-writing-plans` Step 2 끝 자체 검토 에 추가:

> Same-file mechanical 묶음 룰 위반 검사 — task 들 중 같은 파일만 만지는 chain 이 2건 이상 있으면 D1 의 3 조건 재검토 → 묶을지 결정.

메인이 룰 적용 깜박했을 때 안전망. auto 모드도 동일 catch (메인이 직접 결정, 사용자 응답 wait X).

### D3. multi-step task 의 byte-copy 정합성 — 첫 step 만 검증 (defer 가정)

multi-step task 의 step N 의 `**원본**` 은 step N-1 적용 후 파일 상태 기준이어야 byte-equal. plan_byte_check helper (v2.0.0) 는 plan 작성 시점 디스크 파일과만 비교 → 첫 step 만 정확 검증.

**가정**: mechanical 변경들은 대부분 file 의 다른 위치 (independent insertions — Scaffold 옵션 + LaunchedEffect + BackHandler + placeholder 처럼 별도 라인 영역). step 1 적용해도 step 2~N 의 `**원본**` 영역 안 건드림 → 자연 byte-equal.

가정 깨지면 implementer Stage 1 BLOCKED → Stage 2 reorder dispatch 가 처리 (v2.0.0 패턴).

→ plan_byte_check 시뮬레이션 확장 (step N-1 적용 후 가상 파일 만들고 step N 원본 검증) 은 본 release 미포함. 향후 dogfood 마찰 발생 시 v2.0.x 후속 검토.

### D4. Python 자동 merge helper (Approach B) — v2.1+ defer

`scripts/dag_builder.py` 에 same-file-only chain 자동 검출 + 라인 영역 disjoint 검사 → 자동 task collapse. 결정론적 + plan 작성자 룰 모르고 분해해도 자동 catch.

**거부 이유 (본 release)**:

- 자동 merge 가 라인 영역 detect 잘못 catch 시 v2.0.0 byte-copy 룰 깨짐 → BLOCKED → reorder dispatch 비용 ↑
- DAG 와 plan 본문 분리 보존 시 spec-reviewer / commit message 처리 복잡

→ **defer to v2.1+** — D1 룰 운영 결과 보고 자동화 검토.

### D5. fixture H12 신규 — 메타 dogfood

`skills/js-super-subagent-driven-development/tests/H12-same-file-merge/README.md` — 같은 파일 mechanical 4 변경 plan → 1 task multi-step 으로 묶이는지 검증 시나리오. ChatScreen.kt 4 변경 케이스 그대로 fixture 화.

## 3. 우려 / 해결

### 우려 1 — mechanical 판정 모호성

메인 에이전트가 "이 변경 mechanical?" 판단해야. 사용자 동의 안 할 수 있음.

**해결**:
- 룰 본문에 mechanical 예시 명시 (modifier 추가 / handler 등록 / Scaffold 옵션 / placeholder text 등)
- 사용자가 plan 검토 단계에서 "분리해" 또는 "묶어" override 가능
- auto-flow 도 메인 자체 검토 (Step 2 끝 catch)
- 보수적 default — 애매하면 분리 (false negative 회복 비용 < false positive)

### 우려 2 — 1 task 비대화

같은 파일에 logical change 10건 묶이면 task 비대화.

**해결**: 룰에 "최대 5건까지 merge" cap 명시 가능. 단 5+ 건이면 보통 mechanical 만으로 끝나지 않음 (구조 변경 동반) → 자연 분리. 명시 cap 은 본 release 보류, 첫 dogfood 결과 보고 결정.

### 우려 3 — multi-step byte-copy 정합성

D3 에서 다룸 — independent insertions 가정 + BLOCKED → reorder 안전망. plan_byte_check 시뮬레이션 확장 defer.

### 우려 4 — spec-reviewer 검증 부담 ↑

multi-step task 1건 vs single-step task N 건 비교 시 spec-reviewer 가 한 번에 N 변경 검증 → review 깊이 ↓ 가능.

**해결**: spec-reviewer prompt 에 "multi-step task 의 경우 step 별로 각각 검증" 명시. 단 같은 task 안 변경이라 cross-reference 자연. tech-design 단계에서 prompt patch 명세 확정.

### 우려 5 — RISK 분류 step 단위

multi-step task 안 한 step 만 RISK 트리거 (다른 step trivial) 면 task 단위 RISK 분류 over-conservative 가능.

**해결**: wave finalization 메인 RISK 3-checklist 에 step 별 분리 가능. RISK 주석은 코드 라인 직접 박아 정확.

### 우려 6 — 진짜 atomic multi-file 변경 (예: interface + impl) 분리 강제

D1 의 (1) "같은 파일" 조건이 다른 파일 atomic 변경을 분리 강제.

**해결**: 본 룰은 "묶음 catch" 만 다룸 — multi-file atomic 은 별도 issue (v2.0.0 도 동일 분리). 본 release 미포함.
