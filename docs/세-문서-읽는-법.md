## 세 장의 성격이 다르다

| | 성격 | 답하는 질문 | 읽는 사람 |
|---|---|---|---|
| **① 요구사항** | **계약** | 뭘 하고, 뭘 **안** 하나 | 사람 · 기획 |
| **② 개발방향** | **도면** | 왜 이 방법인가 | 사람 · 리뷰어 |
| **③ 구현계획서** | **공정표 + 부품 전부** | 어떤 순서로 짓나 | **실행자가 그대로 따라감** |

③만 유일하게 **기계가 그대로 집행하는 문서**입니다. 그래서 코드 · 명령어 · 기대출력이 다 박혀 있어요. ①②는 사람이 **판단하려고** 읽고, ③은 **실행하려고** 읽습니다.

④ 실행은 문서를 만들지 않습니다. 규칙이 *"고친 문서에 기록을 남긴다"* 인데, ④는 문서가 아니라 **코드**를 고치므로 자기가 소비한 ③ 하단에 기록만 남깁니다.

## 축이 두 개다 — 가장 헷갈리는 곳

> **"② 변경이력에 ①과 뭐가 달라졌는지 적는 건가요?"** — 아닙니다.

축이 **두 개**입니다. 이걸 하나로 보면 계속 어긋나요.

```
                      가로축 — 문서 사이 (번역)
       ① 요구사항  ───→  ② 개발방향  ───→  ③ 구현계획서  ───→  ④ 코드
            │                │                │
   세로축   │  CH-001        │  CH-002        │  CH-003      ← 만든 날 (최초 생성)
 (변경이력) │  CH-004        │  CH-005        │  CH-006      ← 바뀐 날
            ▼                ▼                ▼
```

**가로 = 번역.** ②가 ①을 *고치는* 게 아니라, ①을 **입력으로 받아 다른 층을 쓰는** 겁니다. 층이 달라요.

**세로 = 시간.** 그 문서 한 장이 v1 → v2 로 바뀌는 것. **변경이력은 이 세로축입니다.**

그래서 `② 개발방향`의 변경이력은 —

- ❌ "①과 뭐가 다른가"
- ✅ "**이 ② 문서가** 언제 · 왜 이렇게 됐나"

### 그럼 "문서 사이 연결"은 어디 있나

변경이력이 아니라 **본문**에 있습니다.

| 연결 | 어디에 |
|---|---|
| ① → ② | ② `§2 영향 컴포넌트` 표가 **FR 번호를 가리킴** (`FR-1,3,8 → auth_service.py`) |
| ② → ③ | ③ 헤더의 `Spec inputs:` (`요구사항 — FR-1~8 / 개발방향 — 결정 A~E`) |
| 바뀌면 누가 영향받나 | 변경이력의 **`영향범위`** 칸 |

둘이 만나는 지점은 딱 하나 — 변경이력의 `영향범위` 칸이 *"내가 바뀌면 저기도 바뀐다"* 를 가리킬 때뿐입니다.

## 기획서는 ①만 읽는다

```
기획서 (PPT · PDF · 회의록)
      │
      ▼
 ① 요구사항     ← 여기서만 기획서를 읽는다
      │
      ▼
 ② 개발방향     ← 기획서 안 봄. ① + 기존 코드를 봄
      │
      ▼
 ③ 구현계획서   ← 기획서 안 봄. ② + 기존 코드 실물을 봄
      │
      ▼
 ④ 코드         ← 기획서 안 봄. ③만 봄
```

②③④는 원본 기획서를 **다시 읽지 않습니다.** ①이 정본이 되니까요.

**이게 "의도 잠금"의 실체입니다.** 만약 ③에서 기획서를 다시 보면 — ①에 *"xlsx 는 범위 밖"* 이라고 못 박아 둔 걸 무시하고, 기획서의 "엑셀 파일" 이라는 표현을 보고 xlsx 지원을 만들어 버립니다. 한 번 잠근 의도가 풀리는 거죠.

각 단계는 **바로 위 문서**만 재료로 씁니다.

| 쓸 때 | 실제로 읽는 것 |
|---|---|
| ① 요구사항 | 기획서 + 사용자 답변 + 기존 코드(대조용) |
| ② 개발방향 | **①** + 기존 코드베이스 |
| ③ 구현계획서 | **②** + 기존 코드 실물(Read — `원본` 블록을 byte 단위로 채우려고) |
| ④ 코드 | **③** |

## 용어 — FR · NFR · AC · CH

| 약자 | 원어 | 한국어 | 예 |
|---|---|---|---|
| **FR** | Functional Requirement | 기능 요구사항 | `FR-5: CSV 파일 1개를 업로드해 다수 항목을 한 번에 등록한다` |
| **NFR** | Non-Functional Requirement | 비기능 요구사항 | `NFR-1: 1회 1파일, 최대 10MB` |
| **AC** | Acceptance Criteria | 수용 기준 | `AC-5: 10MB 초과 파일은 거부되고 안내가 뜬다` |
| **CH** | Change History | 변경이력 항목 id | `CH-20260524-004` |
| **R** | Risk | 위험 번호 (② §6) | `R3: 근거 배열이 비어도 응답이 나감` |

세 개만 기억하면 됩니다.

- **FR** = 뭘 하나 (`~한다`)
- **NFR** = 얼마나 · 어디까지 (숫자)
- **AC** = 됐나 안 됐나 (**예/아니오로 답 가능**해야 함)

> **AC 를 "예/아니오로 답 가능"하게 쓰라는 이유** — 그대로 **테스트가 되려고** 그럽니다. [계획서는 어떻게 생기나](#계획서는-어떻게-생기나) 참조.

### 번호를 붙이는 진짜 이유

장식이 아닙니다. **번호가 있어야 `영향범위`를 grep 으로 찾아냅니다.**

- FR 에 번호가 없으면 → ② 가 뭘 가리키는지 못 씀
- ② §2 매핑 표가 없으면 → FR 이 어느 파일로 갔는지 모름
- 결정에 `A~E` 라벨이 없으면 → 뒤집힌 전제를 못 찾음

## 변경이력 — 비교표가 아니라 항해일지

### 언제 생기나 — 문서를 만든 그 순간

"다음 문서부터" 가 아닙니다. 세 장 **전부** 태어나자마자 첫 엔트리가 박힙니다.

```
① 요구사항    → CH-001 [요구사항-수정]    "영향범위: 없음 (최초 생성)"
② 개발방향    → CH-002 [개발방향-수정]    "영향범위: 없음 (최초 생성)"
③ 구현계획서  → CH-003 [구현계획서-수정]  "영향범위: 없음 (최초 생성)"
```

서로를 비교한 게 아니라, 각자 **"나 태어났음 + 왜 이렇게 태어났음"** 입니다.

### 뭘 적나

공통 스키마 — 어느 문서든 이 다섯 칸:

```markdown
### [YYYY-MM-DD HH:MM] [요구사항-수정 | 개발방향-수정 | 구현계획서-수정 | 코드-수정 | 검증 | 릴리즈 | API테스트]
- **id**: CH-YYYYMMDD-NNN
- **이유**: 왜 이렇게 됐나
- **무엇이**: 어느 섹션 · 파일이
- **영향범위**: 어느 하류 문서 · 코드까지 번지나
- **연관 항목**: CH-... (없으면 생략)
```

`[코드-수정]` 엔트리만 ③ 하단에 붙고, 칸이 더 있습니다 — `위험 카테고리`(`side-effect` · `breaking` · `race` 중), `세부 변경 (N건)`(파일:줄 단위 목록), `변경 전 코드` / `변경 후 코드`.

### 두 번째 엔트리는 언제 생기나 — 두 경우

**(a) 문서가 바뀔 때** — 나중에 "동의어로도 검색되게 해줘" 가 들어오면:

```
① 요구사항   → CH-004 [요구사항-수정]
                 이유: 동의어 검색 요청
                 영향범위: 개발방향 §5, 계획서 Task 6      ← ★여기가 연결
② 개발방향   → CH-005 [개발방향-수정]
                 이유: CH-004 때문에 결정 E(정규화 안 함) 뒤집음
③ 구현계획서 → CH-006 [구현계획서-수정]
                 이유: CH-005 때문에 Task 6 수정
```

각 문서가 **자기가 왜 바뀌었는지**를 적고, `영향범위` 칸이 다음 문서를 가리켜 연쇄를 추적하게 합니다. 이 연쇄를 도는 게 `change-propagation` 스킬이에요.

**(b) 실행했을 때** — ③ 하단에만 `[코드-수정]` 이 붙습니다:

```
CH-...-003 [구현계획서-수정]        "최초 생성 (Task 1~8)"       ← 계획 세울 때
CH-...-004 [코드-수정] (batch: Task 1~8)                        ← 실제 실행하고 나서
    "실행 중 자가수정 3건: head 0007 가정 → 실제 0020 이라 0021 로 정정 /
     fixture 없어서 mock 으로 / 경로 /api/v1 반영"
```

### 뭘 보고 "이유"를 적나 — 지어내는 게 아니다

출처는 딱 셋입니다. **그 순간에 실제로 일어난 일**이에요.

| 출처 | 예 |
|---|---|
| 사용자가 고른 선택지 | 4개 물었고 사용자가 "미완성 90일 보관" 고름 → 그게 이유 |
| 비교한 대안과 탈락 근거 | "이 방식은 컨테이너 정합성 위험" → 탈락 → 그게 결정의 이유 |
| 실행하며 드러난 사실 | "head 가 0007 이 아니라 0020 이었음" → 그게 보정의 이유 |

그래서 룰이 **"결정하는 그 순간 바로 적는다"** 입니다. 나중에 몰아 적으면 왜 그랬는지 이미 증발해서 **지어내게 됩니다.**

### 왜 "이유"를 적나 — 코드엔 "왜"가 안 남으니까

6개월 뒤 이 코드를 봤다고 합시다.

```python
hit = [t for t in terms if any(a.lower() in low for a in _aliases(t))]
```

> "왜 사전 전체를 안 넣고 굳이 필터링하지? 비효율인데?"
> → "최적화" 랍시고 전체 주입으로 바꿈 → 토큰 폭증 → 비용 터짐.

② §5 에 **"결정 B — 전체 주입 시 사전이 커질수록 토큰 폭증"** 이 적혀 있으면 → **안 건드립니다.**

**이유 = 미래의 나(또는 AI)가 그 결정을 되돌리지 못하게 막는 말뚝.**

### `영향범위`는 뭘 보고 적나 — 감이 아니라 grep

번호가 문서를 가로질러 박혀 있어서 **기계적으로 추적**됩니다.

```
① 요구사항    FR-5 (CSV 일괄 등록) · §5 범위밖 "동의어 검색 제외"   ← 여기가 바뀜
     │  grep "FR-5" / "동의어"
     ▼
② 개발방향    §2 표:  "FR-4,5,6 | glossary_csv.py | 신규"          ← 걸림
              §5:     "결정 E — 정규화는 검색 안 하니 YAGNI"        ← 걸림! 전제가 깨짐
     │  grep "결정 E" / "glossary_csv"
     ▼
③ 구현계획서  Task 5 (CSV 파서) · Task 1 (synonyms 컬럼)           ← 걸림

→ 영향범위: 개발방향 §2·§5(결정 E), 계획서 Task 1·5
```

`영향범위`는 **grep 결과**입니다. 번호가 없으면 못 적어요.

## 계획서는 어떻게 생기나

즉흥이 아닙니다. **전부 위에서 내려옵니다.**

```
FR (뭘 하나)  ────→  ② §2 (어느 파일)  ────→  Task N (그 파일 하나)
AC (판정)     ─────────────────────────────→  그 Task 의 테스트
② §5 결정 (왜) ────────────────────────────→  그 Task 의 구현 방식
② §6 위험     ─────────────────────────────→  ③ §2 위험 코드 지점 + 코드의 RISK 주석
```

### Task 수 = ② §2 표의 행 수

task 는 임의로 나눈 게 아니라 **② `§2 영향 컴포넌트` 표에서 그대로 내려옵니다.** 파일 1개 = task 1개.

**순서는 의존성이 정합니다** — entity 없인 CRUD 를 못 쓰니 entity 먼저. 서비스는 CRUD 가 필요하니 그 뒤. 라우터는 서비스가 필요하니 그 뒤. **최고위험 task 는 일부러 맨 끝**에 둡니다.

> ② `§2` 가 하는 일은 **"새 구조 발명"이 아니라 "이미 있는 레이어의 어디에 이 기능을 끼울지 매핑"** 입니다. 스킬 본문에도 *"기존 코드베이스에선 established pattern 을 따르라, 재구조화하지 마라"* 고 박혀 있어요. 프로젝트가 entity / DTO / CRUD / service / router 로 쪼개져 있다면, 그건 **그 프로젝트가 원래 그렇게 생겨서**지 이 워크플로가 강요한 게 아닙니다.

### 테스트 = AC 를 코드로 옮긴 것

```
NFR-2  "일부 행이 잘못돼도 정상 행은 등록되고, 실패 행은 사유와 함께 보고"
   ↓
AC-4   "필수값 누락 행이 섞여 있으면, 정상 행은 등록되고 실패 건수·사유가 보고된다"
   ↓
② 결정 C  "행별 검증 후 정상행만 커밋 (전체 롤백은 나쁜 UX)"
   ↓
def test_parse_partial_success():
    rows, errors = parse_terms_csv(raw)   # 3행 중 1행 필수값 누락
    assert len(rows) == 2      # ← 정상행만 = AC-4
    assert len(errors) == 1    # ← 실패 보고 = AC-4
```

**테스트를 먼저 쓰고 실패를 확인하는 이유** — 실패하는 걸 봐야 그 테스트가 **진짜 뭔가를 검증한다는 게 증명**됩니다. 구현부터 하면 항상 통과하는 가짜 테스트를 쓰게 돼요.

### 왜 코드까지 박아넣나 — 표류 방지

규칙이 하나 있습니다: **No Placeholders.** 다음은 전부 **계획 실패**로 봅니다.

> ❌ `"적절한 에러 처리 추가"` · `"TBD"` · `"Task 3 와 비슷하게"` · `"위 내용에 대한 테스트 작성"`

계획서를 읽는 건 **실행자**(AI 보조 에이전트든 사람이든)입니다.

- `"검증 추가해"` → 실행자가 알아서 지어냄 → **설계와 다른 게 나옴**
- 코드를 박아두면 → **그대로 복사만** 하면 됨 → 표류 0

그리고 **코드를 짜기 전에 리뷰할 수 있습니다.** 다 만들고 "이게 아닌데" 보다 훨씬 쌉니다.

### 기존 코드를 고칠 땐 — 원본/수정 후 쌍

````markdown
**원본** (`app/common/storage.py:46-54`):   ← 현재 파일과 byte 단위로 동일해야 함
```python
def delete_uri(uri): ...
```

**수정 후**:
```python
def delete_uri(uri): ...
def list_prefix(prefix): ...        ← 추가분
```
````

**원본이 byte 단위로 같아야 하는 이유** — 실행자가 **찾아 바꾸기**를 하는데, 한 글자라도 다르면 못 찾고 **엉뚱한 데를 고칩니다.** 그래서 `scripts/plan_byte_check.py` 가 저장 전에 이걸 검증해요.

## 왜 계획서만 길어지나

짐작 말고 **이 저장소가 자기 자신을 이 워크플로로 개발한 실제 문서**를 재봤습니다 — [`docs/features/2026-05-24-v2.5-no-ask-flag/`](features/2026-05-24-v2.5-no-ask-flag/)

| 문서 | 총 줄 수 | 코드블록 | 산문 · 표 |
|---|---|---|---|
| ① 요구사항 | 136줄 | **7%** | 52% |
| ② 개발방향 | 292줄 | 30% | **45%** |
| ③ 구현계획서 | **425줄** | **38%** | 37% |

무게중심이 다릅니다.

- **① 은 문장** — 계약이라 짧고 정확해야 합니다.
- **② 는 표** — `§2` 매핑 / `§3` 데이터 모델 / `§4` API / `§5` 결정(선택 · 대안 · 이유) / `§6` 위험. **"왜"를 압축**해 담는 문서라 코드가 거의 없어요.
- **③ 은 코드** — task 마다 **테스트 1벌 + 구현 1벌**을 쓰니 자연히 2배가 됩니다. 거기에 No Placeholders 가 곱해져서, 60줄짜리 파서는 60줄 그대로 들어갑니다.

③의 산문 37% 도 대충 쓴 게 아니라 task 마다 반복되는 이것들입니다.

```
**Files:** Create: 정확한/경로.py   Test: 정확한/테스트.py     ← 경로
**Model**: sonnet                                            ← 어느 모델로 시킬지
- [ ] Step 2: 테스트 실패 확인
      Run: pytest tests/... -v                               ← ★명령어 그대로
      Expected: FAIL — ModuleNotFoundError                   ← ★기대 출력까지
```

*"테스트를 돌린다"* 라고 쓰면 안 되고 **명령어와 기대 출력**을 박아야 해서 줄이 늘어요.

> **계획서가 긴 건 의도한 겁니다.** 스킬 본문에 이런 말이 있어요 — *"길이는 문제가 아니다, 모든 줄이 구체적이면. **모호한 짧음이 더 나쁘다.**"* ①② 는 사람이 판단하려고 읽으니 짧고, ③ 은 기계가 집행하니 깁니다.

## 검문은 두 번이다

시점이 다른 검문 두 개가 있습니다. 헷갈리기 쉬워요.

| 검문 | 언제 | 뭘 보나 | 비용 |
|---|---|---|---|
| **`verifying-spec`** | **코드 짜기 전** (②③ 쓰고 나서) | ①의 FR 을 ②가 다 덮었나? ①에 없는 게 ②에 몰래 들어갔나? → 4축 보고서 | 쌈 (문서끼리 대조) |
| **수용 기준(AC)** | **만들고 나서** | 진짜 되나? → 테스트로 예/아니오 판정 | 비쌈 (실물 검증) |

**앞의 건 문서끼리 대조, 뒤의 건 실물 검증.** 그래서 앞에서 먼저 걸러요.

### 범위 하나가 4층을 관통한다

```
① §5 범위밖   "xlsx 제외 — 문구가 '엑셀 파일(csv)' 라 CSV 로 한정"
② 결정 A      "표준 csv 모듈 — 대안 openpyxl 은 의존성 추가, 문구가 csv 라 불필요"
③ Task 5      parse_terms_csv (csv 만)
④ 코드        import csv
```

누가 ③에서 openpyxl 을 넣으면 → **①§5 위반으로 걸립니다.**

### "이유"는 세 군데로 나뉜다

같은 "왜"가 아니라 성격이 달라서 자리가 다릅니다.

| 이유의 종류 | 어디에 | 예 |
|---|---|---|
| **왜 이 방법인가** (선택) | ② `§5 핵심 결정` | `결정 B — 전체 주입 시 토큰 폭증하니 등장 단어만` |
| **왜 이렇게 됐나 · 바뀌었나** (변화) | 변경이력 | `CH-004: 실행해보니 head 가 0020 이라 0021 로 정정` |
| **이 줄이 왜 위험한가** (현장) | 코드 주석 | `# ⚠️ RISK(race): 같은 seq 동시 도착 — 경로 저장이라 멱등` |

RISK 주석의 카테고리는 **정확히 세 개**뿐입니다 — `side-effect` · `breaking` · `race`. *"여기 조심"* 같은 무분류 경고는 금지예요.

## 이 방식의 비용

솔직히 짚습니다. **공짜가 아닙니다.**

이 워크플로는 *"trivial 해도 TDD 사이클을 유지하라"* 를 기계적으로 강제합니다. 그래서 **값어치가 낮은 테스트가 섞입니다.**

| 테스트 | 뭘 잡나 | 값어치 |
|---|---|---|
| CSV 파서 (부분성공 · 인코딩) | 진짜 로직 | 🟢 높음 |
| 회귀 (기존 기능 안 깨지나) | 최고위험 방어 | 🟢 높음 |
| 라우터 (413 / 권한) | 경계 조건 | 🟡 중간 |
| entity ("컬럼이 있나") | ORM 프레임워크를 테스트하는 셈 | 🔴 낮음 |
| DTO ("필수값 걸리나") | 검증 라이브러리를 테스트하는 셈 | 🔴 낮음 |

**얻는 것** — 빠짐없음, 일관성, 실행자(AI)가 헷갈릴 여지 0.
**잃는 것** — 저가치 테스트가 섞이고 문서가 길어짐.

일반적인 표준은 **테스트 피라미드**입니다 — 로직 있는 곳에 단위테스트 많이, 통합 조금, E2E 아주 조금. **레이어마다 1:1 로 테스트를 다는 건 표준이 아닙니다.**

**빠져나갈 구멍이 있습니다.** 이건 취향이 아니라 *"프레임워크를 어디까지 믿느냐"* 의 문제라 — **기준을 ①이나 ②에 못 박아두면 ③이 그걸 따릅니다.** 예를 들어 ① NFR 이나 ② §7 테스트 전략에 이렇게 적어두면 됩니다.

```markdown
## 7. 테스트 전략
- 단위 테스트는 자체 로직이 있는 곳에만 — 파서 · 계산 · 게이트 · 회귀
- entity 컬럼 존재 / DTO 필수값 검증은 프레임워크를 신뢰하고 생략
```

**그게 이 워크플로의 요점입니다** — 판단은 위(①②)에서 하고, 아래(③④)는 그 판단을 따릅니다.

## 한 줄 정리

| 질문 | 답 |
|---|---|
| 변경이력은 언제 생기나 | **문서 만들자마자** (최초 생성). "다음 것부터" 아님 |
| 뭘 적나 | **"왜 이렇게 됐나"** — 이전 버전과의 diff 가 **아님** |
| 뭘 보고 적나 | 그 순간의 결정 근거 — 고른 선택지 / 탈락한 대안 / 실행 중 드러난 사실 |
| `영향범위`는 뭘 보고 | **번호 grep** — FR-N → ② §2 표 → 결정 라벨 → Task N |
| 문서 사이 연결은 어디 | 변경이력 아니고 **본문** — ② §2 의 FR 매핑 + ③ 헤더의 `Spec inputs` |
| 왜 ③만 긴가 | 기계가 집행하는 유일한 문서라 **코드 · 명령어 · 기대출력**이 다 들어감 |

**변경이력은 비교표가 아니라 그 문서의 항해일지입니다** — 왜 이 결정을 했고, 나중에 뭐가 바뀌었고, 실제로 만들었더니 뭐가 달랐는지.
