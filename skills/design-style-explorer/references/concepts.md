# 설계 원리 — 까먹으면 안 되는 개념

design-style-explorer 를 **수정·확장하거나 새 스타일을 추가할 때 반드시 먼저 읽는다.**
프롬프트·린터·워크플로가 왜 이렇게 생겼는지, 무엇을 지켜야 흔들리지 않는지의 근거.

## 1. 매력 축 (이 스킬의 뼈대)

각 스타일은 **"왜 매력적인가"** 하나의 축을 가진다 (SF HUD=연극성, 공공포털=신뢰, 럭셔리=품격 …).
**고도화 = 기법 나열이 아니라 그 매력 축의 완성이다.**

- 연출·인터랙션·색·타이포는 **전부 매력 축을 강화하는 방향으로만** 쓴다.
- 축과 무관한 "예쁜 기법"을 얹지 마라. 화려함이 목적인 스타일(HUD)과 절제가 목적인 스타일(공공·미니멀)은 정반대다.

## 2. 콘텐츠 고정 — 스타일 비교의 전제

한 탐색 세션 안에서 **콘텐츠·데이터·구성은 절대 바꾸지 않는다.**
시안마다 내용이 다르면 사용자가 "디자인"이 아니라 "내용"을 비교하게 된다 (사과 vs 오렌지).
→ master-prompt 의 `{{DOMAIN}}` 을 1회 확정하고, 전 스타일에 **한 글자도 다르지 않게** 적용.

## 3. 금지 = 스타일 정체성의 절반

각 스타일의 `금지` 목록은 장식이 아니라 **정체성이다.**

- 브루탈리즘에 부드러운 이징, 럭셔리에 바운스, 미니멀에 블롭, 공공에 카운트업 — **이런 "안티 이식"이 스타일을 죽인다.**
- 그래서 `scripts/style-lint.py` 가 금지 조항을 기계적으로 검사한다 (사람의 "느낌상 점검"은 편차가 크다).
- 린터는 **구조적 위반(라운드/그림자/그라디언트 함수/폰트)만** 잡는다. 의미 규칙(주황 정확히 3곳 등)은 `[MANUAL]` 로 안내 → 크리틱 루프에서 확인.
- 린터는 **보수적**이다: 애매하면 FAIL 로 flag 하고 사람/에이전트가 판단 (예: 브루탈리즘의 hard-stop 색 띠 gradient 는 기법상 OK지만 flag 됨).

## 4. 고도화 방법론 (v1 → v2 → v3)

- **v1**: 무드·팔레트·타이포·시그니처·금지 (정적 골격).
- **v2**: 여기에 **6 품질 레이어** — 히어로 연출(로드 시퀀스) / 깊이 3층 / 마이크로 인터랙션 3+ / 타이포 위계 4배 / 자체 크리틱 루프 / 안전장치(reduced-motion·JS 실패 폴백·focus). **단, 스타일 언어에 맞게** (럭셔리는 느리게, 브루탈은 하드하게, 공공은 연출 최소).
- **v3**: 사용자 피드백을 반영해 특정 스타일을 파고든 버전.

## 5. 아키텍처 — MD source / HTML output

- **MD(프롬프트)가 원본(source of truth)** — AI 가 읽고 작업하는 층. 토큰 효율·의미 명확.
- **HTML(시안)이 결과물** — 사람이 보는 층. 디자인의 본질이 CSS라 HTML 이 최적.
- **병렬**: 서브에이전트 가능 환경이면 스타일당 1에이전트 (Workflow `parallel()` 또는 한 응답 다중 dispatch). **각 에이전트는 자기 스타일 파일만 읽는다** — 다른 파일을 읽으면 스타일이 오염된다.
- **린트 게이트**: 생성 → 린트 → FAIL 수정 → 재린트 (품질 편차 제거).
- **갤러리**: `gallery.html` 로 30개를 패밀리(실무 코어·신뢰·다크·절제·부드러움·개성)로 묶어 한 화면에 비교.
- **콘텐츠 견고성**: style-lint(금지)와 **직교하는 축** — 큰 숫자·긴 제목·결측·빈 상태·접근성·차트 왜곡을 `references/content-checklist.md` 로 점검. "예쁜 예시가 아니라 최악 콘텐츠 기준". 린터에 공통 WARN(tabular-nums·반응형·reduced-motion)도 포함.

## 6. 로드맵 (차후 고도화 — 아직 미반영)

1~3(린트·병렬·갤러리)이 자리잡았다. 다음 후보:

- **④ tokens.json** — 각 스타일의 팔레트·radius·모션을 토큰으로 보관 → "2번 레이아웃 + 8번 컬러" 하이브리드를 **축 단위로 기계 병합** (재현 가능).
- **⑤ 변형 매트릭스** — 각 스타일에 강도(100/70/40%)·라이트/다크 축 → 30 → 30×N 탐색 공간.
- **⑥ 역추출 확장** — 5단계에서 디자인 시스템 프롬프트뿐 아니라 `tailwind.config.js` + CSS 변수 파일까지 → 확정 즉시 개발 착수.

## 7. 새 스타일 추가·수정 절차

1. `references/prompts/style-NN-<슬러그>.md` 작성 (매력 축 + v2 6레이어 포함).
2. `SKILL.md` 의 스타일 표에 행 추가 (번호·파일·매력 축·시그니처).
3. `SKILL.md` 2단계 배치 순서에 배치.
4. `scripts/style-lint.py` 의 `RULES`·`MANUAL`·`STYLE_NAME` 에 그 스타일의 검사 규칙 추가.
5. 도메인 하나로 생성 → 린트 통과 확인 후 확정.

> 요약: **매력 축을 지키고 · 콘텐츠를 고정하고 · 금지를 정체성으로 다루고 · MD를 원본으로 · 병렬은 자기 파일만.** 이 다섯이 흔들리면 이 스킬의 가치가 무너진다.
