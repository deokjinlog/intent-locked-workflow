# 개발로그 (devlog)

공부하고 삽질하고 깨달은 걸 **개발로그로 쌓아두고**, 필요할 때 **티스토리 발행용 글**로 정리한다. 한 스킬 안에 두 모드가 있다:

- **기록** — 문제·삽질·깨달음·부족한 점을 주제별 로그로 누적한다. 미래의 내가 다시 꺼내 쓰고, 나중에 글의 재료가 된다.
- **발행** — 쌓인 로그(또는 지금 세션)를 **문제 → 삽질 → 깨달음 → 결과 → 다음** 구조의 티스토리 개발로그 글로 정리한다.

핵심: **삽질과 깨달음이 살아있어야 좋은 로그 = 좋은 글감.** 성공만 남은 로그는 튜토리얼이 되고, 개발로그의 값어치는 "과정에서 배운 것"에서 나온다.

## 언제 발동하나 (auto-trigger)

**기록**:
- "이거 개발로그로 정리해줘" / "오늘 삽질 기록해줘" / "방금 배운 거 로그로 남겨줘"
- "이 로그에 이어줘" / "여기 추가해줘" (누적)
- "그 시도 빼줘" / "이 개발로그 지워줘" (삭제)

**발행**:
- "티스토리에 올릴 글로 정리해줘" / "이거 블로그 글로 써줘" / "개발로그 발행용으로 정리"

**발동 안 함** (오발동 방지): 기능 개발 요구사항(→ brainstorming), 저장·발행 요청 없는 단순 설명·요약, 코드 리뷰·디버깅·구현. 애매하면 발동 X.

## 입력 판별

- **대화 기반** — "지금까지 한 거 개발로그로" → 현재 세션에서 문제·삽질·깨달음·결과 추출
- **지정 자료 기반** — 파일/커밋/URL을 가리킴 → 그 자료를 읽어 정리
- 섞임 OK

## 모드 1 — 기록 (로그 누적)

주제는 **작업/챕터 단위**로 잡는다. 같은 작업의 후속은 새 파일이 아니라 그 로그에 **append** — 주제당 문서 하나로 자란다 (조각남 방지).

### 1. 로그 결정 (신규 or 누적)

1. 발동 시 `docs/devlog/` 의 기존 로그 목록(폴더명)을 먼저 확인
2. 현재 작업이 기존 로그를 확장하면 → 그 로그에 **append**
3. 명확히 다른 작업이면 → 신규
4. 애매하면 딱 한 번 물어봄: "기존 '<주제>' 로그에 이어붙일까요, 새로 만들까요?"

- **신규**: `docs/devlog/<YYYY-MM-DD>-<주제-슬러그>/<슬러그>.md` 생성 (폴더/파일명이 날짜+주제를 담아 스캔만으로 파악)
- **누적**: 기존 파일을 열어 이어씀

### 2. 로그 재료 (느슨한 앵커 — 내용에 맞게, 억지로 X)

**형식은 내용이 이끄는 대로.** 매번 같은 틀을 채우지 말고, 아래 재료를 자연스럽게 챙긴다. 이 재료가 살아있으면 나중에 회상도 되고 글로도 풀린다:

1. **한 줄 요약** *(맨 위)* — 뭘 하다 뭘 배웠나. 다시 열었을 때 3초 파악용. (누적 시 갱신)
2. **문제 / 목표** — 뭘 하려 했나, 왜.
3. **삽질 로그** — 어디서 막혔고 뭘 시도했나. **실패한 시도도 그대로 남긴다.** 개발로그에서 제일 값진 재료. `### 시도 N. ... / 결과. ...` 블록 단위로 (개별 추가·삭제 쉽게).
4. **깨달음** — 원인은 무엇이었고 어떻게 풀렸나, 무엇을 배웠나.
5. **결과** — 뭐가 됐나 (수치·산출물·경로).
6. **부족한 점 / 다음** — 한계, 다음에 팔 것. (연재 글의 떡밥이 된다)

(선택) **연결** — 이미 아는 것에 어떻게 붙나. 출처 `파일:라인`·커밋 해시는 검증 가능한 로그면 붙인다.

> 핵심: 순수 필사가 아니라 **"미래의 내가 다시 꺼내 쓰고, 나중에 글로 풀 수 있게".** 삽질·깨달음·결과 이 셋이 살아있으면 좋은 로그.

### 3. HTML 사본 (재)생성

- `generating-html` 방식으로 `<슬러그>.html` 생성
- 신규·누적·삭제 — `.md` 가 바뀔 때마다 HTML 재생성 (.md 가 source-of-truth, HTML 은 gitignore)

### 4. 위치 안내

- 저장 경로 + HTML 위치 한 줄. 끝.

## 모드 2 — 발행 (티스토리 개발로그 글)

발행 요청 시, 쌓인 로그(또는 세션)를 사람이 읽는 개발로그 글로 정리한다.

### 1. 재료 확보

- 지정 로그(`docs/devlog/<...>`) 또는 지금 세션에서 문제·삽질·깨달음·결과를 모은다.
- 재료가 부족하면 **딱 필요한 것만 한두 개** 물어본다 (예: 실패 에피소드의 구체적 증상, 정량값). 없는 걸 지어내지 않는다.

### 2. blog-craft 읽기 (필수)

- **`references/blog-craft.md` 를 매번 읽고** 그 원칙대로 쓴다. 톤·구조·태그 전략이 거기 있다. 읽지 않고 쓰면 밋밋한 필사가 된다.

### 3. 글 작성 → `<슬러그>-post.md`

blog-craft 의 구조를 따른다:

- **제목** — 짧은 질문형 ("AI 디자인은 왜 맨날 똑같이 생겼을까"). 검색 키워드 1개 자연스럽게.
- **결과물 먼저** — 첫 화면에 결과 showcase (이미지 + 굵은 한 줄). "그래서 뭐가 나오는데?" 를 3초 안에.
- **본문** — 문제 → 원인 → 실패한 시도 → 해결 → 검증 → 다음.
- 짧은 문장. 실패 루트를 명시적으로 분리. 추정 → 소거 → 확신 → 검증 흐름.
- 증거물(코드블록·정량 조항·폴더 트리)을 원문 그대로 인용 — 개발로그의 신뢰는 여기서 나온다.
- 이미지 자리를 `(이미지: 무엇을 넣을지)` 로 명시.
- 담백한 1인칭 평가는 남기고 수사는 뺀다.

### 4. 태그 + 발행 팁

- blog-craft 의 **태그 전략**대로 추천: 유입 키워드(클로드코드·바이브코딩 등 그때그때 뜨는 것) + 틈새 키워드 + 시리즈 연결용.
- 티스토리 **마크다운 모드**로 붙여넣으면 코드블록·표가 그대로 살아난다고 안내.
- 이미지 자리에 캡처를 넣으라고 안내 (결과가 비주얼인 글이면 이미지가 사실상 본문).

### 5. 위치 안내

- `<슬러그>-post.md` 경로 + 이미지 넣을 자리 몇 곳인지 알려준다. 끝.

## 삭제 / 정리 (prune)

- **부분 삭제** — "그 <X> 시도/섹션 빼줘" → 해당 블록만 제거 → HTML 재생성
- **전체 삭제** — "이 개발로그 지워줘" → 해당 `docs/devlog/<...>/` 폴더 통째 삭제
- **정리** — "이 로그 정리해줘" → 겹치거나 낡은 부분 다듬기 (의미 보존, 재구조화 X) → HTML 재생성

## 원칙 / 범위 밖

- **기록·발행 정리까지만** — 티스토리 API 자동 업로드 X (사람이 붙여넣기). 복습 알림·스페이스드 리피티션 X.
- **실제 겪은 것만** — 지어낸 삽질·과장된 실패 X. 개발로그의 신뢰는 진짜 원문·정량값에서 나온다.
- **개인 로그** — `docs/devlog/` 는 gitignore. 다국어·자동 게시 X.

## Anti-Patterns

| Wrong | Right |
|---|---|
| 저장·발행 요청 없는데 요약만 했는데 발동 | 명시적 "개발로그로/기록/발행/이어줘/빼줘" 의도일 때만 |
| 같은 작업 후속인데 새 파일 생성 (조각남) | 기존 로그에 **append** |
| 삽질·실패 빼고 성공만 남김 (튜토리얼됨) | 실패 루트가 개발로그의 핵심 — 그대로 남긴다 |
| 긴 인트로·분위기 서두 | 2줄 상황 설명 후 바로 본론 |
| 긴 설명형 제목 | 짧은 질문형 제목 + 검색 키워드 1개 |
| 지어낸 에피소드·과장된 수사 | 실제 원문·정량값만, 담백하게 |
| 결과물을 맨 뒤에 배치 | **결과물 먼저** (이탈률 ↓) |
| 발행 시 blog-craft 안 읽고 바로 씀 | 매번 `references/blog-craft.md` 읽고 그 원칙대로 |
| `.md` 만 고치고 HTML 그대로 둠 | 변경 시 HTML **재생성** |
| 기능 개발 요구사항인데 발동 | 그건 `brainstorming`. devlog 는 로그·발행 |

## 참조 파일

- `references/blog-craft.md` — **잘 읽히는 한국 개발 블로그의 글쓰기 원칙** (서사 구조·실패 루트·결과물 먼저·짧은 문장·증거물·제목·태그 전략). **발행(모드 2) 전 반드시 읽는다.** 새 원칙을 배우면 여기에 누적한다.
