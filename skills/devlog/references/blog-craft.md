# 블로그 글쓰기 원칙 — 까먹으면 안 되는 개념

devlog **발행(모드 2) 전 반드시 읽는다.** 잘 읽히는 한국 개발 블로그를 분석해 뽑은 원칙.
이걸 안 읽고 쓰면 로그의 밋밋한 필사가 된다. 새 원칙을 배우면 여기에 누적한다.

## 0. 두 결의 글 — 내용이 아키타입을 정한다

devlog 로 뽑는 글은 크게 둘이다. **먼저 이 글이 어느 쪽인지 정하고** 그 뼈대로 쓴다.
억지로 한쪽 틀에 밀어넣지 마라 — 개념을 이해한 글에 "삽질" 틀을 씌우면 어색해진다.

| | **개념 정리형** | **삽질기형** |
|---|---|---|
| 언제 | 뭔가를 **이해**했을 때 (원리·구조 파악) | 뭔가를 **만들며 겪었**을 때 (문제 해결) |
| 예 | "브라우저는 미니 OS다 — 저장 영역 전부 정리" | "AI 디자인은 왜 맨날 똑같을까 — 스킬 만들기" |
| 뼈대 | 훅 → 개념 섹션 → 헷갈리는 점 → 연결 → 정리 | 결과물 먼저 → 문제 → 삽질 → 해결 → 검증 → 다음 |
| 핵심 재료 | 비유·비교표·헷갈리는 점 구분 | 실패한 시도·정량 증거·검증 |

섞여도 된다 (개념을 이해하며 삽질한 글). 하지만 **주된 결 하나를 정하고** 거기에 맞춘다.

---

## A. 개념 정리형 — "내가 이해한 걸 남도 이해하게"

원리·구조·개념을 파악하고 정리하는 글. (예: 브라우저 저장소 정리)

1. **훅 오프닝** — 궁금증에서 출발해 확인으로. *"~ 아닌가 싶었다. 맞았다."* 배우게 된 계기를 한두 줄로. 교과서처럼 정의부터 늘어놓지 않는다.
2. **개념 섹션 + 비교표** — 각 개념을 구조로 설명. 특히 **여러 개를 나란히 놓을 땐 비교표** (영역/수명/용량/용도 식). 표 하나가 문단 다섯 개를 이긴다.
3. **헷갈리기 쉬운 점 콕 집기** *(능동 회상)* — 배우는 사람이 **반드시 헷갈리는 지점**을 집어서 구분해준다 (*"sessionStorage ≠ 서버 세션"*). 이게 개념 글의 값어치.
4. **비유** — 추상 개념을 익숙한 것에 매핑 (*"임차 메모리 = 집주인이 언제든 방 빼는"*). 한 방에 이해된다.
5. **도식** — 관계·흐름은 다이어그램으로. `브라우저 메모리 → 디스크 캐시 → CDN → Redis → DB`.
6. **연결** *(정교화)* — 이미 아는 것과 잇는다 (*"서버에서 배운 구조와 똑같다"*). 고립된 지식은 안 남는다.
7. **정리** — 맨 끝에 bullet 3~4개로 압축. 다시 볼 때 여기만 읽어도 핵심이 잡히게.

## B. 삽질기형 — "겪은 사람만 쓸 수 있는 글"

뭔가를 만들며 문제를 겪고 푼 글. 개발로그는 튜토리얼이 아니다 — **"완성된 지식"이 아니라 "과정에서 배운 것"을 판다.**

1. **서사** — 문제 상황(2줄) → 원인 추정 → 실패한 시도 → 해결 → 검증 → 다음 계획.
2. **실패 루트를 명시적으로 분리** — 성공/실패 경로 구분이 잘 쓴 개발기의 조건. 본문에 **추정 → 소거 → 확신 → 검증** 흐름을 심는다.
3. **검증 섹션 따로** — 해결로 끝내지 말고 "정말 고쳐졌나"를 다른 조건으로 확인. 직접 측정·검증한 내용이 신뢰를 만든다.
4. **구체적 실패 묘사** — *"이벤트 당첨 화면이 됐다"* 같은 장면이 있어야 진짜 겪은 글로 읽힌다. 성공만 남으면 튜토리얼.

## 공통 크래프트 (둘 다 적용)

- **핵심/결과 먼저** — 개념형은 한 줄 요약·훅으로, 삽질기형은 결과물 쇼케이스(이미지+굵은 한 줄)로. "그래서 뭔데?"를 3초 안에. 결과 → 과정 순서면 결과만 궁금한 독자도, 방법 궁금한 독자도 다 잡는다.
- **짧은 문장 + 시각적 쉼표** — 문장을 쪼갠다. 때로 *"망했다."* 한 문장 단락. 섹션마다 표·코드블록·인용구로 쉼표를 준다. 분량은 "필요한 정보 + 한 줄 의견"으로 압축.
- **증거물** — 코드블록·정량 조항(*"골드 면적 5% 초과 시 실패"*)·폴더 트리·커밋 해시. 신뢰는 **원문 인용**에서 나온다.
- **담백한 1인칭** — *"이게 있고 없고의 품질 차이가 생각보다 크다."* 평가는 남기고 수사는 뺀다.
- **짧은 질문형 제목** — ✅ *"AI 디자인은 왜 맨날 똑같이 생겼을까"* / ❌ 길고 설명형. 검색 노출용 키워드 1개를 뒤에 살짝.
- **인트로 최소 + 한 단어 섹션 제목** — 분위기 서두 삭제, 두 줄 후 바로 본론. 섹션 제목은 **원인 / 검증 / 다음 / 정리**.
- **이미지 자리 명시** — `(이미지: 무엇을 넣을지)`. 결과가 비주얼인 글은 이미지가 사실상 본문.
- **태그 전략** — 유입 키워드(발행 시점에 뜨는 것: 클로드코드·바이브코딩 등) + 틈새 키워드 + 시리즈 연결용. 티스토리 태그는 구글에 직접 안 잡히니 **진짜 검색 유입은 제목·본문 키워드**에서 나온다 → 본문 첫 문단에 핵심 키워드가 자연스럽게.
- **연재 떡밥** — 마지막에 다음 글 예고. (모드 1 로그의 "부족한 점 / 다음"이 그대로 떡밥.)

## 발행 전 체크리스트

- [ ] 이 글의 아키타입(개념 정리형 / 삽질기형)을 정했나
- [ ] 핵심/결과를 앞에 뒀나 (한 줄 요약·훅 or 결과물 쇼케이스)
- [ ] **개념형**: 헷갈리는 점 구분 · 비유 · 비교표 · 연결 · 정리 bullet 있나
- [ ] **삽질기형**: 실패 루트 분리 · 검증 섹션 · 구체적 실패 묘사 있나
- [ ] 증거물(코드블록·정량값·트리) 인용했나
- [ ] `(이미지: ...)` 자리를 명시했나
- [ ] 제목이 짧은 질문형 + 검색 키워드 1개인가
- [ ] 태그: 유입 + 틈새 + 시리즈 (발행 시점 트렌드 반영)
- [ ] 다음 글 예고(떡밥) 걸었나
- [ ] 지어낸 내용 0 — 전부 실제 원문/정량값인가

> 요약: **먼저 아키타입을 정하고 · 핵심을 앞에 · 짧게 · 증거로 · (개념형은 헷갈리는 점·비유로, 삽질기형은 실패·검증으로) · 짧은 질문형 제목 · 트렌드 태그.** 좋은 글은 "이해한 사람" 또는 "겪은 사람"만 쓸 수 있고, 그 진짜배기 느낌이 원문 인용과 구체성에서 나온다.
