# 블로그 글쓰기 원칙 — 까먹으면 안 되는 개념

devlog **발행(모드 2) 전 반드시 읽는다.** 잘 읽히는 한국 개발 블로그를 분석해 뽑은 원칙.
이걸 안 읽고 쓰면 로그의 밋밋한 필사가 된다. 새 원칙을 배우면 여기에 누적한다.

## 1. 서사 구조 — 문제 → 삽질 → 깨달음 → 결과 → 다음

개발로그는 튜토리얼이 아니다. **"완성된 지식"이 아니라 "과정에서 배운 것"을 파는 글**이다.
그래서 뼈대는 항상 이 흐름:

```
문제 상황 (2줄) → 원인 추정 → 실패한 시도 → 해결 → 검증 → 다음 계획
```

- 실제 있었던 **전환점**을 삽질 에피소드로 살린다. "이벤트 당첨 화면이 됐다" 같은 **구체적 실패 묘사**가 있어야 진짜 겪은 글로 읽힌다.
- 성공 경로만 있으면 튜토리얼. **실패 경로가 개발로그의 정체성.**

## 2. 실패 루트를 명시적으로 분리

잘 쓴 개발기의 조건은 **성공/실패 경로 구분**이다. 목차 자체를 이렇게 배치:

```
실패한 첫 시도 → 두 번째 시도 → 삽질 1 → 삽질 2 → 검증
```

본문에 **추정 → 소거 → 확신 → 검증** 구조를 심는다:
- "원인을 추정해봤다" (문제 정의)
- "소거 끝에 남은 답은" (삽질)
- "도메인을 바꿔서 다시 돌려봤다" (검증)

## 3. 검증 섹션은 따로 둔다

**직접 측정하고 검증한 내용**이 담긴 글이 높게 평가받는다. 해결로 끝내지 말고,
"정말 고쳐졌나"를 다른 조건으로 확인한 섹션을 붙인다. 신뢰가 여기서 나온다.

## 4. 결과물 먼저 (이탈률 ↓)

독자는 "그래서 뭐가 나오는데?"를 빨리 보고 싶어한다. 특히 **결과가 비주얼인 글**은
도입부 바로 다음에 **결과물 쇼케이스**를 넣는다:

- 이미지 한 장 + 굵은 이름 + 특징 한 줄 세트를 3~4번 반복 → 스크롤하며 "이미지 보고 → 한 줄 읽고" 리듬.
- 정적 이미지가 못 담는 **움직임은 텍스트로 보완** ("새로고침하면 타이핑과 함께 전원이 들어온다").
- 결과 → 과정 순서 = 결과만 궁금한 독자는 앞에서 만족, 방법 궁금한 독자는 계속 읽는 **이중 구조**.

## 5. 호흡 — 짧은 문장 + 시각적 쉼표

- 문장을 짧게 쪼갠다. 때로 **"망했다." 한 문장 단락**도 쓴다.
- 섹션마다 **표·코드블록·인용구**로 시각적 쉼표를 넣는다.
- 분량은 과감히 압축: 각 섹션을 **"필요한 정보 + 한 줄 의견"**으로. 묘사·수사는 뺀다.
- 담백한 1인칭 평가는 남긴다: "이게 있고 없고의 품질 차이가 생각보다 크다."

## 6. 증거물을 곳곳에 배치

개발로그의 신뢰는 **원문 인용**에서 나온다. 지어낸 문장이 아니라:

- 금지 목록·설정 **코드블록**
- **정량 조항** ("골드 면적 5% 초과 시 실패")
- **폴더 트리**, 실제 파일 경로, 커밋 해시

이런 증거물이 "진짜 만든 사람의 글"임을 증명한다.

## 7. 제목 — 짧은 질문형

- ✅ "AI 디자인은 왜 맨날 똑같이 생겼을까" / "컨트롤러에 로직 넣어도 될까요?"
- ❌ "~를 그만두기까지 — Claude 스킬 개발기" (길고 설명형)
- 검색 노출을 챙기려면 뒤에 키워드를 살짝: "AI 디자인은 왜 맨날 똑같이 생겼을까 — 클로드 스킬로 해결하기".

## 8. 인트로 최소 + 섹션 제목은 한 단어

- 분위기 잡는 서두, 블로그 이름 드립 마무리 **삭제**. 두 줄 상황 설명 후 바로 "## 원인"으로.
- 섹션 제목은 한 단어 위주: **원인 / 검증 / 다음 / 정리**.
- 첫 글이면 "[개발로그]" 말머리는 제목에서 빼고 **카테고리**로 처리.

## 9. 태그 전략 (검색 유입)

태그는 **유입과 직결**된다. 세 종류를 섞는다:

| 역할 | 예시 | 이유 |
|---|---|---|
| 주제 정확도 | 클로드스킬 · ClaudeSkills (한/영 둘 다) | 검색어가 갈린다 |
| 유입량 | 클로드코드 · 바이브코딩 | 지금 가장 뜨는 키워드 |
| 시리즈 연결 | 다음 글의 메인 키워드 | 연재 연결 |

- **뜨는 키워드는 그때그때 다르다** — 발행 시점에 실제로 뭐가 통용되는지 확인하고 정한다 (지금은 바이브코딩·클로드코드가 대세, 클로드스킬은 경쟁 덜한 틈새).
- 티스토리 태그는 **구글 검색에 직접 안 잡힌다** (내부 관련글용). 진짜 검색 유입은 **제목과 본문 키워드**에서 나온다 → 본문 첫 문단에 핵심 키워드가 자연스럽게 들어가야 한다.
- "이 글은 A인데 B 태그 달아도 되나?" — 글의 "다음" 섹션에 B가 예고돼 있으면 거짓이 아니다. 시리즈 연결용으로 미리 달아둔다.

## 10. 연재 떡밥

마지막 문단에 **다음 글 예고**를 건다. 첫 글에서 연재 떡밥을 깔면 재방문 이유가 생긴다.
(모드 1 로그의 "부족한 점 / 다음"이 그대로 떡밥이 된다.)

## 11. 발행 전 체크리스트

- [ ] 결과물 쇼케이스가 도입부 바로 다음에 있나
- [ ] 실패 루트가 성공 경로와 분리돼 있나
- [ ] 검증 섹션(직접 확인)이 있나
- [ ] 증거물(코드블록·정량값·트리) 인용했나
- [ ] `(이미지: ...)` 자리를 명시했나 — 결과물 컷 우선
- [ ] 제목이 짧은 질문형 + 검색 키워드 1개인가
- [ ] 태그: 유입 + 틈새 + 시리즈 (발행 시점 트렌드 반영)
- [ ] 다음 글 예고(떡밥) 걸었나
- [ ] 지어낸 내용 0 — 전부 실제 원문/정량값인가

> 요약: **결과물 먼저 · 실패를 숨기지 말고 · 짧게 · 증거로 · 검증까지 · 짧은 질문형 제목 · 트렌드 태그.** 개발로그는 "겪은 사람"만 쓸 수 있는 글이다. 그 진짜배기 느낌이 원문 인용과 구체적 삽질에서 나온다.
