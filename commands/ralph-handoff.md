# /ralph-handoff — 무인 루프에 넘길 워크트리 싸기

승인된 산출물을 **새 워크트리**로 격리하고, 거기에 `PROMPT.md` 를 자동 작성해서 `ralph-loop` 가 바로 물 수 있는 상태로 만듭니다. **현재 브랜치는 건드리지 않습니다** — 무인 실행이 망쳐도 원본이 남습니다.

<HARD-GATE>
이 명령은 **워크트리를 만들고 파일을 지웁니다.** 사용자가 `/ralph-handoff` 를 **명시 호출**한 경우에만 실행하세요. "랄프한테 넘기자" 같은 대화 중 언급으로 절대 발동하지 마세요. `/new-skill` · `/remove-skill` 이 명령인 것과 같은 이유입니다 (META-BUILDER 룰).
</HARD-GATE>

## 1. 인자 파싱

- `<slug>` — 피처 슬러그. 누락 시 `docs/features/` 의 가장 최근 폴더를 쓰고 **어느 걸 골랐는지 한 줄 보고**.
- `--from=requirements` (**기본**) — ① 만 넘김. 랄프가 ②③④ 를 만듭니다.
- `--from=plan` — ①②③ 을 넘김. 랄프는 ④ 만 합니다.
- `--dry-run` — 뭘 할지만 출력하고 아무것도 안 만듦.

## 2. 사전 검증 (하나라도 실패하면 중단)

```bash
F=$(ls -d docs/features/*-<slug> 2>/dev/null | head -1)
```

| 검사 | 실패 시 |
|---|---|
| `git rev-parse --git-dir` 성공 | "git 저장소가 아닙니다. 워크트리를 만들 수 없어요." → 중단 |
| `git status --porcelain` 이 비어 있음 | "커밋 안 된 변경이 있습니다. 먼저 커밋하거나 stash 해주세요." → 목록 보여주고 중단 |
| `$F` 폴더 존재 | "`<slug>` 피처 폴더가 없습니다. `/brainstorming` 부터 하세요." → 중단 |
| ① 의 `## 변경이력` 엔트리 ≥ 1 | **"① 이 아직 승인 전입니다."** → 중단 |
| (`--from=plan` 일 때) ②③ 도 엔트리 ≥ 1 | 같은 이유로 중단 |

**승인 판정은 변경이력 엔트리 존재로 합니다.** 이 저장소가 이미 쓰는 신호예요 (`generating-html`: *"If ANY entry exists, this is NOT initial creation"*).

```bash
sed -n '/^## 변경이력/,$p' "$F"/*-requirements.md | grep -c '^- \*\*id\*\*:'
```

## 3. 워크트리 생성

```bash
W="../$(basename $(pwd))-ralph-<slug>"
git worktree add "$W" -b "ralph/<slug>"
```

이미 있으면 **덮어쓰지 말고** 경로를 보고한 뒤 중단하세요.

## 4. 범위 정리 (`--from=requirements` 일 때만)

워크트리 **안에서만** 지웁니다. 원본 브랜치는 그대로입니다.

```bash
cd "$W"
git rm -q "docs/features/"*"-<slug>/"*-tech-design.md \
          "docs/features/"*"-<slug>/"*-implementation-plan.md 2>/dev/null || true
# 그 피처가 만든 코드 디렉토리가 있으면 함께 (계획서 §1 의 Files 목록을 근거로 판단)
git commit -q -m "chore: 무인 실행용 — 요구사항만 남김"
```

**①(`-requirements.md`)은 절대 지우지 마세요.** 그게 넘기는 대상입니다.

## 5. `PROMPT.md` 작성

`$W/PROMPT.md` 를 만듭니다. 아래 다섯 가지는 **반드시** 들어가야 합니다 — 각각 실제 무인 실행에서 터진 걸 막는 장치입니다.

### 5-1. 상태 판정 (매 반복 첫 동작)

랄프는 매 반복 기억이 없습니다. **변경이력 엔트리 수로 단계를 판정**하게 하세요.

```
② 없음/엔트리 0        → ② 를 쓴다 → 커밋 → 멈춘다
② 완성 · ③ 없음/엔트리 0 → ③ 을 쓴다 → 커밋 → 멈춘다
③ 완성 · VERIFY.md 없음  → ③.5 검문 → ④ 진입
③ 완성                  → 체크 안 된 첫 task
```

**한 반복에 한 단계.** `auto-tech-design` 은 Step 7 에서 `auto-writing-plans` 를 자동 호출하는데 **그걸 하지 말라고** 명시하세요. 안 그러면 한 반복에 ②③④ 를 다 하려다 컨텍스트가 터지고, 반쯤 쓰다 죽으면 다음 반복이 그걸 완성본으로 오해합니다.

### 5-2. ★ 완료 조건에 사람이 센 숫자를 넣지 마라

```
❌  pytest 25건 통과              ← 사람이 셈 → 틀림 → 루프가 틀린 기준을 쫓음
✅  pytest 전부 통과 (실패 0 · 에러 0)
    + 계획서가 정의한 테스트가 전부 수집됨:
      grep -oE 'def test_[a-z0-9_]+' <계획서> | sort -u | wc -l
        ==  pytest --collect-only -q | grep -c '::test_'
```

**실제로 터진 적 있습니다** — 계획서 §4 에 `22건` 이라 적혔는데 본문이 정의한 건 `25건` 이었습니다 (한 task 의 테스트 3건이 집계 누락). 사람이 안 세면 틀릴 수가 없습니다.

### 5-3. ★ ③.5 검문 관문 (`--from=requirements` 일 때 필수)

**랄프가 쓴 ③ 으로 랄프가 ④ 를 채점하면, 오류가 사라지는 게 아니라 안 보이게 됩니다.** 자기 답안지로 자기 채점이니까요.

④ 진입 전 **딱 1회**, 서브에이전트를 **구현자가 아닌 검문자**로 띄우게 하세요:

> 이 계획서만 읽어라. **코드는 절대 보지 마라.**
> 네 목적은 구현이 아니라 **어긋남 찾기**다.
> 1. §4 완료 조건의 **모든 숫자**를 본문과 대조해라
> 2. 본문에 없는 파일명·함수명이 §4 에 있는지 봐라
> 3. 계획서가 정의한 테스트를 **직접 세어** §4 와 비교해라
> 어긋남을 전부 나열해라. 없으면 `어긋남 없음` 한 줄만.

어긋남이 나오면 → **계획서(정본)를 고치고 CH 를 남긴 뒤** ④ 진입. 코드에서 우회 금지.

**다른 프롬프트 · 다른 컨텍스트 · 다른 목적** — 이 셋이 달라야 가로질러 읽힙니다.

### 5-4. ★ 나가도 증거가 남게

루프의 Stop 훅은 상태파일이 사라지면 종료를 못 막습니다. 그런데 훅은 **9군데**에서 그 파일을 지웁니다 — promise 감지·최대반복은 정상이지만, **transcript 파싱 실패 4개와 상태파일 손상 3개는 조용히 죽습니다.** 프롬프트로 막을 수 없어요.

**막으려 하지 말고 증거를 남기게 하세요:**

> ④ 의 마지막 task 뒤 → `VERIFY.md` 에 §4 의 검증 명령을 **하나씩 실제로 돌린 터미널 출력을 그대로 붙여넣어라.** 요약·재작성 금지.

**`VERIFY.md` 가 있으면 진짜 돌린 거고, 없으면 안 돌린 겁니다.** promise 는 말이지만 이건 출력입니다.

### 5-5. 금지 목록

| 금지 | 왜 |
|---|---|
| 사람에게 묻고 기다림 | 답할 사람이 없다. 무한 루프가 된다 → `BLOCKED.md` 에 적고 건너뛴다 |
| **`-requirements.md` 수정** | **① 은 사람 것이다.** 고치면 이 도구를 쓰는 의미가 사라진다 |
| **`/cancel-ralph` 호출 · 상태파일 삭제·수정** | 완료 안 됐는데 빠져나가는 문 |
| 참이 아닌 `<promise>` | 루프의 안전장치를 무력화한다 |
| `git push` · PR · 외부 API | 되돌릴 수 없다. 로컬 커밋까지만 |
| `rm -rf` · `reset --hard` · force-push | 되돌릴 수 없다 |
| 한 반복에 두 단계 | 컨텍스트가 터진다 |
| 커밋 없이 다음으로 | 다음 반복이 진척을 못 읽는다 |
| 테스트를 통과시키려 테스트를 고침 | 테스트가 곧 수용 기준이다. 구현을 고쳐라 |

## 6. 보고

```
✅ 워크트리를 만들었습니다.

  경로     ../<repo>-ralph-<slug>
  브랜치   ralph/<slug>
  넘긴 것  요구사항 (FR N개 / NFR N개 / AC N개)
  지운 것  개발방향 · 구현계획서 (이 워크트리에서만 — 원본 브랜치는 그대로)
  만든 것  PROMPT.md

다음 — 새 세션에서 실행하세요 (루프가 실행한 세션을 그대로 잡습니다):

  cd ../<repo>-ralph-<slug> && claude

  /ralph-loop "PROMPT.md 를 읽고 그대로 따라라. 매 반복 그 파일이 지시서다." \
    --completion-promise "<완료문구>" --max-iterations 30

--max-iterations 를 꼭 주세요. 완료 문구는 하나만 매칭되므로 반복 상한이 주 안전장치입니다.

아침에 볼 것:
  cat VERIFY.md     ← 검증을 진짜 돌렸나 (없으면 안 돌린 것)
  cat BLOCKED.md    ← 막힌 것 · 사람이 볼 것
  git log --oneline ← 뭘 했나
```

`--dry-run` 이면 위 내용을 출력만 하고 **아무것도 만들지 마세요.**

## 7. 끝난 뒤 정리 (사용자 안내만, 자동 실행 X)

```bash
git worktree remove ../<repo>-ralph-<slug>     # 버릴 때
git merge ralph/<slug>                          # 살릴 때 — 먼저 diff 로 검토
```

**`--from=requirements` 로 돌렸다면 병합 전에 ② 를 꼭 읽어보세요.** 기술 결정을 랄프가 골랐습니다 (`auto-tech-design` 은 *"alternatives 비교 → recommendation 자동 선택"*). 당신이라면 다르게 골랐을 게 있으면 `change-propagation` 으로 고치면 아래가 따라옵니다.

## Anti-Patterns

| 안티 패턴 | 이유 |
|---|---|
| 대화 중 "랄프한테 넘기자" 로 자동 발동 | 워크트리 생성 + 파일 삭제. 명시 호출만 (META-BUILDER 룰) |
| 현재 브랜치에서 ②③ 삭제 | 원본이 사라진다. **워크트리 안에서만** 지운다 |
| ① 미승인 상태로 넘김 | 랄프가 ② 에서 추측으로 메운다. 게이트를 통과한 ① 만 |
| `PROMPT.md` 에 완료 조건 숫자를 사람이 셈 | 22 vs 25 회귀. 기계끼리 대조시켜라 |
| ③.5 검문 생략 | 자기 답안지로 자기 채점 |
| `--from=requirements` 결과를 검토 없이 병합 | 기술 결정을 아무도 안 봤다 |
