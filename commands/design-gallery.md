# Design Gallery — 디자인 시안 여러 개 비교 후 선택

한 주제에 대해 **확실히 다른 디자인 방향 8~9개**를 한 번에 만들어 HTML 갤러리로 비교합니다. 고른 방향을 재변주하고, 최종 시안에서 **디자인 토큰**을 뽑아 프로젝트 기준으로 삼습니다.

## 핵심 원칙 (안 지키면 실패)

1. **개수보다 다양성** — 각 시안은 서로 다른 아키타입을 **강제 배정**. 같은 프롬프트로 12~30개 뽑으면 절반이 비슷해짐(mode collapse) + 고르기 힘들어짐(choice paralysis).
2. **내용은 고정** — 모든 시안이 **같은 카피·같은 섹션**을 사용. 안 그러면 디자인이 아니라 내용을 비교하게 됨.
3. **디버지 → 컨버지** — 1라운드 넓게(아키타입) → 선택 → 2라운드 좁게(변주). 비싼 풀 디테일 생성을 **이긴 방향에만**.
4. **선택 → 토큰** — 최종 시안에서 디자인 토큰을 뽑아 워크플로 기준으로. ("예쁜 거 고르고 끝" 방지)

---

## 1라운드 (Diverge) — 아키타입 8~9개

### Step 1. 공유 콘텐츠 확정

주제(인자)를 받고, **모든 시안이 공유할 콘텐츠**를 먼저 확정한다:

- 페이지/제품 종류, 핵심 섹션 목록 (예: hero · features · pricing · CTA), 실제 카피(짧게)
- 부족하면 `AskUserQuestion` 으로 1~2개만 질문 (섹션 구성 / 톤 / 타깃)

이 콘텐츠는 **라운드 내내 고정**한다.

### Step 2. 아키타입 병렬 생성

아래 9개 중 8~9개(기본 9, `--count K` 로 조절)를 골라, **각각 별도 보조 에이전트를 병렬 dispatch**한다. 각 에이전트 프롬프트:

- Step 1 의 **동일 콘텐츠**
- 배정된 **아키타입 1개**(아래 표) — 그 스타일 언어를 확실히 밀어붙일 것
- **self-contained `.html`** (인라인 CSS/JS만, 외부 CDN·폰트·이미지 0, 오프라인 렌더)
- `frontend-design` 원칙 참고 (의도적 타이포·색·레이아웃)

| # | 아키타입 | 스타일 언어 |
|---|---|---|
| 01 | 미니멀 | 넓은 여백, 흑백 + 액센트 1개, 큰 타이포, 절제 |
| 02 | 브루탈리즘 | 날것, 굵은 보더, 모노스페이스, 원색, 비대칭 |
| 03 | 에디토리얼(잡지) | 세리프, 멀티컬럼 그리드, 큰 헤드라인, 여백 리듬 |
| 04 | 다크 프리미엄 | 딥 다크 배경, 글로우/네온 액센트, 유리 카드 |
| 05 | 플레이풀 | 밝은 색, 둥근 모서리, 일러스트/이모지, 경쾌 |
| 06 | 코퍼릿/SaaS | 깔끔, 블루 계열, 카드 그리드, 신뢰감 |
| 07 | 글래스모피즘 | 반투명 블러 패널, 그라데이션 배경, 깊이감 |
| 08 | 레트로/Y2K | 네온, 픽셀/그리드, 90s~00s 감성 |
| 09 | 뉴모피즘 | 소프트 섀도, 단색 배경, 부드러운 입체 |

저장: `docs/design-gallery/<YYYY-MM-DD>-<주제-슬러그>/01-minimal.html` … `09-neumorphism.html`

### Step 3. 갤러리 index.html

9개를 **한눈에 비교**하는 `index.html` 을 생성한다:

- 번호 매긴 카드 그리드, 각 카드에 `<iframe>` 미리보기 + 아키타입 이름
- 카드 클릭 → 새 탭 풀뷰
- self-contained (다크 배경, 반응형)

저장: `docs/design-gallery/<...>/index.html`. 사용자에게 **경로 안내** + "몇 번이 맘에 드세요? (1~2개)" 질문(`AskUserQuestion`).

---

## 선택 → 2라운드 (Converge, 선택)

사용자가 1~2개 고르면:

### Step 4. 변주 생성 (`--refine <번호>`)

고른 아키타입 **안에서** 3~4개 변주(색·타이포·레이아웃 미세 차이)를 병렬 생성 → 2라운드 갤러리 → 최종 1개 선택.

콘텐츠는 여전히 고정. 변주는 **디자인 세부만** 바꾼다.

---

## 최종 → 디자인 토큰

### Step 5. 토큰 추출

최종 시안에서 뽑아 `docs/design-gallery/<...>/design-system.md` 로 저장:

- **색 팔레트** — primary / background / surface / text / accent (hex)
- **타이포** — heading 폰트·크기 스케일 / body 폰트·크기
- **간격** — spacing 스케일 (4/8/16/24/…)
- **컴포넌트** — 버튼 / 카드 / 입력 필드 기본 스타일

이 `design-system.md` 가 이후 **기획 → 구현 워크플로의 디자인 기준**이 된다. (`brainstorming` / `tech-design` 이 참조)

---

## 금지 / 원칙

- **30개+ 남발 금지** — mode collapse + choice paralysis. 8~9개(다양) > 30개(비슷).
- **아키타입 겹침 금지** — 배정된 스타일을 확실히 밀 것.
- **`.html` self-contained** — 외부 의존성 0, 오프라인 렌더.
- **콘텐츠 고정** — 라운드 내내 같은 카피·섹션 (디자인만 비교).
- **`.html` 은 gitignore 대상** — 사람용 시각 산출물 (`docs/design-gallery/**/*.html`). `design-system.md` 는 커밋.
- **선택으로 끝내지 말 것** — 반드시 토큰 추출까지 이어 프로젝트 기준화.
